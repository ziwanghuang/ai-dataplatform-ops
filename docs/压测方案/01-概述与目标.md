## 1. 压测背景

### 1.1 为什么需要压测

AI DataPlatform Ops Agent 是一个**多组件协同**的智能运维系统，包含：

- **Python FastAPI** — HTTP API 入口（异步）
- **Go MCP Server** — 工具执行网关（Fiber + 8 层中间件）
- **LLM 多供应商** — DeepSeek/OpenAI/Anthropic/Local（含熔断+容灾）
- **RAG 混合检索** — Milvus + ES + Neo4j + Reranker
- **LangGraph 状态图** — 9 节点 12 边有状态编排

每一层都有独立的性能特征和瓶颈点。压测的核心目的：

1. **找到系统瓶颈** — 是 LLM API 延迟？还是 RAG 检索？还是 MCP 工具调用？
2. **验证容灾机制** — 熔断器、降级、容灾切换在高负载下是否正确触发
3. **确定容量上限** — 单实例能支撑多少并发诊断请求
4. **建立性能基线** — 为后续迭代提供对比基准

### 1.2 压测不覆盖的范围

- 前端 UI 性能（本项目无前端）
- 第三方 LLM API 本身的性能（通过 Mock 模拟）
- 大数据集群（HDFS/YARN/Kafka）本身的性能

## 2. 性能目标 (SLO)

### 2.1 延迟目标

| 链路 | P50 | P95 | P99 | 超时上限 |
|------|-----|-----|-----|---------|
| **健康检查** GET /health | < 5ms | < 10ms | < 50ms | 1s |
| **工具列表** GET /tools | < 10ms | < 50ms | < 100ms | 1s |
| **快速路径** DirectTool | < 500ms | < 1s | < 2s | 5s |
| **完整诊断** Triage→Report | < 5s | < 10s | < 15s | 30s |
| **告警处理** /alert | < 8s | < 15s | < 20s | 30s |
| **MCP 工具调用**（单次） | < 100ms | < 500ms | < 2s | 30s |
| **RAG 混合检索** | < 200ms | < 500ms | < 1s | 2s |
| **LLM 调用**（单次） | < 2s | < 5s | < 10s | 60s |
| **HITL 审批轮询** | < 50ms | < 100ms | < 200ms | 1s |

### 2.2 吞吐量目标

| 指标 | 目标值 | 说明 |

### 3.2 关键性能路径

**路径 A：快速路径（~40% 请求）**
```
Client → FastAPI → Triage(LLM) → DirectTool → MCP Server → END
延迟预算: ~500ms = Triage(200ms) + MCP(200ms) + 其他(100ms)
```

**路径 B：完整诊断（~50% 请求）**
```
Client → FastAPI → Triage(LLM) → Planning(LLM+RAG) → Diagnostic(LLM+MCP*3)
  → Report(LLM) → KnowledgeSink → END
延迟预算: ~8s = Triage(200ms) + Planning(1.5s) + Diagnostic(4s) + Report(1.5s) + 其他(800ms)
```

**路径 C：告警聚合（~10% 请求）**
```
Client → FastAPI → Triage(LLM) → AlertCorrelation(LLM) → Planning → Diagnostic → Report → END
延迟预算: ~12s = 路径 B + AlertCorrelation(2s)
```

### 3.3 压测分层策略

```
┌───────────────────────────────────────────────┐
│ L5: 端到端场景压测                              │ ← 最接近真实场景
├───────────────────────────────────────────────┤
│ L4: Agent 图编排压测                            │ ← 9 节点状态流转
├───────────────────────────────────────────────┤
│ L3: 子系统压测 (LLM / RAG / MCP)               │ ← 单组件极限
├───────────────────────────────────────────────┤
│ L2: API 层压测 (FastAPI / Go Fiber)            │ ← HTTP 层
├───────────────────────────────────────────────┤
│ L1: 基础设施压测 (PG/Redis/ES/Milvus/Neo4j)    │ ← 依赖服务
└───────────────────────────────────────────────┘
```

**压测顺序**: L1 → L2 → L3 → L4 → L5（自底向上，先确保底层没问题）

## 4. 核心压测点清单

### 4.1 按风险等级分类

**🔴 P0 — 必须压测（阻塞上线）**

| # | 压测点 | 子系统 | 风险描述 |
|---|--------|--------|----------|
| 1 | LLM API 延迟长尾 | LLM Client | P99 可能 >30s，阻塞整个诊断链路 |
| 2 | LLM Provider 熔断切换 | Provider Manager | 主 Provider 挂了能否自动切换 |
| 3 | MCP 工具并发调用 | MCP Client | Semaphore(5) + 连接池(100) 是否够用 |
| 4 | Diagnostic 自环爆炸 | Agent Graph | 自环 ≤5 轮，超过后能否正确退出 |
| 5 | 告警风暴并发 | /alert API | 100 条告警同时到达 |
| 6 | 内存泄漏 | 全链路 | 长时间运行内存是否持续增长 |
| 7 | Token 预算耗尽 | Token Budget | 超预算后是否正确降级 |
| 8 | Redis 故障降级 | HITL/Cache/RBAC | Redis 挂了所有 fallback 是否工作 |

**🟡 P1 — 应该压测（影响稳定性）**

| # | 压测点 | 子系统 | 风险描述 |
|---|--------|--------|----------|
| 9 | RAG 检索单路失败 | HybridRetriever | Dense 挂了 Sparse 能否兜底 |
| 10 | MCP 缓存命中率 | MCP Client L1+L2 | 诊断循环中重复工具调用 |
| 11 | Go 限流精度 | RateLimit MW | 100 RPS 全局 + 10 RPS/user |
| 12 | Go 熔断器状态机 | CircuitBreaker MW | CLOSED→OPEN→HALF_OPEN 转换正确性 |
| 13 | 图编译缓存 | Agent Graph | 首次编译 ~200ms 后续是否走缓存 |
| 14 | Session 内存 store | Agent Routes | 大量 session 是否导致 OOM |
| 15 | Prompt Injection 性能 | Security | 正则+关键词检测的延迟 |
| 16 | Context Compressor | Agent | 8000 字符截断的 CPU 开销 |
