## 2. 压测工具选型

### 2.1 主力工具对比

| 工具 | 语言 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|----------|
| **k6** | Go | 脚本化、指标丰富、低资源占用 | 不原生支持 Python | HTTP API 压测 |
| **Locust** | Python | Python 生态、分布式、自定义强 | 单 Worker 受 GIL 限制 | 复杂场景压测 |
| **wrk** | C | 极高性能、简单 | 脚本能力弱 | 基准测试 |
| **vegeta** | Go | 恒定速率攻击 | 场景编排弱 | 限流验证 |
| **pytest-benchmark** | Python | 集成到测试框架 | 只能测函数级 | 微基准测试 |

### 2.2 本项目工具组合

```
┌─────────────────────────────────────┐
│ k6              → HTTP API 压测      │  主力
│ Locust          → 端到端场景压测      │  补充
│ wrk             → 基线吞吐量          │  辅助
│ vegeta          → 限流精度验证         │  辅助
│ pytest-benchmark → 函数级微基准       │  内部
│ go test -bench  → Go 组件微基准       │  内部
│ redis-benchmark → Redis 基线          │  辅助
│ pgbench         → PostgreSQL 基线     │  辅助
│ esrally         → Elasticsearch 基线  │  辅助
└─────────────────────────────────────┘
```

## 3. Mock 服务设计

### 3.1 为什么需要 Mock

压测 AI Agent 系统的**核心难题**: LLM API 是外部依赖，有速率限制且费钱。

```
真实 LLM 调用:   DeepSeek 1000 RPM | OpenAI 500 RPM | Anthropic 400 RPM
压测需要的 QPS:  10-100 QPS
结论:            必须 Mock LLM，否则压测会被限流
```

### 3.2 LLM Mock Server

```python
# mock_llm_server.py — 模拟 LLM API 响应
"""
Mock LLM Server: 模拟 OpenAI/DeepSeek 兼容 API

启动: uvicorn mock_llm_server:app --port 8001
配置: LLM_PRIMARY_MODEL=mock-gpt-4o, OPENAI_API_BASE=http://localhost:8001/v1

支持的模拟行为:
1. 正常响应 (configurable latency)
2. 慢响应 (模拟 P99 长尾)
3. 错误响应 (模拟 Provider 故障)
4. 限流响应 (模拟 429 Too Many Requests)
"""

from fastapi import FastAPI, Request
from pydantic import BaseModel
import asyncio
import random
import time
import uuid

app = FastAPI(title="Mock LLM Server")

# ── 可配置的模拟参数 ──
class MockConfig:
    latency_ms: int = 200          # 正常延迟
    latency_p99_ms: int = 5000     # P99 延迟
    error_rate: float = 0.0        # 错误率 (0.0-1.0)
    rate_limit_rate: float = 0.0   # 限流率
    slow_rate: float = 0.05        # 慢请求比例

config = MockConfig()

# ── Mock 响应模板 ──
TRIAGE_RESPONSE = '''{
  "route": "direct_tool",
  "component": "hdfs",
  "confidence": 0.92,
  "tool_suggestion": "hdfs_namenode_status",
  "reasoning": "用户查询涉及 HDFS NameNode 状态，直接调用工具即可"
}'''

DIAGNOSTIC_RESPONSE = '''{
  "root_cause": "NameNode heap 使用率 95%，Full GC 频率过高",
  "confidence": 0.85,
  "evidence": ["heap_usage=95%", "full_gc_count=15/min"],
  "next_steps": ["增加 NameNode heap 到 8GB", "检查小文件合并"]
}'''

PLANNING_RESPONSE = '''{
  "hypotheses": [
    {"id": "H1", "description": "NameNode OOM", "probability": 0.7},
    {"id": "H2", "description": "网络分区", "probability": 0.2}
  ],
  "diagnostic_plan": [
    {"tool": "hdfs_namenode_status", "purpose": "检查 heap 使用率"},
    {"tool": "search_logs", "purpose": "搜索 GC 日志"}
  ]
}'''

REPORT_RESPONSE = '''# 诊断报告

## 根因分析
NameNode heap 使用率过高 (95%)，触发频繁 Full GC。

## 修复建议
1. 增加 NameNode heap: -Xmx8g
2. 执行小文件合并: hadoop archive

## 风险评估
- 修复风险: 低 (需重启 NameNode)
- 不修复风险: 高 (可能 OOM 导致集群不可用)
'''

RESPONSE_MAP = {
    "triage": TRIAGE_RESPONSE,
    "diagnostic": DIAGNOSTIC_RESPONSE,
    "planning": PLANNING_RESPONSE,
    "report": REPORT_RESPONSE,
}

@app.post("/v1/chat/completions")
async def mock_chat_completion(request: Request):
    """OpenAI 兼容的 chat completion 端点."""
    body = await request.json()
    model = body.get("model", "mock-gpt-4o")

    # 模拟错误
    if random.random() < config.error_rate:
        return {"error": {"message": "Mock server error", "type": "server_error"}}

    # 模拟限流
    if random.random() < config.rate_limit_rate:
        from fastapi.responses import JSONResponse
        return JSONResponse(status_code=429, content={"error": {"message": "Rate limited"}})

    # 模拟延迟
    if random.random() < config.slow_rate:
        await asyncio.sleep(config.latency_p99_ms / 1000)
    else:
        await asyncio.sleep(config.latency_ms / 1000)

    # 根据消息内容选择响应
    messages = body.get("messages", [])
    last_msg = messages[-1]["content"] if messages else ""
    response_content = TRIAGE_RESPONSE  # 默认

    for key, template in RESPONSE_MAP.items():
        if key in last_msg.lower():
            response_content = template
            break

    prompt_tokens = sum(len(m.get("content", "")) // 4 for m in messages)
    completion_tokens = len(response_content) // 4

    return {
        "id": f"chatcmpl-{uuid.uuid4().hex[:8]}",
        "object": "chat.completion",
        "created": int(time.time()),
        "model": model,
        "choices": [{
            "index": 0,
            "message": {"role": "assistant", "content": response_content},
            "finish_reason": "stop",
        }],
        "usage": {
            "prompt_tokens": prompt_tokens,
            "completion_tokens": completion_tokens,
            "total_tokens": prompt_tokens + completion_tokens,
        },
    }

@app.post("/v1/config")
async def update_config(request: Request):
    """动态调整 Mock 参数（压测中实时注入故障）."""
    body = await request.json()
    if "latency_ms" in body:
        config.latency_ms = body["latency_ms"]
    if "error_rate" in body:
        config.error_rate = body["error_rate"]
    if "rate_limit_rate" in body:
        config.rate_limit_rate = body["rate_limit_rate"]
    if "slow_rate" in body:
        config.slow_rate = body["slow_rate"]
    return {"status": "ok", "config": vars(config)}

@app.get("/health")
async def health():
    return {"status": "healthy", "type": "mock_llm_server"}
```

### 3.3 MCP 工具 Mock 策略

Go MCP Server 已内置 Mock 实现（Sprint 2），无需额外 Mock：

| 工具 | Mock 行为 | 延迟 |
|------|----------|------|
| hdfs_namenode_status | 返回固定状态 JSON | 10-50ms |
| kafka_consumer_lag | 返回 Mock lag 数据 | 10-50ms |
| search_logs | 返回模拟日志行 | 20-100ms |
| es_cluster_health | 返回 green/yellow/red 状态 | 10-30ms |
| yarn_queue_status | 返回队列利用率 | 10-30ms |
| restart_service | 返回 "simulated restart" | 100-500ms |
