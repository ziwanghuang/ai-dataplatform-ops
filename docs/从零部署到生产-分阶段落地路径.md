### 2.2 改造 Go 工具：Mock → 真实 HTTP 调用

以 HDFS NameNode 状态为例，当前 `go/internal/tools/hdfs/tools.go` 是 Mock 实现。改造策略：

**改造原则**：每个工具保留 Mock fallback，通过配置决定用哪个。

```go
// go/internal/tools/hdfs/tools.go 改造示例
func (t *NamenodeStatusTool) Execute(ctx context.Context, params map[string]interface{}) (*protocol.ToolResult, error) {
    cfg := config.Get()
    
    // 有真实集群配置时，走真实 HTTP 调用
    if cfg.Clusters.HDFS.NamenodeURL != "" {
        return t.fetchRealStatus(ctx, cfg.Clusters.HDFS.NamenodeURL)
    }
    
    // 无配置时，返回 Mock 数据
    return t.mockStatus(), nil
}

func (t *NamenodeStatusTool) fetchRealStatus(ctx context.Context, namenodeURL string) (*protocol.ToolResult, error) {
    // 调用 NameNode JMX API
    resp, err := http.Get(namenodeURL + "/jmx?qry=Hadoop:service=NameNode,name=NameNodeInfo")
    if err != nil {
        return nil, fmt.Errorf("namenode jmx query failed: %w", err)
    }
    defer resp.Body.Close()
    
    body, _ := io.ReadAll(resp.Body)
    return protocol.TextResult(string(body)), nil
}
```

## Phase 3：RAG 知识库 + 知识沉淀（1-2 天）

> **目标**：诊断时能检索历史案例和运维文档；诊断结果自动沉淀回知识库。

### 3.1 选择 RAG 后端策略

你已有 ES 集群，可以直接复用：

| RAG 组件 | 推荐方案 | 原因 |
|----------|---------|------|
| **稀疏检索 (BM25)** | ✅ 直接用你的 ES 集群 | 零额外成本，BM25 对运维文档效果好 |
| **稠密检索 (向量)** | ⚠️ Phase 3 先用 Mock | Milvus 需要额外部署，可后续加 |
| **图检索 (Neo4j)** | ⚠️ Phase 4+ 再加 | 组件拓扑有价值但不是 MVP 必需 |

### 4.5 Session Store 内存泄漏修复（P0）

`routes/agent.py` 的 `_session_store` 是内存 dict，无 TTL 清理：

```python
# 修改 routes/agent.py — 用 OrderedDict + 容量限制替换
from collections import OrderedDict
import time

_session_store: OrderedDict[str, dict[str, Any]] = OrderedDict()
_SESSION_MAX = 1000
_SESSION_TTL = 3600  # 1 小时

def _cleanup_sessions():
    """清理过期 session."""
    now = time.time()
    while _session_store:
        _, oldest = next(iter(_session_store.items()))
        if now - oldest.get("created_at", 0) > _SESSION_TTL:
            _session_store.popitem(last=False)
        else:
            break
    while len(_session_store) > _SESSION_MAX:
        _session_store.popitem(last=False)
```

## 附录 B：常见问题

### Q1: 没有 LLM Key 能跑吗？

可以。Triage 的规则引擎路径（15 条正则）不需要 LLM，简单查询会走 DirectTool 快速路径。但复杂诊断（需要 LLM 推理）会失败。

### Q2: 必须用 Docker 吗？

不是。Phase 0-2 完全可以裸机运行（Python venv + Go binary）。Docker 只在 Phase 5 部署时才需要。

### Q3: Milvus/Neo4j 必须部署吗？

不是。代码中所有 Retriever 都有 Mock fallback：
- 无 Milvus → DenseRetriever 返回空结果（不影响流程）
- 无 Neo4j → GraphRAGRetriever 返回空结果
- 有 ES 就够用（BM25 对运维文档效果已经不错）

### Q4: 怎么确认工具连接的是真实集群而不是 Mock？

看 Go MCP Server 的日志。真实调用会有 HTTP 请求日志和延迟：

```
{"level":"info","tool":"hdfs_namenode_status","latency_ms":45,"source":"real","msg":"tool executed"}
```

Mock 调用延迟 <1ms 且 source 为 "mock"。

### Q5: 生产环境建议的资源配置？

| 组件 | 最低配置 | 推荐配置 |
|------|---------|---------|
| Agent Service (Python) | 2C4G | 4C8G |
| MCP Server (Go) | 1C2G | 2C4G |
| PostgreSQL | 1C2G | 2C4G |
| Redis | 1C1G | 1C2G |
| 总计 | 5C9G | 9C18G |

### Q6: 如何从 DeepSeek 切换到 GPT-4o？

```bash
# .env
LLM_PRIMARY_MODEL=gpt-4o-2024-11-20     # 诊断/规划
LLM_TRIAGE_MODEL=deepseek-chat           # 分诊保持 DeepSeek（便宜）
LLM_OPENAI_API_KEY=sk-xxx
```

`ModelRouter` 会自动按任务类型路由到对应模型。
