## 一、架构追问 & 设计挑战（Q101-Q115）

### Q101 🟡 系统的单点故障有哪些？怎么消除？
> Agent 本身（K8s 多副本）、Redis（Sentinel/Cluster）、Milvus（分布式部署）、LLM API（多供应商）。最大单点是 LLM API，通过 ModelRouter 备用模型缓解。

### Q102 🔴 如果要支持多租户（不同团队独立使用），架构怎么改？
> State 加 tenant_id、RBAC 加租户隔离、RAG 按租户分 collection、MCP 工具加租户配额。核心改动在数据层隔离，Agent 逻辑不变。

### Q103 🔴 这个系统能水平扩展吗？瓶颈在哪？
> Agent 无状态可水平扩，瓶颈在：(1) LLM API 速率限制 (2) Milvus 单 collection 写入吞吐 (3) Neo4j 单机写。解法：LLM 多 Key 轮询、Milvus 分片、Neo4j 读写分离。

### Q104 🟡 为什么用 FastAPI 不用 Flask/Django？
> 原生 async、自动 OpenAPI 文档、Pydantic 原生集成（和 Agent 的 schema 统一）、SSE 支持好。Django 太重，Flask 不原生 async。

### Q105 🔴 如果延迟要求从 30s 降到 5s，你怎么优化？
> (1) 规则引擎覆盖率从 80%→95%（减少 LLM 调用）(2) 流式输出（边诊断边返回）(3) 小模型替代（DeepSeek → 本地 7B）(4) 预计算常见故障的诊断模板 (5) 工具调用预热连接池。

### Q106 🟡 SSE 和 WebSocket 你选了哪个？为什么？
> 当前 /diagnose 用同步响应。设计中支持 SSE（服务端单向推送足够，不需要双向通信）。WebSocket 更适合聊天场景，运维诊断是请求-响应模式。

### Q107 🔴 如果要同时支持 100 个并发诊断请求，系统怎么表现？
> 瓶颈在 LLM API（假设 RPM=60）。方案：请求队列 + 优先级调度（critical 优先）+ 低优先级请求走规则引擎降级。Agent 节点是 async 的，不阻塞线程。

### Q108 🟡 为什么用 TypedDict 做 AgentState 而不是 Pydantic BaseModel？
> LangGraph 要求 State 是 TypedDict（图编译时做类型推断）。Pydantic 可以用但有序列化开销。TypedDict 是零开销的类型标注，运行时就是普通 dict。

### Q109 🔴 有没有考虑过事件驱动架构？比如用 Kafka 做 Agent 之间的消息传递？
> 考虑过，但过度设计。当前 9 个节点在同一进程内通过 State 传递数据，延迟 ~0。引入 Kafka 增加 ~10ms 延迟 + 运维复杂度。适合多 Agent 分布式部署场景，当前单进程不需要。

### Q110 🟡 微服务拆分的边界在哪？Agent 和 MCP Server 已经拆了，还需要继续拆吗？
> 当前两服务够用。如果要继续拆：RAG Engine 可独立（被多个 Agent 共享）、HITL 审批可独立（对接企微/钉钉）。不建议拆太细——分布式事务复杂度会吞噬收益。

### Q111 🔴 你的 Mock 降级图和正式图的行为差异有多大？怎么保证降级不引入新 bug？
> Mock 图是 5 节点线性（Triage→Planning→Diagnostic→Report→KnowledgeSink），跳过条件路由和自环。差异：无 HITL、无 AlertCorrelation、无迭代。通过共享相同的 BaseAgentNode 模板和单元测试保证节点级行为一致。

### Q112 🟡 为什么选 Monorepo 而不是 Multi-repo？
> 一个项目两种语言，Monorepo 好处：(1) 统一版本管理 (2) 一个 PR 可以同时改 Python 和 Go (3) 共享 CI/CD 配置 (4) 文档和代码放一起。Multi-repo 在团队 >5 人时才有优势。

### Q113 🔴 如果面试官说"9 个节点太多了，LangChain 一个 chain 就搞定"，怎么回应？
> 一个 chain 能跑通 happy path，但处理不了：(1) 条件分流（简单查询 vs 复杂诊断）(2) 自环（假设-验证迭代）(3) 人工审批门控 (4) 独立降级。运维系统的核心不是"能跑通"而是"出错时可控"。

### Q114 🟡 数据流和控制流是怎么分离的？
> 数据流：State dict 在节点间传递（evidence、hypotheses 等）。控制流：条件路由函数决定下一个节点。分离的好处：路由逻辑变更不影响数据处理，反之亦然。

### Q115 🔴 如果要加一个"协作诊断"功能（多人同时参与），架构怎么改？
> (1) Session 级别加锁（防止并发写 State）(2) WebSocket 广播诊断进度 (3) HITL 扩展为多人投票 (4) State 加 participants 字段。核心挑战是并发冲突和一致性。

## 二、LLM 深度追问（Q116-Q130）

### Q116 🔴 DeepSeek-V3 和 GPT-4o 在运维诊断上的实际差异？
> DeepSeek-V3：成本低 10x、中文好、结构化输出稳定。GPT-4o：复杂推理链更强、多步诊断少跑偏。实测：Triage 两者差异 <3%，Diagnostic 差异 ~10%（GPT-4o 更准）。

### Q117 🟡 Prompt 工程在这个项目里有多重要？举个具体优化的例子
> 极其重要。例：Diagnostic prompt 从"请分析根因"改为"列出 3 个假设，每个假设给出支持/反驳证据，最后选择置信度最高的"→ 诊断质量提升 ~20%。结构化 prompt 比自由 prompt 效果好得多。

### Q118 🔴 如果 LLM 返回的 JSON 格式不对（即使用了 instructor），怎么处理？
> instructor 的 retry 机制：(1) 解析失败自动将错误信息反馈给 LLM (2) 重试最多 3 次 (3) 3 次都失败 → 降级返回规则引擎结果。实测 instructor + DeepSeek-V3 的一次性解析成功率 >95%。

### Q119 🟡 system prompt 和 user prompt 的划分原则是什么？
> system prompt：角色定义 + 工具描述 + RAG 文档（稳定不变，利用 Prompt Caching）。user prompt：用户输入 + 当前诊断上下文（每次不同）。划分原则：变化频率。

### Q120 🔴 Chain-of-Thought 和 ReAct 你用了哪种？为什么？
> 本质是 ReAct 的变体：Thought（LLM 生成假设）→ Action（调用工具）→ Observation（分析证据）→ 循环。但不是 ReAct 原版的单步循环——Planning 和 Diagnostic 分离，Plan 阶段一次性生成多步计划。

### Q121 🟡 Few-shot 示例在 prompt 中怎么用的？
> Triage prompt 包含 3 个示例（HDFS 空间查询、Kafka lag 告警、复杂诊断请求），帮助 LLM 理解输出 schema。Diagnostic prompt 包含 1 个完整的假设-验证循环示例。示例选择标准：覆盖不同 intent_type。

### Q122 🔴 Temperature 怎么设？不同节点不一样吗？
> Triage: 0.0（需要确定性分类）。Diagnostic: 0.1（允许微小创造性生成假设）。Planning: 0.2（假设多样性）。Report: 0.0（格式化输出）。原则：决策类用 0，生成类用 0.1-0.2。

### Q123 🟡 如果用户用英文提问，系统怎么处理？
> BGE-M3 支持多语言 embedding、DeepSeek-V3 支持中英。不做语言检测——直接处理。规则引擎用正则匹配，也覆盖了英文关键词（"HDFS space"）。

### Q124 🔴 长上下文场景（诊断过程中积累了大量工具输出），怎么防止 LLM 遗忘早期信息？
> (1) ContextCompressor 分级截断（保留关键 evidence，截断冗余日志）(2) 每轮循环生成 summary 注入下一轮 (3) 最终分析时只传 evidence 摘要，不传原始工具输出。

### Q125 🟡 Prompt 模板独立化的好处是什么？
> 从 Agent 节点代码中分离到 prompts/ 目录：(1) Prompt 可以独立版本控制和回滚 (2) 非开发人员（运维专家）可以改 prompt (3) A/B 测试不同 prompt 版本 (4) 减少 Agent 节点代码复杂度。

### Q126 🔴 如果要支持多轮对话（用户追问"为什么是这个根因？"），怎么做？
> 当前是单轮请求-响应。多轮需要：(1) 对话历史存入 State (2) LLM 调用时注入历史 (3) 区分"新诊断"和"追问" (4) 追问时不重新跑全流程，只对 Report 做补充分析。

### Q127 🟡 模型输出的 confidence 分数可信吗？怎么校准？
> 不完全可信，LLM 倾向于过度自信。校准方法：(1) 评测数据集上统计 confidence 和实际准确率的映射 (2) Platt Scaling 校准 (3) 后置修正规则（连续 3 个 confidence=0.95 的输出强制降到 0.8）。

### Q128 🔴 如果 LLM 的结论和工具采集的数据矛盾，系统怎么处理？
> evidence 优先于 LLM 结论。DiagnosticOutput 的 model_validator 检查：如果 root_cause 描述和 evidence 中的数值矛盾（如说"CPU 正常"但 evidence 显示 CPU 95%）→ 标记为 unverified，降低 confidence。

### Q129 🟡 你怎么看 Function Calling vs Tool Use 的区别？
> Function Calling（OpenAI 命名）= Tool Use（Anthropic 命名），本质相同。区别在实现细节：OpenAI 用 functions 参数，Anthropic 用 tools 参数。litellm 统一了这层差异。

### Q130 🔴 如果要接入本地部署的开源模型（如 Qwen-72B），需要改什么？
> (1) 本地部署 vLLM/Ollama 提供 OpenAI 兼容 API (2) litellm 配置：`model="openai/qwen-72b", api_base="http://localhost:8000"` (3) ModelRouter 路由表加新条目。零业务代码改动。

## 三、RAG & 知识工程追问（Q131-Q145）

### Q131 🟡 如果检索返回的 top-5 文档全是不相关的，Agent 怎么处理？
> LLM 分析 evidence 时识别"文档不相关" → confidence 保持低值 → 走降级路径（基于 LLM 通用知识诊断，但标注"无运维知识库支持"）。

### Q132 🔴 Embedding 模型的 fine-tune 考虑过吗？
> 考虑过但投入产出比低。运维领域术语（"NN OOM"、"ISR 不足"）在 BGE-M3 预训练中已有覆盖。fine-tune 需要 10K+ 标注对，当前数据量不够。更划算的方案是丰富 sparse 检索的同义词词典。

### Q133 🟡 chunk_size=512 怎么定的？有什么影响？
> 经验值。太小（128）：语义碎片化，检索到的 chunk 缺乏上下文。太大（2048）：检索精度下降，不相关内容混入。512 在运维文档（通常按段落/步骤组织）上的命中率最高。

### Q134 🔴 GraphRAG 的 3 步检索和直接用 Dense 检索比，优势在哪？具体例子？
> Dense 能找到"NN OOM 的处理方法"，但找不到因果链"NN OOM ← 元数据膨胀 ← 小文件过多 ← 任务配置错误"。GraphRAG 能沿因果链追溯上游原因，帮助 Diagnostic 生成更准确的假设。

### Q135 🟡 RRF 的 k=60 是怎么调的？调参过程是什么？
> 在 12 条 RAG 测试集上跑 grid search（k=10,20,40,60,80,100），比较 MRR@5。k=60 时 MRR 最高（0.78）。但样本量太小，更严格的做法是 500+ 测试集 + cross-validation。

### Q136 🔴 如果运维知识库有冲突文档（旧文档说"重启 NN"，新文档说"不要重启 NN"），怎么处理？
> (1) 文档加时间戳，Reranker 优先最新 (2) 检索结果如果包含矛盾信息，LLM 分析时会指出矛盾 (3) 长期方案：知识图谱中标记文档版本和替代关系。

### Q137 🟡 增量索引的去重机制是什么？
> Milvus upsert 基于 doc_id（MD5(content) 或自定义 ID）。相同 doc_id 的新版本覆盖旧版本。ES 用 _id 字段做同样的事。

### Q138 🔴 如果要支持图片检索（比如运维截图、监控图表），架构怎么改？
> (1) 多模态 embedding 模型（如 CLIP 或 BGE-M3 多模态版本）(2) Milvus 存图片 embedding (3) LLM 换多模态模型（GPT-4V）分析图片。核心改动在 embedding 和 LLM 调用层。

### Q139 🟡 Reranker 的延迟开销值得吗？有量化数据吗？
> Reranker 对 top-20 重排耗时 ~200ms。MRR@5 从 0.72（无 rerank）提升到 0.82。在运维诊断场景（总延迟 10-30s），200ms 完全可接受。如果是实时搜索场景（<100ms），可以跳过。

### Q140 🔴 知识闭环（KnowledgeSink → RAG → 更好的诊断）有没有负面效应？
> 有。如果一次误诊被沉淀到知识库 → 后续类似问题检索到错误知识 → 误诊放大。防御：(1) 沉淀前人工审核（PROD 环境）(2) 沉淀的知识标记 confidence (3) 定期清理低 confidence 条目。

### Q141 🟡 ES 的 ik_smart 分词器为什么选这个？
> 中文运维文档需要中文分词。ik_smart 是最粗粒度分词（比 ik_max_word 少歧义），适合检索场景。"NameNode 内存不足" → ["NameNode", "内存", "不足"]，而不是 ["Name", "Node", "内", "存"]。

### Q142 🔴 如果 Neo4j 数据量到 100 万节点，Cypher 查询性能怎么保证？
> (1) 索引：对 name、type 字段建 B-tree 索引 (2) 查询限制：LIMIT + 跳数限制（最多 3 跳）(3) 缓存：热点子图缓存 (4) 读写分离。当前 Mock 数据量远未到瓶颈。

### Q143 🟡 文档预处理（清洗、分块、metadata 提取）的流程是什么？
> (1) 读取 Markdown/文本 (2) 按标题/空行切分段落 (3) 合并过短段落（<50 字）(4) 提取 metadata（标题、组件名、日期）(5) 生成 embedding (6) 写入 Milvus + ES。

### Q144 🔴 RAG 和 fine-tune 的选择标准是什么？为什么不 fine-tune？
> RAG 优势：(1) 知识可动态更新（不需要重新训练）(2) 可审计（看到用了哪个文档）(3) 成本低（不需要 GPU 训练）。fine-tune 适合：领域术语理解、输出格式固定化。运维知识更新频繁，RAG 更合适。

### Q145 🟡 如果用户问的问题完全超出运维领域（如"今天天气怎样"），系统怎么处理？
> Triage 规则引擎不命中 → LLM 分诊 → intent_type=out_of_scope → 直接返回"我是运维诊断 Agent，无法回答非运维问题" → 不进入诊断流程。

## 四、Go MCP Server 深度（Q146-Q158）

### Q146 🟡 Go 中间件的 interface 怎么设计的？
> `type Middleware func(ToolHandler) ToolHandler`，ToolHandler 是 `func(ctx, request) (response, error)`。洋葱模型——每个 middleware wrap 下一层。

### Q147 🔴 Go 的 Context 传递在中间件链中怎么用的？
> 每层中间件从 ctx 读取/写入值。Tracing 写入 span → Audit 读取 span → Timeout 用 context.WithTimeout 包装。ctx cancel 时所有层级同时感知。

### Q148 🟡 MCP Server 的 JSON-RPC 消息格式是什么？
> 标准 JSON-RPC 2.0：`{jsonrpc: "2.0", method: "tools/call", params: {name: "hdfs_check", arguments: {...}}, id: 1}`。响应：`{jsonrpc: "2.0", result: {...}, id: 1}`。

### Q149 🔴 如果要给 Go MCP Server 加一个新工具，最少改几个文件？
> 最少 1 个文件：在 tools/ 下新建工具实现文件，包含 Handler 函数 + init() 中调用 registry.Register()。自动注册，无需改 main.go 或其他文件。

### Q150 🟡 Go 侧的错误处理策略是什么？
> (1) 业务错误：返回 JSON-RPC error（code + message）(2) 系统错误：中间件层 recover panic → 返回 InternalError (3) 超时：ctx.Done() → 返回 TimeoutError。所有错误都带 trace_id。

### Q151 🔴 Fiber 的 fasthttp 和标准 net/http 的具体区别？什么场景下反而不如 net/http？
> fasthttp：零内存分配、连接复用激进、不支持 HTTP/2。不如 net/http 的场景：需要 HTTP/2、需要和标准库中间件兼容（gin/echo）、需要 WebSocket（fasthttp 的 WebSocket 支持差）。

### Q152 🟡 令牌桶限流的参数怎么设的？
> rate=100 req/s，burst=200。WHY：MCP Server 面对的是 Agent（不是直接用户），Agent 并发一般 <50，100 QPS 留有余量。burst=200 应对 Diagnostic 的并行工具调用峰值。

### Q153 🔴 如果 Go MCP Server 要支持流式响应（比如实时返回日志），怎么改？
> (1) 改用 SSE 或 WebSocket（当前 JSON-RPC 是请求-响应）(2) 工具 Handler 返回 channel 而不是 response (3) 中间件需要支持流式传递。或者用 MCP 协议的 notifications 机制。

### Q154 🟡 Go 的 goroutine 泄漏怎么防？
> (1) 所有 goroutine 关联 context，cancel 时退出 (2) select + ctx.Done() 兜底 (3) pprof 定期检查 goroutine 数量 (4) graceful shutdown 等待所有 goroutine 完成。

### Q155 🔴 为什么不用 gRPC 而用 HTTP+JSON？
> MCP 协议标准是 HTTP+JSON-RPC（Anthropic 定义的）。gRPC 性能更好（protobuf + HTTP/2），但：(1) 不符合 MCP 标准 (2) 调试不如 JSON 直观 (3) 浏览器客户端不直接支持 gRPC。

### Q156 🟡 MCP 工具的 inputSchema 是怎么生成的？
> 手写 JSON Schema（每个工具注册时声明）。Go 结构体 → 反射生成 schema 也可以，但当前工具数量少（3 个 mock），手写更精确。生产工具多了应该用代码生成。

### Q157 🔴 如果一个工具执行了一半宕机（比如写了一半配置），怎么保证一致性？
> 当前设计：工具操作要么幂等（查询类），要么有前置检查（修改类先 dry-run）。非幂等写操作需要：(1) 预写日志（WAL）(2) 操作回滚机制 (3) HITL 确认后才真正提交。这是 Phase 2 的增强。

### Q158 🟡 Go 的 zerolog 为什么选这个日志库？
> 零内存分配、JSON 结构化输出、性能全球最快的 Go 日志库之一。和 Python 端 structlog 的 JSON 格式兼容，方便统一日志聚合（ELK/Loki）。

## 五、安全 & 合规深度（Q159-Q170）

### Q159 🔴 Prompt 注入的误报率是多少？怎么降低？
> 当前正则+关键词的误报率估计 ~5%（"ignore this error" 可能被误判）。降低方案：(1) 白名单常见运维短语 (2) 上下文感知（在运维语境中"ignore"不是注入）(3) LLM 做二次判断。

### Q160 🟡 如果 RBAC 用户信息被篡改（伪造 admin 角色），怎么防？
> 当前设计依赖调用方传 user_id，查本地权限表。防篡改：(1) JWT 签名验证 (2) 每次请求验证 token 有效性 (3) 权限表从可信源（LDAP/SSO）加载，不接受客户端自报角色。

### Q161 🔴 如果 LLM 提供商被攻破（数据泄露），你的系统受什么影响？
> 风险：(1) prompt 中的运维信息泄露（已脱敏，但仍有风险）(2) 对话历史泄露。防御：(1) 敏感数据脱敏后才传 LLM (2) 不在 prompt 中放凭据 (3) 可选本地模型替代。

### Q162 🟡 审计日志记录哪些信息？保留多久？
> 记录：who（user_id）、what（tool_name + params）、when（timestamp）、result（success/fail）、trace_id。保留策略：热数据 7 天（ES）、冷数据 90 天（对象存储）。合规需求可延长到 1 年。

### Q163 🔴 如果国内要求数据不出境，LLM 调用怎么处理？
> (1) 路由到国内模型（DeepSeek、Qwen）(2) 敏感场景用本地部署的开源模型 (3) ModelRouter 加地域约束（某些任务类型只允许国内模型）。litellm 支持自定义 api_base 指向私有化部署。

### Q164 🟡 MCP 工具的操作审计和 RBAC 检查的顺序？
> 先 RBAC 检查 → 通过 → 记录审计日志 → 执行。不通过 → 也记录审计日志（记录拒绝事件）→ 返回 403。顺序不能反——执行前必须先鉴权。

### Q165 🔴 红蓝对抗你会怎么设计？测试哪些攻击面？
> 攻击面：(1) 直接 Prompt 注入（用户输入）(2) 间接注入（日志/工具返回中的恶意内容）(3) RBAC 绕过（权限提升）(4) DoS（大量复杂请求耗尽 LLM 配额）(5) 数据泄露（通过 LLM 提取 system prompt）。

### Q166 🟡 你的脱敏是在哪一层做的？会不会影响诊断准确性？
> 在 LLM 调用前做。影响：IP 脱敏后 LLM 无法判断"哪个节点故障"。解法：脱敏保留最后两位（10.x.x.42 → [IP:*.*.*. 42]），足够区分节点但不暴露完整 IP。

### Q167 🔴 GDPR/个人信息保护法对这个系统的要求是什么？
> 运维日志可能包含用户名、IP 等个信。要求：(1) 脱敏处理（已实现）(2) 数据最小化（只传必要信息给 LLM）(3) 数据主体权利（支持删除用户相关数据）(4) 数据处理记录（审计日志）。

### Q168 🟡 content_safety 检查是在 Triage 前还是后？
> 在 Triage 前（PromptInjectionDefense 是 Agent 的第一道关卡）。理由：不安全的输入不应该到达任何业务逻辑——先消毒再处理。

### Q169 🔴 如果攻击者通过慢速请求耗尽 LLM Token 配额（经济攻击），怎么防？
> (1) 每用户/每小时 Token 配额 (2) session 级 Token 预算（已实现 50K 上限）(3) 异常检测：单用户短时间大量请求 → 限流 (4) 高成本操作（Diagnostic 自环）的迭代上限。

### Q170 🟡 不同环境（DEV/STAGING/PROD）的安全策略差异？
> DEV：HITL 自动审批、宽松限流、允许所有工具。STAGING：HITL 需审批、中等限流、禁止 critical 工具。PROD：严格 HITL、严格限流、critical 工具需双人审批 + 录屏。

## 六、工程细节 & 代码质量（Q171-Q182）

### Q171 🟡 108 个单元测试的覆盖率是多少？哪些模块没覆盖？
> 行覆盖率约 65%。未覆盖：(1) 生产后端的真实连接逻辑（需要基础设施）(2) LLM 实际调用（Mock 了 litellm）(3) 部分异常分支。核心业务逻辑覆盖率 >85%。

### Q172 🔴 你怎么 Mock LLM 调用的？Mock 有什么局限性？
> 用 unittest.mock.patch 替换 litellm.completion，返回预设的结构化响应。局限：(1) 不测试 LLM 的实际输出质量 (2) 不测试 Prompt 效果 (3) 不测试 instructor 的重试机制。这些需要集成测试覆盖。

### Q173 🟡 测试数据怎么管理的？fixture 还是 factory？
> 组合使用：(1) conftest.py 中的 pytest fixture 提供公共测试状态 (2) 每个测试文件的辅助函数构造特定数据。没有用 factory_boy——数据结构不复杂，TypedDict 直接构造足够。

### Q174 🔴 代码中有哪些设计模式？除了模板方法还有什么？
> (1) 模板方法（BaseAgentNode）(2) 策略（ModelRouter 的路由策略）(3) 注册器（ToolRegistry）(4) 装饰器（中间件链）(5) 工厂（create_initial_state）(6) 观察者（指标上报）(7) 代理（MCP Client 代理远程工具）。

### Q175 🟡 Python 的 type hints 用到什么程度？mypy 严格模式？
> 全项目 type hints，TypedDict + Pydantic + typing。mypy 配置 strict=false（部分第三方库缺类型存根）。pre-commit 跑 mypy 检查，增量校验不阻塞。

### Q176 🔴 structlog 和标准 logging 的区别？为什么选 structlog？
> structlog：(1) 结构化 JSON（方便 ELK 解析）(2) 支持 context binding（一次设 trace_id，后续日志自动带上）(3) 可插拔 processor 链。标准 logging 的 format string 在日志聚合场景不好解析。

### Q177 🟡 Python 异步代码（async/await）在项目中的使用范围？
> Agent 节点的 execute() 是 async（LLM 调用和工具调用都是 IO-bound）。FastAPI handler 是 async。MCP Client 的 httpx 是 async。同步代码：配置加载、规则引擎匹配（CPU-bound 且极快）。

### Q178 🔴 asyncio.gather 和 asyncio.TaskGroup 你选了哪个？为什么？
> asyncio.gather(return_exceptions=True)。TaskGroup（Python 3.11+）的好处是结构化并发，但缺点是一个 task 异常会取消全部。运维诊断需要部分工具失败不影响其他——gather 更合适。

### Q179 🟡 Pydantic v1 还是 v2？有什么区别？
> Pydantic v2（基于 Rust core，性能提升 5-50x）。区别：BaseSettings 从 pydantic 移到 pydantic-settings、validator 改为 field_validator / model_validator、.dict() 改为 .model_dump()。

### Q180 🔴 如果要把 Python 版本从 3.12 升到 3.14，有什么兼容性风险？
> pyproject.toml 声明 ^3.12。已在 3.14.3 上跑通 108 个测试。风险：(1) 某些第三方库可能不支持 3.14 (2) asyncio API 变更（3.12→3.14 变化不大）。实际验证过，无兼容性问题。

### Q181 🟡 Go 的错误处理风格——if err != nil 有什么改进？
> (1) 自定义 error 类型（AppError 带 code + message + trace）(2) 中间件统一错误格式化 (3) errors.Is/As 做错误类型判断。Go 1.22 没有 try/catch 也没有 Result<T>，只能靠纪律。

### Q182 🔴 项目中有没有竞态条件的风险？怎么避免？
> Agent 内部：单 session 串行执行，无竞态。跨 session：Redis 操作用 atomic（SETNX/INCR），RBAC 内存模式用 threading.Lock。Go 侧：registry 用 sync.RWMutex 保护读写。

## 七、场景题 & 案例分析（Q183-Q192）

### Q183 🔴 面试官给你一个场景：Kafka 消费者 lag 持续增长，走一遍你的系统诊断流程
> Triage: 关键词"kafka lag" → component=kafka, severity=high → Planning。Planning: 假设（消费者 GC/分区不均/生产速率暴增）。Diagnostic: 调用 kafka_consumer_lag + kafka_partition_info + kafka_producer_rate → 发现分区 leader 集中在一个 broker。Remediation: 建议 partition reassignment。

### Q184 🔴 场景：凌晨 3 点同时收到 50 条 HDFS 告警，系统怎么处理？
> AlertCorrelation: 30s 窗口聚合 50 条 → 1 组。组件推断: HDFS。紧急度: 取最高 severity。去重后触发 1 次诊断（而不是 50 次）。如果是 critical → 通知值班人员（HITL 超时不等审批，直接生成建议报告）。

### Q185 🔴 场景：用户报告"集群变慢了"但不知道是哪个组件，怎么诊断？
> Triage: 规则不命中，LLM 分诊 → intent_type=complex_diagnosis, component=unknown。Planning: 广撒网假设（HDFS IO 瓶颈/YARN 资源争抢/网络问题）。Diagnostic: 并行调用多组件巡检工具，缩小范围。可能需要 2-3 轮迭代。

### Q186 🟡 场景：用户问"帮我看看 HDFS 的空间还有多少"——这个走快速路径还是诊断路径？
> 快速路径。Triage 规则命中"HDFS 空间" → DirectToolNode → 直接调用 hdfs_space_check → 格式化返回结果。零 LLM Token、延迟 <1s。

### Q187 🔴 场景：诊断过程中发现是人为误操作导致的问题，系统怎么处理？
> Diagnostic 分析证据发现配置变更时间和故障时间吻合 → root_cause 标记为 human_error → Remediation 建议回滚配置（而不是系统修复）→ Report 记录变更人和变更内容 → KnowledgeSink 沉淀"该配置变更会导致 XX 问题"。

### Q188 🔴 场景：LLM 诊断说是"网络分区"，但实际是"磁盘满了"，系统有什么机制纠正？
> (1) evidence 中有磁盘使用率数据，model_validator 检查 root_cause 和 evidence 一致性 (2) 如果不一致，confidence 被降低 (3) 低 confidence → 触发第二轮迭代，生成新假设 (4) 最终 Report 标注 unverified 结论供人工复核。

### Q189 🟡 场景：HITL 审批超时了（运维人员没看到），怎么处理？
> 超时 300s → 自动拒绝（fail-safe 原则）→ Remediation 标记为 "approval_timeout" → Report 生成建议但标注"未获批准，仅供参考" → 通知运维人员有待处理事项。

### Q190 🔴 场景：两个运维工程师对同一个问题提交了不同的诊断请求，怎么处理？
> 当前设计：每个请求独立 session，各自跑完整链路。优化方案：检测相似请求（embedding 相似度 >0.95）→ 合并到同一 session → 结果共享。需要请求去重模块（Phase 2）。

### Q191 🟡 场景：Agent 诊断结果是"重启 YARN ResourceManager"，但 RM 正在做 failover，怎么办？
> HITL Gate 应拦截（RM 重启是 high 风险操作）。但更好的方案是：Remediation 执行前先调用 yarn_rm_status 检查当前状态，发现正在 failover → 建议等待 failover 完成而不是强制重启。

### Q192 🔴 场景：生产环境 Agent 突然响应变慢（P99 从 10s 涨到 60s），怎么排查？
> (1) 看 Grafana Dashboard: 哪个节点耗时增加 (2) Jaeger trace: 找到最慢的 span（通常是 LLM 调用或 MCP 工具）(3) 如果是 LLM: 检查 API 状态页、看是否触发了限流 (4) 如果是 MCP: 检查目标组件是否响应变慢 (5) 临时方案: 切换备用模型或增大超时。

## 八、跨领域 & 延伸问题（Q193-Q200）

### Q193 🟡 这个系统的思路能不能迁移到其他领域（如数据库运维、网络运维）？
> 完全可以。Agent 框架、HITL、安全层、可观测性都是通用的。需要改的：(1) MCP 工具换成目标领域工具 (2) RAG 知识库换内容 (3) Triage 规则适配 (4) 评测数据集重建。框架复用率 >70%。

### Q194 🔴 和 AWS GuardDuty / Azure Sentinel 等云安全 AI 产品相比，你的设计有什么差异？
> 它们是安全领域的 AI（威胁检测+响应）。本系统是运维领域的 AI（故障诊断+修复）。共同点：事件→分析→响应的流程。差异：本系统更垂直（大数据运维）、更透明（HITL+可解释性）、更轻量（单项目 vs 云服务）。

### Q195 🟡 如果团队有 5 个人一起开发这个项目，你怎么分工？
> (1) 1 人 Go MCP Server + 工具开发 (2) 1 人 Agent 框架 + LLM 集成 (3) 1 人 RAG + 知识工程 (4) 1 人 Web API + DevOps + 监控 (5) 1 人评测 + 安全。自己做架构设计 + Code Review + 关键路径实现。

### Q196 🔴 你觉得 AI Agent 在企业中落地的最大阻力是什么？
> 不是技术，是信任。(1) 运维人员担心 AI 误操作（→ HITL 解决）(2) 管理层看不到 ROI（→ 量化：平均诊断时间从 30min 降到 3min）(3) 安全团队担心数据泄露（→ 脱敏+本地模型）。技术可以迭代，信任需要时间积累。

### Q197 🟡 你项目的文档（21 篇实现细节）是怎么组织的？
> 按编号排序，每篇覆盖一个子系统。前 6 篇 Agent 核心，7-11 工具和集成，12-15 RAG+安全+HITL，16-21 可观测+评估+部署。10 篇核心文档做了深度扩充（~3000 行），每个设计决策有 WHY。

### Q198 🔴 如果这个项目开源，你觉得会有人用吗？需要做什么才能让别人用得起来？
> 需要：(1) Quick Start 指南（5 分钟跑通 Demo）(2) 至少 1 个真实组件对接示例 (3) Plugin 机制（方便添加新工具）(4) 配置化而非硬编码 (5) 丰富文档和示例。当前离"别人能用"还差 Quick Start 和真实工具。

### Q199 🔴 你用 AI（CodeBuddy/GitHub Copilot）辅助写了这个项目，怎么看 AI 辅助开发？
> 大幅提效：(1) 模板代码（测试、Pydantic model）生成快 3x (2) 文档扩充效率高 (3) 但架构设计、关键决策必须自己做。AI 是倍增器不是替代品。用 AI 写代码和用 AI 思考是两回事。

### Q200 🔴 最后一问：为什么这个项目值得我们给你 offer？
> 三个信号：(1) **系统设计能力**——不是调 API 拼 demo，而是设计了完整的工程化系统（分层架构、降级策略、安全体系、可观测性）(2) **领域深度**——4 年大数据运维经验不是背书，是每个设计决策的真实来源 (3) **执行力**——一个人从零到 108 个测试 + 21 篇文档 + 双语言代码，能独立扛复杂项目。
