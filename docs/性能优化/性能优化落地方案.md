## 3. P0 — 立即落地

### 3.1 SemanticCache 接入 LLMClient

**现状问题**

`python/src/aiops/llm/cache.py` 已完整实现 376 行的 `SemanticCache`，支持：
- L1 精确匹配（hash）
- L2 语义匹配（trigram cosine similarity > 0.92）
- 分任务类型 TTL、LRU+TTL 淘汰、CacheStats 统计

但 `python/src/aiops/llm/client.py` 的 `chat()` 和 `chat_structured()` 方法**完全没有调用缓存**：

```python
# 当前 client.py —— 无缓存集成
class LLMClient:
    async def chat(self, messages, model=None, ...):
        response = await litellm.acompletion(model=model, messages=messages, ...)
        return response.choices[0].message.content
```

这意味着相同/相似的 LLM 请求每次都要走 API 全量调用，白白浪费 Token 和延迟。

**目标**

- 相同请求 L1 命中：0ms（跳过 API）
- 相似请求 L2 命中：<5ms（跳过 API）
- 预估缓存命中率：运维场景 30-40%（相似告警、重复分诊）

**实现方案**

修改 `python/src/aiops/llm/client.py`：

```python
import structlog
from aiops.llm.cache import SemanticCache

logger = structlog.get_logger(__name__)


class LLMClient:
    def __init__(self, config=None):
        self._config = config or get_config()
        # 初始化语义缓存
        self._cache = SemanticCache(
            max_l1_size=2000,       # L1 精确匹配容量
            max_l2_size=500,        # L2 语义匹配容量
            default_ttl=300,        # 默认 5 分钟
        )

    async def chat(
        self,
        messages: list[dict],
        model: str | None = None,
        task_type: str = "general",
        temperature: float = 0.0,
        use_cache: bool = True,       # 新增：允许调用方跳过缓存
        **kwargs,
    ) -> str:
        model = model or self._config.llm.default_model

        # ---- 缓存查询 ----
        if use_cache and temperature == 0.0:
            cache_key = self._build_cache_key(messages, model, task_type)
            cached = self._cache.get(cache_key, task_type=task_type)
            if cached is not None:
                logger.info("llm_cache_hit", task_type=task_type, level="L1_or_L2")
                return cached

        # ---- 正常 API 调用 ----
        response = await litellm.acompletion(
            model=model,
            messages=messages,
            temperature=temperature,
            **kwargs,
        )
        content = response.choices[0].message.content

        # ---- 缓存写入 ----
        if use_cache and temperature == 0.0:
            self._cache.put(cache_key, content, task_type=task_type)

        return content

    async def chat_structured(self, messages, response_model, task_type="general", use_cache=True, **kwargs):
        """结构化输出也走缓存（缓存序列化后的 JSON）"""
        model = kwargs.pop("model", None) or self._config.llm.default_model

        if use_cache:
            cache_key = self._build_cache_key(messages, model, task_type)
            cached = self._cache.get(cache_key, task_type=task_type)
            if cached is not None:
                # 反序列化为 Pydantic model
                import json
                return response_model.model_validate_json(cached)

        result = await instructor_client.chat.completions.create(
            model=model,
            messages=messages,
            response_model=response_model,
            **kwargs,
        )

        if use_cache:
            self._cache.put(cache_key, result.model_dump_json(), task_type=task_type)

        return result

    def _build_cache_key(self, messages: list[dict], model: str, task_type: str) -> str:
        """构建缓存 key：model + task_type + 消息内容 hash"""
        import hashlib
        content = f"{model}:{task_type}:" + "|".join(
            f"{m.get('role', '')}:{m.get('content', '')}" for m in messages
        )
        return hashlib.sha256(content.encode()).hexdigest()

    def get_cache_stats(self) -> dict:
        """暴露缓存统计供监控使用"""
        return self._cache.stats()
```

**关键设计决策**

| 决策点 | 选择 | 原因 |
|-------|------|------|
| `temperature == 0.0` 才缓存 | 是 | 非零温度每次结果不同，缓存无意义 |
| `use_cache` 参数 | 是 | HITL 等场景需要跳过缓存获取最新结果 |
| 结构化输出缓存 JSON 序列化 | 是 | Pydantic model 不可直接 hash/序列化存缓存 |
| 缓存 key 包含 model + task_type | 是 | 避免不同模型/任务类型串缓存 |

**预期收益**

| 指标 | 优化前 | 优化后 | 提升 |
|------|-------|-------|------|
| 重复请求延迟 | 800ms-3s | <5ms | -99% |
| Token 消耗 | 100% | 60-70% | -30~40% |
| Triage 缓存命中率 | 0% | ~45% | — |
| Diagnostic 缓存命中率 | 0% | ~20% | — |

### 3.2 LLM 超时与熔断参数调优

**现状问题**

```python
# config.py 当前配置
class LLMConfig:
    timeout: int = 60           # 单次 LLM 调用超时 60s
    # 无 fallback 模型配置
```

```go
// go config.go
CircuitBreaker: CircuitBreakerConfig{
    MaxFailures: 3,
    Interval:    60 * time.Second,
    Timeout:     30 * time.Second,
}
```

**问题分析：**

1. **LLM 超时 60s 太长**：正常 LLM 响应 P99 < 5s，60s 意味着慢请求会挂住连接
2. **熔断恢复慢**：3 次超时 = 3×60s = 180s 才触发熔断，期间所有请求阻塞
3. **无 fallback**：API 不可用时直接报错，没有备用模型

**实现方案**

#### 3.2.1 Python 侧超时调优

修改 `python/src/aiops/core/config.py`：

```python
class LLMConfig(BaseModel):
    timeout: int = 15                    # 从 60s 降到 15s（P99 + 安全余量）
    connect_timeout: int = 5             # 新增：连接超时 5s
    stream_timeout: int = 30             # 新增：Streaming 超时（允许更长）
    daily_limit: int = 1_000_000         # 从 500K 提升到 1M

    # 新增：fallback 配置
    fallback_model: str = "deepseek-chat"    # 主模型不可用时的 fallback
    max_retries: int = 2                     # 最多重试 2 次
    retry_delay: float = 0.5                 # 重试间隔 0.5s
```

#### 3.2.2 LLMClient 超时 + 重试 + Fallback

修改 `python/src/aiops/llm/client.py`，新增重试与 fallback 逻辑：

```python
import asyncio
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type

class LLMClient:
    async def chat(self, messages, model=None, task_type="general", **kwargs):
        model = model or self._config.llm.default_model

        # 缓存查询（同 3.1）...

        # 带重试 + fallback 的 API 调用
        try:
            content = await self._call_with_retry(model, messages, **kwargs)
        except Exception as e:
            logger.warning("llm_primary_failed", model=model, error=str(e))
            # Fallback 到备用模型
            fallback = self._config.llm.fallback_model
            if fallback and fallback != model:
                logger.info("llm_fallback", from_model=model, to_model=fallback)
                content = await self._call_with_retry(fallback, messages, **kwargs)
            else:
                raise

        # 缓存写入（同 3.1）...
        return content

    async def _call_with_retry(self, model: str, messages: list, **kwargs) -> str:
        """带超时和重试的 LLM 调用"""
        last_error = None
        for attempt in range(self._config.llm.max_retries + 1):
            try:
                response = await asyncio.wait_for(
                    litellm.acompletion(
                        model=model,
                        messages=messages,
                        timeout=self._config.llm.timeout,
                        **kwargs,
                    ),
                    timeout=self._config.llm.timeout + 2,  # asyncio 超时比 litellm 超时多 2s
                )
                return response.choices[0].message.content
            except asyncio.TimeoutError:
                last_error = TimeoutError(f"LLM call timed out after {self._config.llm.timeout}s")
                logger.warning("llm_timeout", model=model, attempt=attempt + 1)
            except Exception as e:
                last_error = e
                logger.warning("llm_error", model=model, attempt=attempt + 1, error=str(e))

            if attempt < self._config.llm.max_retries:
                await asyncio.sleep(self._config.llm.retry_delay * (attempt + 1))

        raise last_error
```

### 3.3 Session Store 内存泄漏修复

**现状问题**

`python/src/aiops/web/routes/agent.py` 中：

```python
# 当前实现 —— 内存泄漏
_session_store: dict[str, dict] = {}  # 无限增长，无 TTL，无清理

@router.post("/diagnose")
async def diagnose(req: DiagnoseRequest):
    session_id = str(uuid.uuid4())
    _session_store[session_id] = {"status": "running", "created_at": time.time(), ...}
    # ... 永远不清理
```

按压测预估：50 QPS × 每 session ~0.4KB = ~20KB/s = ~1.7GB/天。无清理机制下 24h 内 OOM。

**实现方案**

```python
import asyncio
import time
from collections import OrderedDict
import structlog

logger = structlog.get_logger(__name__)


class SessionStore:
    """带 TTL 和容量限制的 Session 存储"""

    def __init__(self, max_size: int = 10000, ttl_seconds: int = 3600):
        self._store: OrderedDict[str, dict] = OrderedDict()
        self._max_size = max_size
        self._ttl = ttl_seconds
        self._cleanup_task: asyncio.Task | None = None

    def start_cleanup(self):
        """启动后台清理任务"""
        if self._cleanup_task is None:
            self._cleanup_task = asyncio.create_task(self._periodic_cleanup())

    async def _periodic_cleanup(self):
        """每 60s 清理过期 session"""
        while True:
            await asyncio.sleep(60)
            self._evict_expired()

    def _evict_expired(self):
        now = time.time()
        expired = []
        for sid, data in self._store.items():
            if now - data.get("created_at", 0) > self._ttl:
                expired.append(sid)
            else:
                break  # OrderedDict 按插入顺序，后面的更新
        for sid in expired:
            del self._store[sid]
        if expired:
            logger.info("session_cleanup", evicted=len(expired), remaining=len(self._store))

    def put(self, session_id: str, data: dict):
        # 容量驱逐：超限时移除最老的 10%
        if len(self._store) >= self._max_size:
            evict_count = self._max_size // 10
            for _ in range(evict_count):
                self._store.popitem(last=False)
            logger.warning("session_capacity_evict", evicted=evict_count)

        data.setdefault("created_at", time.time())
        self._store[session_id] = data

    def get(self, session_id: str) -> dict | None:
        return self._store.get(session_id)

    def update(self, session_id: str, **kwargs):
        if session_id in self._store:
            self._store[session_id].update(kwargs)

    def __len__(self):
        return len(self._store)


# 模块级单例
_session_store = SessionStore(max_size=10000, ttl_seconds=3600)
```

在 `app.py` 的 `lifespan` 中启动清理：

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    from aiops.web.routes.agent import _session_store
    _session_store.start_cleanup()
    yield
    # 清理资源...
```

**预期收益**

| 指标 | 优化前 | 优化后 | 提升 |
|------|-------|-------|------|
| 内存增长 | ~0.4MB/min 无限 | 封顶 ~40MB | 避免 OOM |

### 3.4 Token 预算扩容与智能降级

**现状问题**

```python
# config.py
class LLMConfig:
    daily_limit: int = 500_000  # 50 QPS × 2000 token → ~8h 耗尽
```

8 小时后所有 LLM 调用被拒绝，整个系统降级为规则引擎模式。

**实现方案**

#### 3.4.1 预算扩容 + 分级限速

```python
class LLMConfig(BaseModel):
    daily_limit: int = 1_000_000       # 基础预算翻倍
    hourly_limit: int = 80_000         # 新增小时级限制，防止短期突发
    burst_limit: int = 10_000          # 每分钟突发限制

    # 分级降级阈值
    degrade_threshold: float = 0.8     # 80% 用量时开始降级
    critical_threshold: float = 0.95   # 95% 用量时强制规则引擎
```

#### 3.4.2 Token 管理器 — 带分级降级

```python
class TokenBudgetManager:
    """分级 Token 预算管理"""

    def __init__(self, config: LLMConfig):
        self._config = config
        self._daily_used = 0
        self._hourly_used = 0
        self._last_hour_reset = time.time()
        self._last_day_reset = time.time()

    def can_spend(self, estimated_tokens: int) -> tuple[bool, str]:
        """检查是否可消费，返回 (允许, 降级策略)"""
        self._maybe_reset()

        daily_ratio = self._daily_used / self._config.daily_limit

        if daily_ratio >= self._config.critical_threshold:
            return False, "force_rule_engine"
        elif daily_ratio >= self._config.degrade_threshold:
            return True, "use_cheap_model"     # 切到便宜模型
        else:
            return True, "normal"

    def spend(self, tokens: int):
        self._daily_used += tokens
        self._hourly_used += tokens

    def _maybe_reset(self):
        now = time.time()
        if now - self._last_hour_reset > 3600:
            self._hourly_used = 0
            self._last_hour_reset = now
        if now - self._last_day_reset > 86400:
            self._daily_used = 0
            self._last_day_reset = now

    @property
    def usage_ratio(self) -> float:
        return self._daily_used / self._config.daily_limit
```

#### 3.4.3 集成到 LLMClient

```python
class LLMClient:
    def __init__(self, config=None):
        self._config = config or get_config()
        self._cache = SemanticCache(...)
        self._budget = TokenBudgetManager(self._config.llm)

    async def chat(self, messages, model=None, task_type="general", **kwargs):
        # 预估 token 数
        estimated = sum(len(m.get("content", "")) // 4 for m in messages) * 2

        can_spend, strategy = self._budget.can_spend(estimated)

        if not can_spend:
            logger.warning("token_budget_exhausted", strategy=strategy)
            raise TokenBudgetExhausted("Daily token limit reached")

        if strategy == "use_cheap_model":
```

```python
                del self._embed_cache[k]

        return vector
```

### 4.3 MCP Client 缓存调优

**现状问题**

`python/src/aiops/mcp_client/client.py` 中 L1 缓存参数偏保守：

```python
# 当前配置

# LRU 淘汰用 min() 遍历，O(n) 复杂度
```

问题：
1. 容量 200 太小：19 个工具 × 10 种常见参数组合 = 190，几乎无缓存空间
2. TTL 30s 太短：工具元数据（工具列表、schema）变化频率远低于 30s
3. `min()` 扫描 O(n) 在 200 条时尚可，扩大后成为瓶颈

**实现方案**

```python
from collections import OrderedDict
import time

class LRUTTLCache:
    """高效 LRU + TTL 缓存，O(1) 淘汰"""

    def __init__(self, max_size: int = 2000, default_ttl: int = 60):
        self._store: OrderedDict[str, tuple[float, any]] = OrderedDict()
        self._max_size = max_size
        self._default_ttl = default_ttl

    def get(self, key: str) -> any | None:
        if key not in self._store:
            return None
        expire_at, value = self._store[key]
        if time.time() > expire_at:
            del self._store[key]
            return None
        # 移到末尾（最近访问）
        self._store.move_to_end(key)
        return value

    def put(self, key: str, value: any, ttl: int | None = None):
        ttl = ttl or self._default_ttl
        if key in self._store:
            self._store.move_to_end(key)
        self._store[key] = (time.time() + ttl, value)
        # 超限淘汰最旧的
        while len(self._store) > self._max_size:
            self._store.popitem(last=False)
```

**应用到 MCP Client：**

```python
class MCPClient:
    def __init__(self, ...):
        # 分类缓存，不同 TTL
        self._tool_list_cache = LRUTTLCache(max_size=100, default_ttl=300)   # 工具列表 5min
        self._tool_result_cache = LRUTTLCache(max_size=5000, default_ttl=60) # 工具结果 60s
```

**分类 TTL 策略：**

| 缓存类型 | 容量 | TTL | 理由 |
|---------|------|-----|------|
| 工具列表 | 100 | 5min | 元数据极少变化 |

| 工具执行结果 | 5000 | 60s | HDFS/Kafka 状态有时效性 |
| 只读查询结果 | 2000 | 120s | 监控指标可稍久 |

```python
    MAX_TOTAL_CHARS = 6000      # 从 8000 降到 6000
```

### 5.2 Go MCP Server 缓存优化

**现状问题**

`go/internal/middleware/cache.go` 当前缓存淘汰使用遍历查找最旧条目，O(n) 复杂度：

```go
// 当前：O(n) 遍历找最旧条目
func (c *cacheMiddleware) evict() {
    var oldest string
    var oldestTime time.Time
    for k, v := range c.cache {  // O(n)
        if oldest == "" || v.CreatedAt.Before(oldestTime) {
            oldest = k
            oldestTime = v.CreatedAt
        }
    }
    delete(c.cache, oldest)
}
```

**实现方案**

替换为 Go 标准的 LRU 实现：

```go
package middleware

import (
    "container/list"
    "sync"
    "time"
)

type lruCache struct {
    mu       sync.RWMutex
    capacity int
    ttl      time.Duration
    items    map[string]*list.Element
    order    *list.List
}

type cacheEntry struct {
    key       string
    value     interface{}
    createdAt time.Time
}

func newLRUCache(capacity int, ttl time.Duration) *lruCache {
    return &lruCache{
        capacity: capacity,
        ttl:      ttl,
        items:    make(map[string]*list.Element, capacity),
        order:    list.New(),
    }
}

func (c *lruCache) Get(key string) (interface{}, bool) {
    c.mu.RLock()
    elem, ok := c.items[key]
    c.mu.RUnlock()
    if !ok {
        return nil, false
    }

    entry := elem.Value.(*cacheEntry)
    if time.Since(entry.createdAt) > c.ttl {
        c.mu.Lock()
        c.removeElement(elem)
        c.mu.Unlock()
        return nil, false
    }

    c.mu.Lock()
    c.order.MoveToFront(elem)
    c.mu.Unlock()
    return entry.value, true
}

func (c *lruCache) Put(key string, value interface{}) {
    c.mu.Lock()
    defer c.mu.Unlock()

    if elem, ok := c.items[key]; ok {
        c.order.MoveToFront(elem)
        elem.Value.(*cacheEntry).value = value
        elem.Value.(*cacheEntry).createdAt = time.Now()
        return
    }

    // 超容量淘汰
    for c.order.Len() >= c.capacity {
        tail := c.order.Back()
        if tail != nil {
            c.removeElement(tail)
        }
    }

    entry := &cacheEntry{key: key, value: value, createdAt: time.Now()}
    elem := c.order.PushFront(entry)
    c.items[key] = elem
}

func (c *lruCache) removeElement(elem *list.Element) {
    c.order.Remove(elem)
    entry := elem.Value.(*cacheEntry)
    delete(c.items, entry.key)
}
```

**预期收益**：淘汰操作从 O(n) 降到 O(1)，在 5000+ 条缓存时差异显著。

```python
def append_collected_data(state: AgentState, new_data: dict) -> list[dict]:
    """追加并截断，保留最近 N 条"""
    data = list(state["collected_data"])
    data.append(new_data)
    if len(data) > MAX_COLLECTED_DATA:
        # 保留首条（原始告警上下文）+ 最近 N-1 条
        data = [data[0]] + data[-(MAX_COLLECTED_DATA - 1):]
    return data
```

当前 `MemorySaver` 不支持多实例、不持久化：

```python

from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver
```

### 6.3 多级 Reranker 流水线

当前 Reranker 是 Mock 实现。生产级方案：

```
Stage 1: 轻量级 → bge-reranker-v2-m3-lite (10ms, top-20 → top-10)
Stage 2: 精排     → bge-reranker-v2-m3      (50ms, top-10 → top-5)
```

只在 top-20 候选 > 阈值时触发 Stage 2，避免不必要的重排序开销。

### 6.4 动态模型路由

当前 `ModelRouter` 是静态映射表。升级为动态路由：

```python
class DynamicModelRouter:
    """基于实时指标动态选择模型"""

    async def select_model(self, task_type: str, complexity: str) -> str:
        # 1. 检查首选模型的健康状态
        primary = self._static_routes[(task_type, complexity)]
        if await self._is_healthy(primary):
            return primary

        # 2. Fallback 到备用模型
        fallback = self._fallback_routes.get(primary, "deepseek-chat")
        logger.warning("model_fallback", primary=primary, fallback=fallback)
        return fallback

    async def _is_healthy(self, model: str) -> bool:
        """基于最近 5 分钟的成功率和延迟判断"""
        stats = self._model_stats.get(model)
        if not stats:
            return True
        return stats.success_rate > 0.9 and stats.p95_latency < 5.0
```
