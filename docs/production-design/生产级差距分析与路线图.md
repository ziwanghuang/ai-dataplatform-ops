# 生产级差距分析与路线图

> 最后更新：2026-03-30
>
> 本文档客观分析当前项目（面试 Showcase 级别）与真正生产部署之间的差距，
> 并给出按优先级排列的补全路线图。
> 这不是"项目不行"的反思，而是"我清楚知道下一步该做什么"的证明。

---

## 1. 项目定位

```
├─ 玩具 Demo          ← 一个 main.py + 几个 API 调用
├─ PoC 验证           ← 跑通核心链路，证明可行性
├─ 面试 Showcase ✅   ← 当前位置
├─ MVP 可试用         ← 填 Key + 对接 1-2 个真实工具后
├─ 内部试运行         ← 3-6 个月打磨
└─ 生产级             ← 1 年+，需要团队
```

**当前项目展示的核心价值：**
- 完整的生产级架构设计（9 节点 LangGraph 状态图 + 双模式降级）
- 每个设计决策有 WHY 解释（21 篇文档，78,000+ 行）
- 工程化规范（CI/CD、K8s、监控、安全分层）
- 23,000+ 行代码，108 个单元测试全通过

## 3. 生产差距详解

### 3.1 P0 — 阻塞生产部署（必须解决）

#### 3.1.1 MCP 工具：Mock → 真实大数据 API

**当前状态：** Go MCP Server 有 3 个 Mock 工具（HDFS/Kafka/Log），返回硬编码假数据。

**生产要求：** 对接真实 HDFS WebHDFS API、YARN ResourceManager REST API、Kafka Admin Client、ES Cluster API。

**工作量估算：** 每个工具 200-500 行，共需 15-20 个工具，约 **5,000-8,000 行 Go 代码**。

**具体缺失的工具：**

| 组件 | 工具名 | 对接 API | 预估行数 |
|------|--------|---------|---------|
| HDFS | `hdfs_namenode_status` | WebHDFS `/jmx` | ~300 |
| HDFS | `hdfs_cluster_overview` | WebHDFS `/webhdfs/v1/?op=GETCONTENTSUMMARY` | ~200 |
| HDFS | `hdfs_datanode_list` | WebHDFS `/jmx` (DataNode info) | ~250 |
| HDFS | `hdfs_block_report` | WebHDFS `/jmx` (FSNamesystem) | ~200 |
| YARN | `yarn_cluster_metrics` | RM REST `/ws/v1/cluster/metrics` | ~200 |
| YARN | `yarn_queue_status` | RM REST `/ws/v1/cluster/scheduler` | ~300 |
| YARN | `yarn_applications` | RM REST `/ws/v1/cluster/apps` | ~250 |
| Kafka | `kafka_cluster_overview` | Admin Client `DescribeCluster` | ~300 |
| Kafka | `kafka_consumer_lag` | Admin Client `ListConsumerGroupOffsets` | ~400 |
| Kafka | `kafka_topic_list` | Admin Client `ListTopics` | ~200 |
| ES | `es_cluster_health` | `_cluster/health` | ~150 |
| ES | `es_node_stats` | `_nodes/stats` | ~200 |
| 通用 | `search_logs` | ES `_search` on log index | ~300 |
| 通用 | `query_alerts` | Prometheus `/api/v1/alerts` | ~200 |
| 通用 | `query_metrics` | Prometheus `/api/v1/query_range` | ~250 |

#### 3.1.2 LLM 真实调用

**当前状态：** LLMClient 代码完整（litellm + instructor），但没有 API Key，所有 Agent 节点走降级路径。

**生产要求：** 配置 OpenAI / DeepSeek / Anthropic API Key，验证端到端 LLM 调用链路。

**工作量：** 代码已就绪，**填 Key + 调试 Prompt 即可**。预估 2-3 天。

**需要验证的链路：**
- Triage: DeepSeek-V3 结构化输出 → TriageOutput
- Diagnostic: GPT-4o 根因分析 → DiagnosticOutput
- Planning: GPT-4o-mini 诊断规划 → PlanningOutput
- 降级路径：API 超时/限流时的 fallback 行为

#### 3.1.3 知识库数据

**当前状态：** 8 条 Mock 文档 + 5 条因果链 + 3 个历史案例（硬编码在代码中）。

**生产要求：** 500+ 运维文档入库（SOP/故障案例/配置手册/变更记录）。

**工作量：**
- 文档收集：从现有 Wiki/Confluence 导出（1-2 周）
- 文档处理：OpsDocumentProcessor 已实现，批量跑 `index_batch()` 即可
- 知识图谱：Neo4j 中构建组件拓扑 + 因果链（1 周）
- 质量验证：Recall@20 > 85%，需要标注 50+ 查询的相关文档（1 周）

### 3.2 P1 — 影响可用性（上线前必须解决）

#### 3.2.1 端到端集成测试

**当前状态：** `test_e2e.py` 8 个测试，全部基于 Mock 数据，未对接真实集群。

**生产要求：**
- 对接测试集群的端到端测试（至少覆盖 HDFS/YARN/Kafka/ES）
- 负载测试（100 QPS 下延迟 P99 < 30s）
- 故障注入测试（LLM API 断开、MCP Server 超时、Redis 宕机）

**工作量：** 2-3 周（需要测试环境）。

#### 3.2.2 Prompt 工程调优

**当前状态：** Triage/Diagnostic/Planning 的 Prompt 基于经验编写，未经真实数据验证。

**生产要求：**
- 用评估数据集（92 条）跑 Eval Gate，确认准确率
- 针对失败 case 迭代 Prompt
- A/B 测试不同 Prompt 版本
- 目标：Triage 准确率 > 95%，Diagnostic 准确率 > 80%

**工作量：** 持续迭代，首轮 1-2 周。

#### 3.2.3 监控告警验证

**当前状态：** Prometheus 指标定义好了，Grafana Dashboard 和告警规则已配置，但没有真实数据流。

**生产要求：**
- Dashboard 在真实流量下验证可读性
- 告警规则阈值调优（避免误报）
- SLO/SLI 定义验证（诊断延迟 P95 < 15s、可用性 > 99.5%）

**工作量：** 1-2 周（与集成测试并行）。

---

### 3.3 P2 — 影响扩展性（上线后逐步解决）

#### 3.3.1 高可用

**当前状态：** 单实例部署，LangGraph 用 MemorySaver（进程内存）。

**生产要求：**
- 多副本 + 负载均衡
- LangGraph 切换 AsyncPostgresSaver（跨实例 checkpoint）
- Redis Sentinel / Cluster（审批状态持久化）
- 限流（每用户 QPS 限制）

**工作量：** 1-2 周（架构已预留，主要是配置和验证）。

#### 3.3.2 多租户

**当前状态：** 单用户单集群。

**生产要求：**
- 租户隔离（数据/配置/Token 预算）
- 多集群管理（不同集群的 MCP 连接配置）
- 计费和配额

**工作量：** 3-4 周（需要数据模型改动）。

#### 3.3.3 安全加固

**当前状态：** Prompt 注入正则检测 + 关键词评分，无 LLM 二次检测。

**生产要求：**
- LLM 二次检测（可疑输入让 LLM 判断是否合法）
- 红蓝对抗评估
- 审计日志持久化（当前内存 list，需要入库）
- HTTPS 全链路 + mTLS（MCP 通信）

**工作量：** 持续过程，首轮 2-3 周。

#### 3.3.4 可观测性增强

**当前状态：** OTel/LangFuse/Prometheus 集成代码已就绪，但都是 Noop 模式运行。

**生产要求：**
- OTel SDK 安装 + Jaeger 端到端追踪验证
- LangFuse 上报真实 trace/generation/score
- 日志从 stdout 切换到 ELK/Loki
- BadCase 自动导出到评估数据集的流水线

**工作量：** 1-2 周（代码已就绪，主要是部署和配置）。

## 4. 生产化路线图

### Phase 1：MVP 可试用（4-6 周）

```
Week 1-2: LLM 真实调用 + Prompt 调优
  - 配置 API Key
  - 跑通 Triage/Diagnostic/Planning 真实 LLM 链路
  - 用评估数据集验证准确率，迭代 Prompt

Week 3-4: MCP 工具对接（优先 HDFS + YARN）
  - 实现 5 个核心工具（HDFS 3 个 + YARN 2 个）
  - 对接测试集群验证

Week 5-6: 知识库 + 端到端验证
  - 导入 100+ 运维文档
  - 端到端测试：用户查询 → 完整诊断 → 报告生成
```

**交付物：** 可以在测试集群上运行的真实 Agent，支持 HDFS/YARN 基本诊断。

### Phase 2：内部试运行（2-3 个月）

```
Month 1: 完善工具 + 知识库
  - 补全 Kafka/ES 工具
  - 知识库扩充到 500+ 文档
  - Neo4j 知识图谱构建

Month 2: 稳定性 + 监控
  - 高可用部署（多副本 + PostgresSaver）
  - 监控告警调优
  - 故障注入测试

Month 3: 安全 + 评估
  - 安全红蓝对抗
  - 持续评估流水线上线
  - 首批用户试用反馈收集
```

**交付物：** 内部运维团队可日常使用的 Agent。

### Phase 3：生产级（6-12 个月）

```
- 多租户支持
- 全链路 mTLS
- SLO/SLI 达标（99.5% 可用性）
- 自动化回归测试 + 持续部署
- 知识库自动更新流水线
- 用户反馈闭环（BadCase → 改进 → 重新评估）
```
