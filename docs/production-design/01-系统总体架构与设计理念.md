## 一、系统定位与目标

### 1.1 我们要解决什么问题

大数据平台运维团队每天面对的现实：

```
┌─────────────────────────────────────────────────────────────────────┐
│                   大数据平台运维的典型痛点                              │
├──────────────────────┬──────────────────────────────────────────────┤
│ 痛点                  │ 具体表现                                     │
├──────────────────────┼──────────────────────────────────────────────┤
│ 故障定位耗时长         │ HDFS NameNode 异常 → 平均 2-4 小时才能定位根因  │
│ 告警风暴难以处理       │ 一次级联故障 → 500+ 告警涌入 → 人工筛选效率极低  │
│ 知识沉淀困难           │ 老运维离职 → 故障处理经验随人走 → 新人重复踩坑    │
│ 被动式运维             │ 等故障发生才响应 → 无法提前预防 → 业务影响大      │
│ 跨组件关联分析缺失     │ Kafka 积压 → 可能是下游 ES 慢 → 但告警只看单组件  │
│ 操作风险不可控         │ 运维误操作（如滚动重启顺序错误）→ 数据丢失         │
└──────────────────────┴──────────────────────────────────────────────┘
```

### 1.2 系统目标（量化指标）

| 指标 | 当前（人工运维） | 目标（AIOps） | 提升幅度 |
|------|----------------|--------------|---------|
| **平均故障定位时间（MTTD）** | 30-120 分钟 | 3-10 分钟 | **10x** |
| **平均故障恢复时间（MTTR）** | 2-8 小时 | 15-60 分钟 | **5-8x** |
| **告警处理效率** | 人均 50 条/天 | Agent 自动处理 80%+ | **4x+** |
| **故障预测准确率** | 无 | >75% | 从 0 到 1 |
| **知识复用率** | <20%（依赖个人） | >80%（RAG 知识库） | **4x** |
| **运维操作安全率** | 95%（人工检查） | 99.5%+（HITL 审批） | **显著提升** |

### 1.3 核心设计原则

```
七大设计原则：

1. 🎯 确定性优先
   "能用代码控制的流程，绝不交给 LLM 决策"
   LLM 负责理解和生成，流程控制交给状态机

2. 🛡️ 安全即底线
   "所有写操作必须经过人工审批，Agent 只有建议权没有执行权（初期）"
   分级审批、操作沙箱、审计日志

3. 📊 可观测是生命线
   "看不到的系统等于不存在的系统"
   全链路 Tracing、每步可解释、决策可追溯

4. 💰 成本可控
   "每个 LLM 调用都要算账"
   模型路由、语义缓存、Prompt 精简

5. 🔄 渐进式演进
   "从只读诊断开始，逐步放开写操作权限"
   Phase 分层推进，每层都是可用产品

6. 🧩 标准化集成
   "用 MCP 协议统一工具层，避免定制化陷阱"
   Tool/Resource/Prompt 三原语标准化

7. 🏗️ 领域驱动设计
   "大数据运维领域知识是核心壁垒，不是 LLM 能力"
   运维 SOP 结构化、故障模式库、因果关系图谱
```

## 三、系统组件概览

### 3.1 Agent 矩阵

| Agent | 职责 | LLM 要求 | 触发方式 |
|-------|------|---------|---------|
| **Triage Agent** | 接收告警/用户请求 → 意图识别 → 路由到合适的处理流程 | 轻量模型即可 | API/告警/定时 |
| **Planning Agent** | 分析问题 → 制定诊断计划 → 确定需要采集的数据和检查步骤 | 强推理模型 | Triage 路由 |
| **Diagnostic Agent** | 执行诊断计划 → 调用工具采集数据 → 分析数据 → 定位根因 | 强推理+工具调用 | Planning 输出 |
| **Remediation Agent** | 根据诊断结果 → 生成修复方案 → 经审批后执行修复操作 | 强推理模型 | Diagnostic 输出 |
| **Report Agent** | 汇总诊断和修复过程 → 生成结构化报告 → 沉淀知识 | 生成模型 | 流程结束 |
| **Patrol Agent** | 定时巡检 → 健康评分 → 趋势预测 → 预防性告警 | 轻量模型 | Cron 定时 |
| **Alert Correlation Agent** | 接收多条告警 → 关联分析 → 收敛去重 → 识别根因告警 | 强推理模型 | 告警触发 |

## 七、关键设计决策记录 (ADR)

### ADR-001: 为什么选择 LangGraph 而非 Temporal

| 考量 | LangGraph | Temporal |
|------|-----------|----------|
| **学习曲线** | 中等，Python 原生 | 较高，需要理解 Workflow/Activity |
| **LLM 集成** | 原生支持 | 需要自行封装 |
| **状态持久化** | 内置 Checkpointer | 内置，更强大 |
| **断点续传** | 支持 | 更好支持 |
| **社区生态** | AI/Agent 生态好 | 通用工作流生态好 |
| **适用规模** | 中小规模 | 大规模分布式 |

**决策**：Phase 1-3 使用 LangGraph，如果 Phase 4 需要跨小时级别的复杂编排再引入 Temporal。

### ADR-002: 为什么用 MCP 而非直接 Function Calling

| 考量 | MCP | 直接 Function Calling |
|------|-----|----------------------|
| **标准化** | 开放协议，跨框架 | 绑定特定 LLM 供应商 |
| **工具复用** | Server 可被任何 MCP Client 复用 | 每个框架重新实现 |
| **语言解耦** | Server 用 Go，Client 用 Python | 通常同一语言 |
| **中间件** | 可在 Server 层加审计/限流 | 需要 Client 侧处理 |
| **生态** | 快速增长中 | 成熟但碎片化 |

**决策**：使用 MCP 协议统一工具层，Go 实现 MCP Server（高性能），Python Client 集成到 Agent 中。

### ADR-003: 为什么 Agent 分诊需要独立的 Triage Agent

**问题**：所有请求都走完整的 Planning→Diagnostic→Report 流程太重了。

**决策**：引入 Triage Agent 作为入口：
- 简单查询（"HDFS 容量多少"）→ 直接调工具返回，不走完整流程
- 复杂诊断（"集群为什么慢"）→ 路由到多 Agent 流程
- 告警处理（批量告警）→ 先关联收敛，再诊断

**收益**：预计 40-50% 的请求可通过 Triage 直接处理，节省 70%+ 的 LLM 调用成本。

### ADR-004: 自部署 LLM vs 云 API

**决策**：混合方案
- **云 API (GPT-4o/Claude)**：复杂推理和诊断任务，准确率优先
- **自部署 (vLLM + Qwen2.5)**：简单分诊/分类、敏感数据不出域场景
- **模型路由器**：根据任务复杂度 + 数据敏感度自动选择

### ADR-005: 知识图谱 vs 纯 RAG

**问题**：纯 RAG 无法回答"Kafka 积压可能是哪些组件导致的"这类关系推理问题。

**决策**：RAG + 知识图谱混合
- RAG 负责：文档检索、SOP 匹配、历史案例检索
- 知识图谱负责：组件依赖关系、故障传播路径、因果推理
- 两者融合：Graph-RAG，检索时同时查向量库和图数据库
