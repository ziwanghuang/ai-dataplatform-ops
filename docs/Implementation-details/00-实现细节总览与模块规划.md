### WHY 分层架构

```
为什么系统分为 5 层？

┌─ L5: 运维层 ──────────────────────────────────┐
│  20-部署CICD | 19-数据采集                      │
│  WHY: 系统的"骨架"，没有它们其他层无法运行      │
└────────────────────────────────────────────────┘
┌─ L4: 横切层 ──────────────────────────────────┐
│  16-安全 | 17-可观测 | 18-评估 | 15-HITL        │
│  WHY: 贯穿所有层的非功能性需求，不可或缺        │
└────────────────────────────────────────────────┘
┌─ L3: Agent 层 ────────────────────────────────┐
│  03-框架 | 04-08 各 Agent                       │
│  WHY: 业务逻辑所在，系统的"大脑"                │
└────────────────────────────────────────────────┘
┌─ L2: 能力层 ──────────────────────────────────┐
│  02-LLM | 12-RAG | 13-Graph | 14-索引          │
│  WHY: Agent 的"智慧来源"——语言理解+知识检索      │
└────────────────────────────────────────────────┘
┌─ L1: 工具层 ──────────────────────────────────┐
│  09-MCP Server | 10-中间件 | 11-MCP 客户端      │
│  WHY: Agent 的"双手"——与真实基础设施交互的能力    │
└────────────────────────────────────────────────┘
┌─ L0: 基础层 ──────────────────────────────────┐
│  01-工程化基础                                   │
│  WHY: 所有模块共享的基础设施（日志/配置/错误处理）│
└────────────────────────────────────────────────┘

每层只依赖下一层，不跨层依赖（L3 不直接调用 L1，
而是通过 L2 间接调用）。这保证了模块的可替换性：
如果要把 Milvus 换成 Pinecone，只需要改 L2，L3 完全不感知。
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    20. 部署 CI/CD                               │

│  19. 数据采集           18. 评估体系          17. 可观测性       │

├─────────────────────────────────────────────────────────────────┤
│  08. Report   07. Patrol/Alert   06. Planning/Remediation       │
├─────────────────────────────────────────────────────────────────┤
│        04. Triage Agent        05. Diagnostic Agent             │
├─────────────────────────────────────────────────────────────────┤
│              03. Agent 核心框架与状态机                          │
├─────────────────────────────────────────────────────────────────┤
│  02. LLM 客户端   12. RAG 引擎   13. Graph-RAG   14. 索引管线  │
├─────────────────────────────────────────────────────────────────┤
│  09. MCP Server   10. MCP 中间件   11. MCP 客户端               │
├─────────────────────────────────────────────────────────────────┤
│              01. 项目工程化基础与代码结构                        │
└─────────────────────────────────────────────────────────────────┘
```

**依赖规则**：
- **01** 是所有模块的基础
- **02、09-11** 是 Agent 层的基础设施（LLM 调用和工具调用）
- **12-14** 是知识层基础设施
- **03** 依赖 02 和 11，是所有具体 Agent (04-08) 的基础
- **15-16** 是横切关注点，嵌入到 03-11 各模块中
- **17** 贯穿所有层，但可以增量集成
- **18** 依赖 03-14 产出的模块才能评估
- **19** 相对独立，可并行建设
- **20** 依赖所有模块的容器化产出

> **工程难点：80+ 模块间的依赖关系管理**
>
> **挑战**：系统涉及 80+ 模块、21 篇文档、5 个技术栈（Python/Go/Java/K8s/Flink），模块间存在复杂的上下游依赖关系。任意一个模块接口变更都可能引发级联影响，且跨语言通信（Python ↔ Go）的接口同步尤为困难。
>
> **解决方案**：采用 5 层分层架构（L0 基础层 → L5 运维层），严格遵循"只依赖下一层，不跨层依赖"的原则。通过模块交互矩阵精确标注调用关系，确保改 A 时能快速定位受影响的 B。Python/Go 之间通过 MCP JSON-RPC 2.0 协议解耦，接口定义在 Monorepo 中原子提交，保证版本始终一致。

## 九、架构设计 WHY 总结

### 9.1 WHY Python + Go 双语言

```
Python 负责：                    Go 负责：
├── Agent 编排 (LangGraph)       ├── MCP Server (42 个工具)
├── LLM 调用 (litellm)          ├── 中间件链 (8 层)
├── RAG 检索 (Milvus/ES)        ├── HTTP Gateway (Fiber)
├── 评估 (RAGAS/DeepEval)       └── 高性能工具执行
└── 知识图谱 (Neo4j)

WHY 不全用 Python？
  MCP 工具层需要高并发低延迟（100+ QPS，<10ms 中间件开销）。
  Python 的 GIL 和解释器开销在这个场景下是瓶颈。
  Go 的 goroutine + 零 GC 压力完美匹配。

WHY 不全用 Go？
  Agent 编排层需要 LangGraph/litellm/instructor 等 AI 生态库。
  Go 的 AI 生态远不如 Python 成熟。强行用 Go 等于重新造轮子。

双语言的代价：
  - 需要维护两套构建系统（pyproject.toml + go.mod）
  - 跨语言通信通过 HTTP/JSON-RPC（额外 1-2ms 延迟）
  - 团队需要同时掌握两种语言

这个代价是值得的：每种语言用在最合适的场景，总体效率 > 单语言方案。
```

### 9.2 WHY LangGraph 而非 AutoGPT/CrewAI/自研

| 框架 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **LangGraph** ✅ | 状态机模型、可控流程、checkpoint 持久化 | 学习曲线、依赖 LangChain 生态 | — 我们选了它 |
| AutoGPT | 全自动、零配置 | 不可控、Token 爆炸、无法审计 | Agent 行为不可预测，生产环境不可接受 |
| CrewAI | 多 Agent 协作自然 | 流程控制弱、没有 checkpoint | 无法在故障中恢复会话 |
| 自研 | 完全控制 | 重复造轮子、缺乏社区支持 | 投入产出比太低 |

> **核心选择理由**：LangGraph 的 StateGraph 让每个 Agent 节点有**明确的输入输出契约**，
> 流程可审计、可断点续传、可人工介入。这在生产环境是不可妥协的。

### 9.3 WHY MCP 协议

> **为什么不直接用 REST API 或 gRPC 让 Agent 调用工具？**

```
方案 A (REST): Agent 直接 HTTP 调用各组件 API
  ❌ Agent 需要知道每个 API 的格式
  ❌ 没有统一的安全/审计/缓存层
  ❌ 新增工具需要改 Agent 代码

方案 B (gRPC): 自定义 gRPC 服务
  ✅ 性能好
  ❌ 需要生成 proto → stub → client
  ❌ 工具描述不在协议中，Agent 无法自动发现

方案 C (MCP): Model Context Protocol ✅
  ✅ 工具描述在 Server 侧（Agent 启动时自动发现）
  ✅ JSON-RPC 2.0 简单直接
  ✅ 统一中间件链处理安全/审计/缓存
  ✅ 新增工具只需在 Server 侧注册，Agent 自动感知
```

### 9.4 WHY 评估体系这么重

> **AI Agent 项目最大的风险不是"做不出来"，而是"做出来不知道准不准"。**

```
传统软件: 单元测试 + 集成测试 → 确定性验证
AI Agent: LLM 输出不确定 + 工具调用链路复杂 → 需要概率性评估

我们的策略：
  L1: 单元测试（代码逻辑）    → 传统方法
  L2: 离线评估集（Agent 行为） → 156 条标注用例
  L3: CI/CD 门禁（质量阈值）  → 安全 100%、准确率 85%+
  L4: A/B 测试（版本对比）    → Prompt 变更影响评估
  L5: 生产监控（业务指标）    → MTTD/MTTR 追踪
```
