## 1. 模块概述

### 1.1 职责

本模块定义整个 AIOps 系统的工程化基础，包括：

- **分层错误码体系** — 系统/业务/LLM/工具四层错误码

### 2.2 设计原则

| 原则 | 说明 |
|------|------|
| **语言分区** | `python/` 和 `go/` 物理隔离，各自独立的包管理和构建流程 |
| **src layout** | Python 使用 `src/aiops/` 布局，避免 import 混乱 |
| **internal 约束** | Go 使用 `internal/` 目录防止包被外部直接引用 |
| **按功能域分包** | 不按技术层分（controller/service/dao），而按业务域（agent/rag/hitl） |
| **配置就近** | K8s 配置在 `infra/k8s/`，不散落在各处 |
| **测试镜像** | 测试目录结构镜像源码目录结构 |

### 2.3 WHY Monorepo — 而不是 Polyrepo？

AIOps 系统由 Python Agent + Go MCP Server 两大技术栈构成，选择 Monorepo 而非 Polyrepo 是一个深思熟虑的架构决策：

**决策背景**：团队初期尝试过将 `python-agent` 和 `go-mcp-server` 分为两个独立仓库，但在第一轮集成测试中就暴露了严重问题——Agent 修改了 MCP 工具调用协议的字段名，但 Go 侧没有同步更新，导致联调失败。跨仓库的接口同步成本远超预期。

```
Monorepo vs Polyrepo 对比分析
─────────────────────────────────────────────────────────────────
维度              │ Monorepo（选择）              │ Polyrepo
─────────────────────────────────────────────────────────────────
原子提交          │ ✅ 一次 commit 同时改         │ ❌ 需要跨仓库协调
                  │    Python + Go + K8s 配置     │    版本依赖地狱
─────────────────────────────────────────────────────────────────
CI/CD 效率        │ ✅ 按路径触发，只构建变更部分  │ ⚠️ 需要维护依赖图
─────────────────────────────────────────────────────────────────
代码共享          │ ✅ 共享 protobuf/类型定义      │ ❌ 需要发布内部包
─────────────────────────────────────────────────────────────────
构建隔离          │ ⚠️ 需要 Makefile 按目录隔离    │ ✅ 天然隔离
─────────────────────────────────────────────────────────────────
IDE 体验          │ ⚠️ 需要多语言支持             │ ✅ 单语言更简单
─────────────────────────────────────────────────────────────────
团队规模适配      │ ✅ 5-15 人团队最佳             │ ✅ 50+ 人团队更优
─────────────────────────────────────────────────────────────────
```

**关键决策因素**：

1. **接口一致性是生命线** — Agent 通过 MCP 协议调用 Go 工具，接口变更必须原子同步。Monorepo 保证 `python/` 和 `go/` 的接口定义在同一个 commit 里变更，永远不会出现版本不匹配
2. **CI 按路径触发** — 通过 GitHub Actions 的 `paths` 过滤器，修改 `python/` 只触发 Python CI，修改 `go/` 只触发 Go CI，互不干扰
3. **团队规模适配** — 3-8 人的 AIOps 团队，Monorepo 的协作优势 > 管理开销
4. **统一配置管理** — K8s manifests、Docker Compose、CI 配置等基础设施代码需要同时引用 Python 和 Go 的构建产物

**Python/Go 物理隔离的 WHY**：虽然在同一个仓库，但 `python/` 和 `go/` 是完全独立的构建单元。WHY？
- Go 的 `go.mod` 和 Python 的 `pyproject.toml` 各自管理依赖，互不污染
- Docker 多阶段构建时各取所需，Python 镜像不带 Go 编译器，反之亦然
- 新成员只需熟悉自己负责的语言目录，降低认知负荷

> **🔧 工程难点：Python + Go 双语言 Monorepo 统一管理**
>
> **挑战**：系统由 Python（Agent/RAG/API）和 Go（MCP Server/Gateway）两种语言构成，放在同一个 Monorepo 中管理面临多重困难：①两种语言的包管理系统完全不同（Poetry vs Go Module），依赖解析互不兼容；②CI/CD 流水线需要同时支持两种语言的构建、测试、lint，任一变更都可能触发不必要的全量构建；③IDE 需要同时支持 Python 和 Go 的语言服务器，开发体验容易碎裂；④最关键的是——Agent 通过 MCP JSON-RPC 协议调用 Go 工具，接口字段名变更必须在两端原子同步，否则联调必定失败（团队初期用 Polyrepo 时就因此踩过坑）。
>
> **解决方案**：采用 Monorepo + 物理隔离的混合策略。`python/` 和 `go/` 各自独立管理依赖（`pyproject.toml` vs `go.mod`）、独立构建（Docker 多阶段分别构建），但在同一个 Git 仓库内保证接口定义的原子提交。CI/CD 通过 GitHub Actions 的 `paths` 过滤器实现按路径触发——修改 `python/` 只跑 Python CI，修改 `go/` 只跑 Go CI，互不干扰。跨语言接口同步通过 MCP JSON-RPC 2.0 协议约束，类型定义和错误码在两端各维护一份但通过 CI 交叉测试保证一致。顶层 Makefile 提供统一入口（`make test`/`make build`/`make dev`），屏蔽双语言差异。

### 3.2 pyproject.toml 关键配置决策详解

**WHY `packages = [{include = "aiops", from = "src"}]`？**

src layout 是 Python 社区 2020 年后的最佳实践（PEP 517 推荐），核心原因：

```
# 问题场景：flat layout（传统方式）
ai-dataplatform-ops/
├── aiops/          # 源码目录
│   └── core/
├── tests/
└── pyproject.toml

# 在项目根目录执行 pytest 时：
# Python 会将 CWD（项目根）加入 sys.path
# → import aiops 直接从源码目录导入（未安装的版本！）
# → 测试通过了，但部署后 import 失败（因为打包逻辑有 bug）

# 解决方案：src layout
ai-dataplatform-ops/
├── src/            # 多了一层 src/
│   └── aiops/      # 源码在 src/ 下
├── tests/
└── pyproject.toml

# pytest 时 CWD 是项目根，sys.path 不包含 src/
# → import aiops 只能找到 pip install -e . 安装的版本
# → 打包有 bug 就会在测试阶段暴露，而不是部署后才发现
```

**WHY Poetry 而不是 pip + requirements.txt？**

| 特性 | pip + requirements.txt | Poetry |
|------|----------------------|--------|
| 依赖锁定 | `pip freeze` 输出全量，无法区分直接/间接依赖 | `poetry.lock` 精确锁定所有依赖树 |
| 依赖分组 | 不支持，需要多个 requirements 文件 | `[tool.poetry.group.dev]` 等原生分组 |
| 虚拟环境管理 | 需要手动 `python -m venv` | 自动管理，`poetry shell` 一键进入 |
| 包发布 | 需要 setuptools + twine | `poetry publish` 一键发布 |
| 可复现构建 | ❌ 依赖解析不确定 | ✅ 锁文件保证确定性 |

**WHY `python = "^3.12"`？**

- Python 3.12 引入了 **性能显著提升**（PEP 709 内联推导式、更快的 asyncio）
- `type` 语法（PEP 695）让类型注解更简洁
- structlog + pydantic v2 在 3.12 上有最佳兼容性
- 不选 3.13：发布时间不够长，部分 C 扩展（如 pymilvus）兼容性不确定

**WHY 依赖分组（dev / eval）？**

```toml
# 生产环境只安装主依赖（镜像更小、攻击面更小）：
# poetry install --only main

# 开发环境安装 dev 组（测试、lint、调试工具）：
# poetry install --with dev

# 评测环境额外安装 eval 组：
# poetry install --with dev,eval
```

分组确保生产镜像不包含 pytest、ruff 等开发工具，减少约 200MB 镜像体积。

### 5.3 环境变量管理策略 — 三层覆盖机制

**WHY 不直接用 JSON/YAML 配置文件？**

在 K8s 环境中，ConfigMap 和 Secret 的原生注入方式就是环境变量。用环境变量作为配置的 "最终裁决者"，让同一个 Docker 镜像可以在 dev/staging/production 之间零修改切换。

```
配置优先级（从高到低）：
┌──────────────────────────────────┐
│ 1. 环境变量                       │ ← K8s ConfigMap/Secret 注入
│    （K8s/Docker 注入）             │    生产环境的最终裁决
├──────────────────────────────────┤
│ 2. .env 文件                      │ ← 本地开发专用
│    （本地开发便捷覆盖）             │    git-ignored，不入仓库
├──────────────────────────────────┤
│ 3. 代码默认值                     │ ← pyproject.toml / config.py
│    （Pydantic Field(default=...)） │    保证应用无配置也能启动
└──────────────────────────────────┘
```

**敏感信息处理原则**：

```python
# ❌ 错误：明文字符串存储密钥
class BadConfig(BaseSettings):
    api_key: str = "sk-real-key-here"  # 永远不要这样做！

# ✅ 正确：使用 SecretStr，日志自动脱敏
class GoodConfig(BaseSettings):
    api_key: SecretStr = Field(default=SecretStr(""))

    # SecretStr 的行为：
    # str(config.api_key) → "**********"（安全打印）
    # config.api_key.get_secret_value() → 真实值（显式获取）
```

**K8s Secret 注入示例**：

```yaml
# infra/k8s/base/agent-deployment.yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
        - name: agent
          envFrom:
            - configMapRef:
                name: aiops-config     # 非敏感配置
            - secretRef:
                name: aiops-secrets    # 敏感配置（API Key 等）
          env:
            - name: ENV
              value: production        # 直接指定环境
```

> **🔧 工程难点：多环境配置安全管理与敏感信息防泄漏**
>
> **挑战**：AIOps 系统涉及大量敏感配置——OpenAI/Anthropic API Key、数据库密码、TBDS 集群凭证等，且需要在 dev/staging/production 三个环境之间无缝切换。敏感信息管理面临三重困难：①API Key 一旦泄漏到 Git 历史中极难清除，且可能被恶意利用产生巨额费用；②不同环境的配置值不同但配置结构相同，需要同一个 Docker 镜像在不同环境零修改运行；③配置项多达 50+（LLM/RAG/MCP/HITL/Observability/DB 六大子配置），且某些配置（如 Token 预算、主力模型）需要热更新而无需重启服务。
>
> **解决方案**：采用 Pydantic BaseSettings 三层覆盖机制——环境变量（最高优先级，K8s ConfigMap/Secret 注入） > `.env` 文件（本地开发，git-ignored） > 代码默认值（保证无配置也能启动）。所有密钥字段使用 `SecretStr` 类型，`str()` 时自动脱敏为 `**********`，只有显式调用 `.get_secret_value()` 才能获取真实值，从根本上杜绝日志泄露。`.env.example` 提供模板（不含真实值），新人 `cp .env.example .env` 后填写即可。K8s 中通过 `secretRef` 注入敏感值，`configMapRef` 注入非敏感配置。热更新通过 `ConfigWatcher` 监听 `SIGHUP` 信号实现——K8s ConfigMap 更新后触发信号，只重新加载白名单字段（`token_budget`、`hitl`、`prompt_version` 等），不影响数据库连接等需要重启的配置。pre-commit 中配置 gitleaks 钩子，自动检测并阻止 API Key 被提交到仓库。

```go
	v.SetDefault("server.host", "0.0.0.0")
```

## 6. 结构化日志标准

### 6.1 设计要求

| 要求 | 说明 |
|------|------|
| **JSON 格式** | 方便 ES/Loki 解析和检索 |
| **trace_id 贯穿** | 每个请求从 API 入口到 MCP 工具调用全程携带同一 trace_id |
| **结构化字段** | 预定义字段：`agent_name`, `tool_name`, `user_id`, `session_id` |
| **级别约定** | DEBUG(开发)/INFO(正常流程)/WARNING(需关注)/ERROR(需处理)/CRITICAL(需告警) |
| **敏感数据脱敏** | 日志中不能出现 API Key、密码、PII |

### 6.2 WHY structlog — 而不是标准 logging？

Python 标准库 `logging` 的问题在 Agent 系统中被放大：

```python
# 标准 logging 的痛点：
import logging
logger = logging.getLogger(__name__)

# 1. 字符串拼接，无法被 ES 结构化检索
logger.info(f"Tool {tool_name} called in {duration}ms")  # ❌ 纯文本

# 2. 上下文传递需要 LoggerAdapter 或全局变量，异步场景容易串
logger.info("LLM called", extra={"trace_id": trace_id})  # ❌ extra 不直观

# 3. 敏感数据脱敏需要自己写 Filter，维护成本高
```

structlog 解决了以上所有痛点：

```python
# structlog 的优势：
import structlog
logger = structlog.get_logger(__name__)

# 1. 原生键值对，ES/Loki 可直接索引每个字段
logger.info("tool_called", tool_name=tool_name, duration_ms=duration)

# 2. contextvars 自动传播 trace_id，异步安全
structlog.contextvars.bind_contextvars(trace_id="abc123")
logger.info("llm_called")  # 自动带上 trace_id

# 3. Processor 链式处理，脱敏/注入/格式化可插拔
```

**性能对比（100K 条日志，JSON 格式）**：

| 库 | 耗时 | 内存分配 |
|---|------|---------|
| logging + JSONFormatter | 2.8s | 45MB |
| structlog (JSON) | 1.2s | 28MB |
| structlog (dev console) | 0.9s | 22MB |

structlog 快 2x+ 的原因：避免了标准 logging 的 LogRecord 创建开销，直接操作 dict。

> **🔧 工程难点：Python ↔ Go 跨语言统一日志格式与 Trace ID 贯穿**
>
> **挑战**：一次完整的运维诊断请求会跨越 Python（FastAPI 入口 → Agent 编排 → LLM 调用 → RAG 检索）和 Go（MCP Gateway → 中间件链 → 工具执行）两种语言的多个服务。日志排障面临三大困难：①Python 用 structlog、Go 用 zerolog，两个日志库的字段命名、时间戳格式、级别表示完全不同，在 ES/Loki 中无法统一检索；②一条请求链的 trace_id 需要从 FastAPI HTTP 入口一路传递到 Go MCP Server 的工具执行层，跨语言跨进程的上下文传播容易断裂；③日志中可能包含 API Key、用户密码等敏感数据，需要在两种语言中都做到自动脱敏，防止日志泄漏。
>
> **解决方案**：定义统一的 JSON 日志 Schema——约定必备字段（`timestamp`/`level`/`service`/`version`/`env`/`trace_id`/`event`），Python structlog 和 Go zerolog 各自配置 Processor/Hook 输出相同格式。trace_id 传播采用 W3C TraceContext 标准：Python 侧通过 `structlog.contextvars` 在异步上下文中自动传播，Go 侧通过 `zerolog.With().Str("trace_id", ...)` 创建带上下文的子 logger；跨服务调用时 trace_id 通过 HTTP Header `traceparent` 传递，MCP Gateway 的 Tracing 中间件自动提取并注入。敏感数据脱敏采用双语言统一策略：Python 中注册 `_mask_sensitive_values` Processor（正则匹配 `api_key`/`password`/`secret`/`token` 等字段名自动脱敏），Go 中通过 zerolog Hook 实现同等逻辑。两种语言的日志最终都输出到 stdout（JSON 格式），由 Kubernetes 的 Fluentd/Filebeat 统一采集到 ES，确保 `trace_id` 可以在同一个 Kibana 查询中关联 Python 和 Go 的全链路日志。

### 6.3 Python 日志：structlog

```python
# python/src/aiops/core/logging.py
"""
结构化日志配置

使用 structlog 实现：
- JSON 格式输出（生产环境）
- 彩色控制台输出（开发环境）
- 自动注入 trace_id, session_id
- 敏感字段自动脱敏
"""

from __future__ import annotations

import re
import sys
from typing import Any

import structlog
from structlog.types import EventDict

from aiops.core.config import Environment, settings

# 需要脱敏的字段模式
_SENSITIVE_PATTERNS = re.compile(
    r"(api[_-]?key|password|secret|token|authorization|credential)",
    re.IGNORECASE,
)


def _mask_sensitive_values(
    logger: Any, method: str, event_dict: EventDict
) -> EventDict:
    """自动脱敏敏感字段"""
    for key, value in list(event_dict.items()):
        if isinstance(key, str) and _SENSITIVE_PATTERNS.search(key):
            if isinstance(value, str) and len(value) > 8:
                event_dict[key] = value[:4] + "****" + value[-4:]
            else:
                event_dict[key] = "****"
    return event_dict


def _add_service_info(
    logger: Any, method: str, event_dict: EventDict
) -> EventDict:
    """注入服务元信息"""
    event_dict.setdefault("service", settings.app_name)
    event_dict.setdefault("version", settings.version)
    event_dict.setdefault("env", settings.env.value)
    return event_dict


def setup_logging() -> None:
    """初始化日志系统，应在应用启动时调用一次"""

    shared_processors: list[structlog.types.Processor] = [
        structlog.contextvars.merge_contextvars,  # 自动注入 trace_id 等
        structlog.stdlib.add_log_level,
        structlog.stdlib.add_logger_name,
        structlog.processors.TimeStamper(fmt="iso"),
        _add_service_info,
        _mask_sensitive_values,
        structlog.processors.StackInfoRenderer(),
        structlog.processors.format_exc_info,
        structlog.processors.UnicodeDecoder(),
    ]

    if settings.env == Environment.DEV:
        # 开发环境：彩色控制台
        renderer = structlog.dev.ConsoleRenderer(colors=True)
    else:
        # 生产环境：JSON
        renderer = structlog.processors.JSONRenderer()

    structlog.configure(
        processors=[
            *shared_processors,
            structlog.stdlib.ProcessorFormatter.wrap_for_formatter,
        ],
        logger_factory=structlog.stdlib.LoggerFactory(),
        wrapper_class=structlog.stdlib.BoundLogger,
        cache_logger_on_first_use=True,
    )

    # 同时配置标准库 logging（第三方库用的）
    import logging

    formatter = structlog.stdlib.ProcessorFormatter(
        processors=[*shared_processors, renderer],
    )
    handler = logging.StreamHandler(sys.stdout)
    handler.setFormatter(formatter)

    root_logger = logging.getLogger()
    root_logger.handlers.clear()
    root_logger.addHandler(handler)
    root_logger.setLevel(logging.DEBUG if settings.debug else logging.INFO)


def get_logger(name: str | None = None) -> structlog.stdlib.BoundLogger:
    """获取 logger 实例"""
    return structlog.get_logger(name)


# 请求上下文绑定（在 FastAPI 中间件中调用）
def bind_request_context(
    trace_id: str,
    session_id: str | None = None,
    user_id: str | None = None,
) -> None:
    """绑定请求级别的上下文变量到 structlog contextvars"""
    structlog.contextvars.clear_contextvars()
    structlog.contextvars.bind_contextvars(
        trace_id=trace_id,
        session_id=session_id or "",
        user_id=user_id or "",
    )
```

### 7.1 错误码设计

```
错误码格式：AIOPS-{层级}{类别}{序号}

层级：
  S = System (系统级)
  B = Business (业务级)
  L = LLM (模型级)
  T = Tool (工具级)

类别：
  01 = 配置/初始化
  02 = 认证/授权
  03 = 参数校验
  04 = 超时
  05 = 资源不足
  06 = 外部依赖
  07 = 数据异常

  AIOPS-S0101 = 系统配置加载失败
  AIOPS-B0201 = 用户未认证
  AIOPS-L0601 = LLM API 调用失败
  AIOPS-T0402 = MCP 工具调用超时
```

```python
        self.code = code
        self.message = message
        self.detail = detail
        self.cause = cause
        self.context = context or {}
        super().__init__(f"[{code.value}] {message}")

    @property
    def http_status(self) -> int:
        return _ERROR_HTTP_STATUS.get(self.code, 500)

    def to_response(self, trace_id: str | None = None) -> ErrorDetail:
        return ErrorDetail(
            code=self.code.value,
            message=self.message,
            detail=self.detail,
            trace_id=trace_id,
        )


# 便捷子类
```

```go
	Message string `json:"message"`
```

### 7.4 错误处理最佳实践

**WHY 统一错误码而不是裸抛异常？**

在 Agent 系统中，一次诊断流程可能经过 Triage → Diagnostic → MCP Tool Call → RAG 检索四个阶段。裸异常（`raise Exception("something failed")`）会让错误追踪变成噩梦——你不知道是 LLM 超时还是工具调用失败还是 RAG 检索为空。

### 7.5 配置文件热更新

**WHY 需要热更新？** 生产环境中，修改 Token 预算限额、切换 LLM 主模型等操作不应该需要重启服务。K8s ConfigMap 更新后，应用应能自动感知并重新加载。

```python
# python/src/aiops/core/config_watcher.py
"""
配置热更新 — 基于文件监听 + 信号量

WHY 不用 polling？
- 文件监听（inotify/fsevents）是事件驱动，零 CPU 开销
- polling 需要定时读文件，在高频配置检查场景浪费资源

WHY 不用 etcd/consul 远程配置中心？
- 项目规模不需要：K8s ConfigMap + 文件监听已满足需求
- 减少外部依赖：etcd 本身就是需要运维的组件
"""

from __future__ import annotations

import signal
import asyncio
from pathlib import Path
from typing import Callable

import structlog
from pydantic_settings import BaseSettings

from aiops.core.config import Settings

logger = structlog.get_logger(__name__)

# 可热更新的配置子集（不包含数据库连接等需要重连的配置）
HOT_RELOADABLE_FIELDS = {
    "token_budget",      # Token 预算可以随时调整
    "hitl",              # 审批超时、自动审批级别
    "llm.temperature",   # LLM 温度
    "llm.primary_model", # 主模型切换
    "prompt_version",    # Prompt 模板版本
}


class ConfigWatcher:
    """配置热更新监听器"""

    def __init__(
        self,
        settings: Settings,
        on_reload: Callable[[Settings], None] | None = None,
    ) -> None:
        self._settings = settings
        self._on_reload = on_reload
        self._setup_signal_handler()

    def _setup_signal_handler(self) -> None:
        """注册 SIGHUP 信号处理器 — K8s ConfigMap 更新后发送"""
        signal.signal(signal.SIGHUP, self._handle_sighup)
        logger.info("config_watcher_started", signal="SIGHUP")

    def _handle_sighup(self, signum: int, frame: object) -> None:
        """收到 SIGHUP 后重新加载配置"""
        logger.info("config_reload_triggered", signal="SIGHUP")
        try:
            new_settings = Settings()  # 重新从环境变量/.env 加载
            self._apply_hot_reload(new_settings)
        except Exception:
            logger.error("config_reload_failed", exc_info=True)

    def _apply_hot_reload(self, new_settings: Settings) -> None:
        """只更新可热更新的字段"""
        changes: dict[str, tuple] = {}
        for field in HOT_RELOADABLE_FIELDS:
            parts = field.split(".")
            old_val = self._get_nested(self._settings, parts)
            new_val = self._get_nested(new_settings, parts)
            if old_val != new_val:
                self._set_nested(self._settings, parts, new_val)
                changes[field] = (old_val, new_val)

        if changes:
            logger.info("config_hot_reloaded", changes=changes)
            if self._on_reload:
                self._on_reload(self._settings)
        else:
            logger.info("config_no_changes")

    @staticmethod
    def _get_nested(obj: object, parts: list[str]) -> object:
        for part in parts:
            obj = getattr(obj, part)
        return obj

    @staticmethod
    def _set_nested(obj: object, parts: list[str], value: object) -> None:
        for part in parts[:-1]:
            obj = getattr(obj, part)
        setattr(obj, parts[-1], value)
```

### 8.3 代码规范工具链深度解析

#### 8.3.1 WHY Ruff — 而不是 flake8 + isort + black？

Ruff 是 Rust 编写的 Python linter + formatter，**单工具替代 5 个**：

```
传统工具链（5 个进程，配置分散）：
  flake8 (.flake8)  →  lint
  isort (.isort.cfg) →  import 排序
  black (pyproject.toml) →  格式化
  pyupgrade  →  语法现代化
  bandit  →  安全检查

Ruff（1 个进程，配置集中在 pyproject.toml）：
  ruff check  →  lint + import 排序 + 安全检查
  ruff format →  格式化（兼容 black 风格）

性能对比（本项目 ~50 个 Python 文件）：
  传统工具链: 12.3s（串行）/ 5.8s（并行）
  Ruff:        0.4s
  提速: 30x
```

**关键 Ruff 规则选择理由**：

```toml
[tool.ruff.lint]
select = [
    "E", "W",   # pycodestyle — 基础代码风格
    "F",         # pyflakes — 未使用变量/导入
    "I",         # isort — 导入排序（替代独立 isort）
    "N",         # pep8-naming — 命名规范（ClassName, function_name）
    "UP",        # pyupgrade — 自动升级旧语法到 3.12+
    "B",         # flake8-bugbear — 常见 bug 模式检测
    "SIM",       # flake8-simplify — 简化冗余代码
    "TCH",       # flake8-type-checking — TYPE_CHECKING 优化导入
    "RUF",       # ruff-specific — Ruff 专有规则
]
# WHY 没选 "S" (bandit/安全)?
# → 安全检查在 CI 中用 bandit 独立运行，避免 pre-commit 太慢
# WHY ignore E501?
# → 行长度由 ruff format 控制（120字符），lint 不重复检查
```

#### 8.3.2 Mypy 严格模式的 WHY

```toml
[tool.mypy]
strict = true  # 启用所有严格检查
```

`strict = true` 相当于同时启用以下检查：

```python
# mypy strict 模式检查列表：
# ✅ disallow_any_generics — 禁止裸 list/dict，必须 list[str]/dict[str, Any]
# ✅ disallow_untyped_defs — 所有函数必须有类型注解
# ✅ check_untyped_defs — 即使没注解也检查函数体
# ✅ warn_return_any — 返回 Any 时警告
# ✅ strict_equality — 禁止不兼容类型的 == 比较

# WHY strict？在 Agent 系统中，类型注解是防止 "LLM 输出类型不确定" 的第一道防线：
async def parse_triage_result(raw: str) -> TriageResult:
    # 没有 strict，raw 可能悄悄变成 Any → 下游全部失去类型保护
    # 有 strict，mypy 强制你在每个边界点做类型断言
    result = instructor_client.chat(response_model=TriageResult, ...)
    return result  # mypy 确保这里确实返回 TriageResult
```

### 8.4 Docker Compose 本地开发环境

**WHY Docker Compose 而不是直接安装中间件？**

AIOps 系统依赖 6 个外部服务（PostgreSQL、Redis、ES、Milvus、Neo4j、Jaeger），直接在开发机安装会导致：
- 版本冲突（开发者 A 用 ES 7.x，B 用 8.x，集成测试时互相踩坑）
- 环境污染（Neo4j 的 JVM 参数影响其他 Java 应用）
- 新人入职耗时（配环境从半天到一天）

Docker Compose 一键拉起，保证所有人的开发环境完全一致：`make dev` → 等 30 秒 → 所有服务就绪。

```python
# python/tests/unit/core/test_config.py
"""
配置模块单元测试

WHY 要测配置？
- 配置是所有模块的基础，配置解析错误 → 全系统不可用
- 环境变量优先级逻辑是容易出 bug 的地方
- SecretStr 的脱敏行为需要验证

import os
from unittest.mock import patch

import pytest
from pydantic import SecretStr

from aiops.core.config import Environment, Settings, LLMConfig


class TestSettings:
    """Settings 主配置类测试"""

    def test_default_values(self) -> None:
        """默认值应为 dev 环境"""
        settings = Settings()
        assert settings.env == Environment.DEV
        assert settings.debug is False
        assert settings.app_name == "aiops-agent"

    def test_env_override(self) -> None:
        """环境变量应覆盖默认值"""
        with patch.dict(os.environ, {"ENV": "production", "DEBUG": "false"}):
            settings = Settings()
            assert settings.env == Environment.PRODUCTION
            assert settings.is_production is True

    def test_env_case_insensitive(self) -> None:
        """环境名应大小写不敏感"""
        with patch.dict(os.environ, {"ENV": "Production"}):
            settings = Settings()
            assert settings.env == Environment.PRODUCTION

    def test_secret_str_masking(self) -> None:
        """SecretStr 应该在 str() 时脱敏"""
        config = LLMConfig(openai_api_key=SecretStr("sk-real-key-12345"))
        # str() 不泄露真实值
        assert "sk-real-key-12345" not in str(config.openai_api_key)

        assert config.openai_api_key.get_secret_value() == "sk-real-key-12345"

    def test_invalid_env_rejected(self) -> None:
        """无效的环境名应抛出 ValidationError"""
```

```python
        """错误码应正确映射 HTTP 状态码"""

        assert AIOpsError(code=ErrorCode.TOOL_TIMEOUT, message="").http_status == 504
        # 未映射的错误码默认 500
        assert AIOpsError(code=ErrorCode.INTERNAL_ERROR, message="").http_status == 500
```
