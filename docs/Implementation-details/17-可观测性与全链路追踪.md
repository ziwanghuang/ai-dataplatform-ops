### 1.2 在系统中的位置

可观测性是**横切关注点**，贯穿所有层：

```
用户请求 ──→ [API 层 Span] ──→ [Triage Span] ──→ [Diagnostic Span]
                 │                    │                   │
                 │              [LLM Call Span]     [Tool Call Span ×N]
                 │              (LangFuse 记录)     (OTel + Prometheus)
                 │                    │                   │
                 ▼                    ▼                   ▼
            Prometheus           LangFuse             Jaeger
            (指标)              (LLM 追踪)           (链路追踪)

                 └────────────────────┴───────────────────┘
                                      │
                                  Grafana

```

### 1.3 技术选型与 WHY

#### 1.3.1 WHY OpenTelemetry 而非 Datadog/New Relic

分布式追踪是可观测性的核心能力。市面主流方案有三类：商业 APM（Datadog、New Relic）、开源追踪系统（Jaeger 原生 SDK）、以及 CNCF 标准方案（OpenTelemetry）。

**决策：选择 OpenTelemetry。**

```
选型对比矩阵：

┌────────────────────┬──────────────────┬──────────────────┬──────────────────┐
│ 维度               │ OpenTelemetry    │ Datadog APM      │ Jaeger 原生 SDK  │
├────────────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 协议标准           │ OTLP (CNCF标准)  │ 私有协议          │ Thrift/gRPC      │
│ 供应商锁定         │ 无               │ 强锁定            │ 弱锁定           │
│ 语言覆盖           │ 11+ 语言          │ 8 语言            │ 6 语言           │
│ Python 支持        │ ★★★★★           │ ★★★★             │ ★★★             │
│ Go 支持            │ ★★★★★           │ ★★★★             │ ★★★★            │
│ 自动 instrumentation│ 丰富 (requests,  │ 内置 (ddtrace)    │ 需手动           │
│                    │  aiohttp, etc)   │                  │                  │
│ 后端可选性         │ Jaeger/Zipkin/   │ 仅 Datadog       │ 仅 Jaeger        │
│                    │  Tempo/任意 OTLP │                  │                  │
│ Context Propagation│ W3C + B3 + 自定义│ Datadog 格式      │ Jaeger 格式      │
│ Metrics 集成       │ 原生支持          │ 原生 (StatsD)     │ 无               │
│ Logs 关联          │ 原生支持          │ 需 DD Log Agent   │ 需额外配置       │
│ 社区活跃度         │ CNCF 毕业项目     │ 商业维护          │ CNCF 维护中      │
│ 月费用 (内部部署)  │ ¥0               │ ~¥50,000/月       │ ¥0               │
│ 学习曲线           │ 中等              │ 低（开箱即用）     │ 低               │
└────────────────────┴──────────────────┴──────────────────┴──────────────────┘
```

**WHY 不选 Datadog/New Relic：**

1. **成本不可接受**。Datadog APM 按 host + span 计费，以我们的规模（3-5 节点、每日 ~10 万 span），月费约 ¥50,000。对于面试 showcase 项目和中小团队，这完全不合理。即使在生产环境，大数据平台通常已有 Prometheus+Grafana 基础设施，引入商业 APM 是额外的成本和运维负担。

2. **供应商锁定风险**。Datadog 的 `ddtrace` SDK 生成的 Span 格式是私有的，想迁移到其他后端需要重写所有 instrumentation 代码。OpenTelemetry 的 OTLP 是 CNCF 标准协议，今天用 Jaeger，明天想换 Grafana Tempo 或阿里云 ARMS，只需改 Exporter 配置，**一行代码不用动**。

3. **数据不出域**。大数据平台场景涉及集群拓扑、告警内容等敏感信息。SaaS 方案需要把 trace 数据发到外网，这在很多企业是不允许的。OpenTelemetry + 自建 Jaeger 全链路内网部署，数据主权完全可控。

**WHY 不选 Jaeger 原生 SDK：**

Jaeger 的 `opentracing` SDK 已进入维护模式（OpenTracing 项目已合并入 OpenTelemetry）。新项目不应该再用 `jaeger-client-python`，而应该用 OTel SDK + OTLP Exporter → Jaeger。这样既享受 OTel 生态的丰富 instrumentation，又能用 Jaeger 做后端查询。

```python
# 错误做法：直接用 Jaeger SDK（已废弃）
# from jaeger_client import Config  # ← 不要这样做

# 正确做法：OTel SDK → OTLP → Jaeger
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
# Jaeger 16.0+ 原生支持 OTLP gRPC（端口 4317）
# WHY：OTel SDK 是未来标准，Jaeger 原生 SDK 已停止新功能开发
```

#### 1.3.2 WHY LangFuse 而非 LangSmith/Phoenix

LLM 应用需要专门的追踪系统（不只是分布式追踪）。通用 APM 无法追踪 Prompt 内容、Token 消耗、LLM 特有的延迟模式。

**决策：选择 LangFuse。**

```
LLM 追踪平台对比：

┌────────────────────┬──────────────────┬──────────────────┬──────────────────┐
│ 维度               │ LangFuse         │ LangSmith        │ Arize Phoenix    │
├────────────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 开源               │ ✅ MIT License   │ ❌ 商业           │ ✅ Apache 2.0    │
│ 自部署             │ ✅ Docker/K8s    │ ❌ 仅 SaaS       │ ✅ 需 Phoenix    │
│                    │   一键部署       │                  │   Server         │
│ LangGraph 集成     │ ✅ Callback +    │ ✅ 原生（同公司） │ ⚠️ 需适配       │
│                    │   手动 SDK       │                  │                  │
│ Prompt 管理        │ ✅ 版本追踪 +    │ ✅ Hub           │ ❌ 无            │
│                    │   在线编辑       │                  │                  │
│ 评估/标注          │ ✅ 内置          │ ✅ 功能丰富      │ ✅ 侧重 Eval    │
│ Token 成本追踪     │ ✅ 自动计算      │ ✅               │ ⚠️ 需手动       │
│ A/B 实验           │ ✅ 基于 Prompt   │ ⚠️ 需 LangChain │ ❌               │
│                    │   版本           │   Hub           │                  │
│ 数据导出           │ ✅ API + CSV     │ ⚠️ 有限         │ ✅ Parquet       │
│ 与 OTel 集成       │ ⚠️ 需桥接       │ ❌               │ ✅ OTel 原生    │
│ 月费用 (SaaS)      │ 免费 / ¥350起   │ ~¥700/月起       │ 免费 / ¥200起   │
│ 自部署费用         │ ¥0              │ 不支持           │ ¥0              │
│ 社区规模           │ GitHub 8k+ ★    │ 非开源           │ GitHub 4k+ ★    │
└────────────────────┴──────────────────┴──────────────────┴──────────────────┘
```

**WHY 选 LangFuse：**

1. **自部署能力是刚需**。大数据平台的诊断对话包含集群配置、告警详情、甚至部分业务数据。把这些 Prompt/Response 发送到 LangSmith 的 SaaS 服务？在大多数企业这是不可接受的安全风险。LangFuse 支持 `docker compose up` 一键自部署，数据完全留在内网。

2. **Prompt 版本管理 + 评估闭环**。LangFuse 内置 Prompt 版本管理，可以追踪"哪个版本的 Prompt 产生了什么质量的诊断结果"。配合内置的评估标注功能，形成"LLM 输出 → 人工标注 → 评估数据集 → Prompt 优化 → 再次评估"的完整闭环。LangSmith 也有类似功能，但无法自部署。

3. **成本可控**。LangFuse Cloud 免费版支持 5 万次追踪/月，自部署完全免费。对比 LangSmith 的 ¥700/月起步，更适合我们的规模。

**WHY 不选 Arize Phoenix：**

Phoenix 侧重 LLM 评估（Evals），追踪功能不如 LangFuse 完善。特别是 Prompt 管理和 A/B 实验功能缺失，而这是我们优化诊断 Prompt 的核心需求。此外 Phoenix 的 LangGraph 集成不如 LangFuse 成熟。

#### 1.3.3 WHY 三大支柱必须统一关联

可观测性不是三个独立系统的简单拼接。**如果 Traces、Metrics、Logs 之间没有关联关系，排障效率不会比没有可观测性好多少。**

```
没有关联时的痛点场景：

运维同学看到 Grafana 告警："Agent P99 延迟超过 30s"
  │
  ├─ 打开 Jaeger：搜 trace... 但不知道搜哪个 trace_id
  │   → 看到几千条 trace，哪条才是那个慢请求？
  │
  ├─ 打开 Kibana：搜日志... 但 error 日志没有 trace_id
  │   → 看到 "LLM timeout" 的日志，但不知道是哪次请求触发的
  │
  └─ 打开 LangFuse：搜 LLM 调用... 但 LangFuse 的 trace 和 Jaeger 的 trace 是两套 ID
      → 完全无法关联

排障时间：30 分钟 ~ 1 小时
```

**有统一关联后：**

```
运维同学看到告警，点击告警链接：
  │
  ├─ Grafana Panel → 点击慢请求数据点 → 跳转 Jaeger（带 trace_id）
  │   → 直接看到完整调用链：API → Triage → Diagnostic → 3 次 Tool 调用
  │   → 发现 tool.search_logs 耗时 25s（ES 查询超时）
  │
  ├─ Jaeger Span → trace_id 写入 structlog
  │   → Kibana 搜 trace_id = "abc123" → 看到完整上下文日志
  │
  └─ 同一 trace_id 也是 LangFuse 的 trace ID
      → LangFuse 看到这次诊断的 Prompt/Response/Token 消耗

排障时间：3 分钟
```

**实现关键：统一的 Trace ID 贯穿三层**

```python
# 关联的核心：trace_id 注入到 structlog
# python/src/aiops/core/logging.py 中的 add_trace_context processor

import structlog
from opentelemetry import trace

def add_trace_context(
    logger: structlog.types.WrappedLogger,
    method_name: str,
    event_dict: dict,
) -> dict:
    """将 OTel Trace 上下文注入日志字段"""
    span = trace.get_current_span()
    ctx = span.get_span_context()
    if ctx.is_valid:
        # WHY 用 hex 格式：与 Jaeger UI 和 W3C traceparent 保持一致
        event_dict["trace_id"] = format(ctx.trace_id, "032x")
        event_dict["span_id"] = format(ctx.span_id, "016x")
        event_dict["trace_flags"] = ctx.trace_flags
    return event_dict

# structlog 配置中添加：
# processors=[..., add_trace_context, ...]
# WHY 放在 processors 链中：每条日志自动携带 trace_id，无需手动传递
```

```python
# LangFuse trace 使用相同的 request_id 作为 trace ID
# WHY：这样在 LangFuse 中搜索的 trace_id 和在 Jaeger 中搜索的一致
trace = langfuse.trace(
    id=state["request_id"],  # 与 OTel 的 request_id 一致
    name="aiops_diagnosis",
    metadata={"otel_trace_id": current_span.get_span_context().trace_id},
)
# LangFuse 的 metadata 中记录 OTel trace_id，实现双向关联：
#   LangFuse → OTel：metadata.otel_trace_id
#   OTel → LangFuse：Span attribute "langfuse.trace_url"
```

这种统一关联的设计，使得从任何一个系统都能跳转到另外两个系统，形成闭环排障能力。

## 2. OpenTelemetry 集成

### 2.1 Python 侧初始化

```python
# python/src/aiops/observability/tracing.py
"""
OpenTelemetry 追踪初始化

设计要点：
- BatchSpanProcessor 批量导出（减少网络开销）
- 自动注入 service.name / version / env
- 提供装饰器简化 Agent/Tool 追踪
- 支持跨服务 context propagation（W3C TraceContext）
"""

from __future__ import annotations

import functools
from typing import Any, Callable

from opentelemetry import trace, context
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.resources import Resource
from opentelemetry.propagate import set_global_textmap
from opentelemetry.propagators.b3 import B3MultiFormat
from opentelemetry.trace import StatusCode

from aiops.core.config import settings
from aiops.core.logging import get_logger

logger = get_logger(__name__)


def setup_tracing() -> None:
    """初始化 OpenTelemetry 追踪"""
    if not settings.observability.enabled:
        logger.info("tracing_disabled")
        return

    resource = Resource.create({
        "service.name": settings.observability.service_name,
        "service.version": settings.version,
        "deployment.environment": settings.env.value,
    })

    provider = TracerProvider(resource=resource)

    # OTLP gRPC Exporter → Jaeger
    exporter = OTLPSpanExporter(
        endpoint=settings.observability.exporter_otlp_endpoint,
        insecure=True,  # 内网环境
    )
    provider.add_span_processor(BatchSpanProcessor(
        exporter,
        max_queue_size=2048,
        max_export_batch_size=512,
        schedule_delay_millis=5000,
    ))

    trace.set_tracer_provider(provider)

    # W3C TraceContext + B3 propagation（兼容 Go 侧）
    set_global_textmap(B3MultiFormat())

    logger.info(
        "tracing_initialized",
        endpoint=settings.observability.exporter_otlp_endpoint,
        service=settings.observability.service_name,
    )


def get_tracer(name: str = "aiops") -> trace.Tracer:
    """获取 Tracer 实例"""
    return trace.get_tracer(name, settings.version)


# ── 装饰器 ──

def trace_agent(agent_name: str):
    """Agent 节点追踪装饰器 — 自动创建 Span + 记录异常"""
    tracer = get_tracer("aiops.agent")

    def decorator(func: Callable) -> Callable:
        @functools.wraps(func)
        async def wrapper(state: dict, *args: Any, **kwargs: Any) -> dict:
            with tracer.start_as_current_span(
                f"agent.{agent_name}",
                attributes={
                    "agent.name": agent_name,
                    "request.id": state.get("request_id", ""),
                    "request.type": state.get("request_type", ""),
                    "user.id": state.get("user_id", ""),
                    "cluster.id": state.get("cluster_id", ""),
                },
            ) as span:
                try:
                    result = await func(state, *args, **kwargs)
                    span.set_attribute("agent.tokens_used", state.get("total_tokens", 0))
                    span.set_attribute("agent.error_count", state.get("error_count", 0))
                    return result
                except Exception as e:
                    span.record_exception(e)
                    span.set_status(StatusCode.ERROR, str(e))
                    raise
        return wrapper
    return decorator


def trace_tool_call(tool_name: str):
    """MCP 工具调用追踪装饰器"""
    tracer = get_tracer("aiops.mcp")

    def decorator(func: Callable) -> Callable:
        @functools.wraps(func)
        async def wrapper(*args: Any, **kwargs: Any) -> Any:
            with tracer.start_as_current_span(
                f"tool.{tool_name}",
                attributes={"tool.name": tool_name},
            ) as span:
                try:
                    result = await func(*args, **kwargs)
                    span.set_attribute("tool.result_size", len(str(result)))
                    return result
                except Exception as e:
                    span.record_exception(e)
                    span.set_status(StatusCode.ERROR, str(e))
                    raise
        return wrapper
    return decorator


def trace_llm_call(model: str, provider: str):
    """LLM 调用追踪装饰器"""
    tracer = get_tracer("aiops.llm")

    def decorator(func: Callable) -> Callable:
        @functools.wraps(func)
        async def wrapper(*args: Any, **kwargs: Any) -> Any:
            with tracer.start_as_current_span(
                f"llm.{provider}.{model}",
                attributes={
                    "llm.model": model,
                    "llm.provider": provider,
                },
            ) as span:
                result = await func(*args, **kwargs)
                if hasattr(result, "usage"):
                    span.set_attribute("llm.prompt_tokens", result.usage.prompt_tokens)
                    span.set_attribute("llm.completion_tokens", result.usage.completion_tokens)
                    span.set_attribute("llm.total_tokens", result.usage.total_tokens)
                if hasattr(result, "latency_ms"):
                    span.set_attribute("llm.latency_ms", result.latency_ms)
                return result
        return wrapper
    return decorator
```

### 2.2 Go 侧初始化

```go
// go/internal/pkg/tracing/tracing.go
package tracing

import (
	"context"
	"log"

	"go.opentelemetry.io/otel"
	"go.opentelemetry.io/otel/exporters/otlp/otlptrace/otlptracegrpc"
	"go.opentelemetry.io/otel/propagation"
	"go.opentelemetry.io/otel/sdk/resource"
	sdktrace "go.opentelemetry.io/otel/sdk/trace"
	semconv "go.opentelemetry.io/otel/semconv/v1.24.0"
)

// Setup 初始化 OTel 追踪（Go MCP Server 侧）
func Setup(ctx context.Context, serviceName, version, endpoint string) (func(), error) {
	exporter, err := otlptracegrpc.New(ctx,
		otlptracegrpc.WithEndpoint(endpoint),
		otlptracegrpc.WithInsecure(),
	)
	if err != nil {
		return nil, err
	}

	res, _ := resource.New(ctx,
		resource.WithAttributes(
			semconv.ServiceNameKey.String(serviceName),
			semconv.ServiceVersionKey.String(version),
		),
	)

	tp := sdktrace.NewTracerProvider(
		sdktrace.WithBatcher(exporter),
		sdktrace.WithResource(res),
	)

	otel.SetTracerProvider(tp)
	otel.SetTextMapPropagator(propagation.NewCompositeTextMapPropagator(
		propagation.TraceContext{},
		propagation.Baggage{},
	))

	cleanup := func() {
		_ = tp.Shutdown(ctx)
	}

	log.Printf("OTel tracing initialized: service=%s, endpoint=%s", serviceName, endpoint)
	return cleanup, nil
}
```

### 2.4 Agent 多步追踪的 Span 设计

一次完整的诊断请求会经过 Triage → Diagnostic → 多次工具调用 → Report 等多个步骤。如果不精心设计 Span 层级，在 Jaeger 中看到的只是一堆扁平的 Span，无法理解调用的因果关系和层级结构。

#### 2.4.1 完整 Span 树设计

```
一次诊断请求的完整 Span 树（Jaeger 视图）：

[agent.request] ─────────────────────────────────────────────── 35.2s
  │
  ├── [agent.triage] ──────────────────── 2.1s
  │     ├── [llm.deepseek.deepseek-v3] ─── 1.8s    ← 分诊 LLM 调用
  │     └── [triage.route_decision] ───── 0.1s      ← 路由决策记录
  │
  ├── [agent.diagnostic] ────────────────────────────────── 28.5s
  │     │
  │     ├── [diagnostic.round_1] ──────────────────── 12.3s
  │     │     ├── [llm.openai.gpt-4o] ──────── 3.2s  ← 生成假设
  │     │     ├── [diagnostic.hypothesis] ───── 0.01s  ← Event: 假设="HDFS NameNode GC"
  │     │     ├── [tool.hdfs_namenode_status] ── 2.1s  ← Go MCP Server Span
  │     │     ├── [tool.search_logs] ────────── 4.8s  ← Go MCP Server Span
  │     │     └── [llm.openai.gpt-4o] ──────── 2.1s  ← 分析工具结果
  │     │
  │     ├── [diagnostic.round_2] ──────────────────── 10.8s
  │     │     ├── [llm.openai.gpt-4o] ──────── 2.5s  ← 生成新假设
  │     │     ├── [diagnostic.hypothesis] ───── 0.01s  ← Event: 假设="YARN 队列满"
  │     │     ├── [tool.yarn_queue_status] ──── 1.5s
  │     │     ├── [tool.impala_query_profile] ── 3.2s
  │     │     └── [llm.openai.gpt-4o] ──────── 3.5s  ← 综合分析
  │     │
  │     └── [diagnostic.conclusion] ─────── 0.01s     ← Event: 置信度=0.85
  │
  ├── [agent.report] ──────────────── 3.8s
  │     └── [llm.openai.gpt-4o] ──── 3.5s             ← 生成诊断报告
  │
  └── [agent.knowledge_sink] ──────── 0.8s
        └── [rag.index_update] ────── 0.6s             ← 写入知识库
```

**WHY 设计这种层级结构：**
- **agent.request** 是根 Span，代表一次完整的用户请求。通过它可以看到端到端延迟。
- **diagnostic.round_N** 把每轮假设-验证循环包装成一个子 Span。这样在 Jaeger 中可以直观看到"诊断做了几轮、每轮耗时多少"，这是诊断效率的核心指标。
- **diagnostic.hypothesis** 不是 Span（瞬时事件），而是 Span Event，记录每轮假设的内容。WHY 用 Event 而不是独立 Span：假设记录没有持续时间，它是一个瞬时点，用 Event 更语义化。

#### 2.4.2 DiagnosticAgent 多步调用的 Span 创建

```python
# python/src/aiops/agent/nodes/diagnostic.py
"""
DiagnosticAgent 的 OTel Span 创建逻辑

设计要点：
- 每轮假设-验证循环是一个独立子 Span（diagnostic.round_N）
- 工具调用使用 @trace_tool_call 装饰器（自动创建子 Span）
- 关键决策点使用 Span Events 记录（假设内容、验证结果）
- WHY 不用 structlog 记录这些？日志是文本，Span 是结构化的时间线，
  在 Jaeger 中可以直观看到每一步的耗时和依赖关系
"""

from opentelemetry import trace
from opentelemetry.trace import StatusCode, SpanKind

from aiops.observability.tracing import get_tracer

tracer = get_tracer("aiops.diagnostic")


class DiagnosticAgentNode(BaseAgentNode):
    agent_name = "diagnostic"

    async def __call__(self, state: AgentState) -> AgentState:
        with tracer.start_as_current_span(
            "agent.diagnostic",
            kind=SpanKind.INTERNAL,
            attributes={
                "aiops.request_id": state["request_id"],
                "aiops.cluster_id": state.get("cluster_id", ""),
                "aiops.alert_type": state.get("alert_type", ""),
                "aiops.triage_route": state.get("triage_route", ""),
            },
        ) as root_span:
            round_num = 0
            max_rounds = state.get("max_diagnostic_rounds", 3)

            while round_num < max_rounds:
                round_num += 1
                # WHY 每轮一个子 Span：可以在 Jaeger 中清晰看到每轮的耗时分布
                with tracer.start_as_current_span(
                    f"diagnostic.round_{round_num}",
                    attributes={
                        "aiops.diagnostic.round": round_num,
                        "aiops.diagnostic.max_rounds": max_rounds,
                    },
                ) as round_span:
                    # 1. LLM 生成假设
                    hypothesis = await self._generate_hypothesis(state)
                    # Span Event：记录假设内容（瞬时事件，不是 Span）
                    round_span.add_event(
                        "hypothesis_generated",
                        attributes={
                            "aiops.hypothesis.content": hypothesis.description[:500],
                            "aiops.hypothesis.category": hypothesis.category,
                            "aiops.hypothesis.tools_needed": ",".join(hypothesis.tools),
                        },
                    )

                    # 2. 并行调用工具验证
                    tool_results = await self._execute_tools(state, hypothesis.tools)

                    # 3. LLM 分析结果
                    analysis = await self._analyze_results(state, hypothesis, tool_results)

                    # Span Event：记录验证结果
                    round_span.add_event(
                        "hypothesis_verified",
                        attributes={
                            "aiops.hypothesis.verified": analysis.is_confirmed,
                            "aiops.hypothesis.confidence": analysis.confidence,
                            "aiops.hypothesis.needs_more_data": analysis.needs_more_data,
                        },
                    )

                    if analysis.confidence >= 0.8 and not analysis.needs_more_data:
                        root_span.add_event(
                            "diagnosis_concluded",
                            attributes={
                                "aiops.diagnosis.rounds_used": round_num,
                                "aiops.diagnosis.confidence": analysis.confidence,
                                "aiops.diagnosis.root_cause": analysis.root_cause[:200],
                            },
                        )
                        break

            # 记录最终指标到 Span
            root_span.set_attribute("aiops.diagnostic.total_rounds", round_num)
            root_span.set_attribute("aiops.diagnostic.final_confidence",
                                    state.get("diagnosis", {}).get("confidence", 0))
            root_span.set_attribute("aiops.diagnostic.tools_called",
                                    state.get("tools_called_count", 0))

            return state
```

```python
# python/src/aiops/observability/span_events.py

WHY 用 Span Events 而不是独立 Span：
```

#### 2.4.5 自定义 SpanProcessor：智能采样

生产环境中，并非所有请求都需要保留完整 trace。但错误请求和慢请求的 trace 必须保留，这对排障至关重要。

```python
# python/src/aiops/observability/sampling.py
"""
自定义 SpanProcessor：尾部采样（Tail Sampling）

WHY 用尾部采样而非头部采样：
- 头部采样（HeadSampler）在请求开始时决定是否采样
  → 问题：此时不知道请求是否会出错，可能丢掉关键错误 trace
- 尾部采样在 Span 结束后决定
  → 可以基于结果决策：错误请求 100% 保留，正常请求采样 10%

采样策略：
1. 错误请求 → 100% 保留（排障刚需）
2. 慢请求（>15s）→ 100% 保留（性能分析刚需）
3. 正常请求 → 10% 采样（降低存储成本）
4. 健康检查/metrics → 0%（完全跳过）
"""

import random
from opentelemetry.sdk.trace import SpanProcessor, ReadableSpan
from opentelemetry.trace import StatusCode

from aiops.core.logging import get_logger

logger = get_logger(__name__)


class TailSamplingProcessor(SpanProcessor):
    """基于结果的尾部采样 SpanProcessor"""

    def __init__(
        self,
        delegate: SpanProcessor,
        normal_sample_rate: float = 0.1,
        slow_threshold_seconds: float = 15.0,
    ) -> None:
        self._delegate = delegate
        self._normal_rate = normal_sample_rate
        self._slow_threshold = slow_threshold_seconds
        # WHY 缓存已决策的 trace_id：
        # 同一个 trace 下的所有 Span 要么全部保留，要么全部丢弃
        # 否则 Jaeger 中看到的 trace 是残缺的
        self._sampled_traces: set[int] = set()
        self._dropped_traces: set[int] = set()

    def on_start(self, span, parent_context=None) -> None:
        """Span 开始时不做决策（尾部采样的核心）"""
        pass

    def on_end(self, span: ReadableSpan) -> None:
        """Span 结束时决定是否保留"""
        trace_id = span.context.trace_id

        # 如果同一 trace 已有决策，保持一致
        if trace_id in self._sampled_traces:
            self._delegate.on_end(span)
            return
        if trace_id in self._dropped_traces:
            return

        # 仅对根 Span（无 parent）做采样决策
        if span.parent is not None:
            # 非根 Span 暂时缓存，等根 Span 决策
            # 简化实现：非根 Span 直接发送（实际生产中建议用 OTel Collector 做尾部采样）
            self._delegate.on_end(span)
            return

        should_sample = self._decide(span)

        if should_sample:
            self._sampled_traces.add(trace_id)
            self._delegate.on_end(span)
        else:
            self._dropped_traces.add(trace_id)

        # 防止内存泄漏：定期清理旧 trace_id
        if len(self._sampled_traces) > 10000:
            self._sampled_traces.clear()
        if len(self._dropped_traces) > 10000:
            self._dropped_traces.clear()

    def _decide(self, span: ReadableSpan) -> bool:
        """采样决策逻辑"""
        # 规则 1：错误请求 100% 保留
        if span.status.status_code == StatusCode.ERROR:
            return True

        # 规则 2：慢请求 100% 保留
        duration_s = (span.end_time - span.start_time) / 1e9
        if duration_s > self._slow_threshold:
            return True

        # 规则 3：正常请求按比例采样
        return random.random() < self._normal_rate

    def shutdown(self) -> None:
        self._delegate.shutdown()

    def force_flush(self, timeout_millis: int = 30000) -> bool:
        return self._delegate.force_flush(timeout_millis)
```

```python
# 在 setup_tracing() 中使用 TailSamplingProcessor
def setup_tracing() -> None:
    # ... 现有代码 ...
    
    base_processor = BatchSpanProcessor(exporter, ...)
```

## 3. LangFuse 集成

### 3.1 LangFuse 初始化

```python
# python/src/aiops/observability/langfuse_integration.py
"""
LangFuse LLM 追踪集成

追踪内容：
- 每次 LLM 调用的 Prompt、Response、Token、Cost、Latency
- Agent 会话级追踪（一次诊断 = 一个 LangFuse Trace）
- Prompt 版本追踪（哪个版本的 Prompt 产生了什么结果）
- 评估分数记录（置信度、RAGAS 指标）
"""

from __future__ import annotations

from datetime import datetime, timezone
from typing import Any

from langfuse import Langfuse
from langfuse.callback import CallbackHandler

from aiops.core.config import settings
from aiops.core.logging import get_logger

logger = get_logger(__name__)


def get_langfuse_handler() -> CallbackHandler | None:
    """获取 LangFuse callback handler（用于 LangChain/LangGraph 集成）"""
    if not settings.observability.langfuse_public_key:
        return None

    return CallbackHandler(
        public_key=settings.observability.langfuse_public_key,
        secret_key=settings.observability.langfuse_secret_key.get_secret_value(),
        host=settings.observability.langfuse_host,
    )


class LangFuseTracker:
    """手动 LangFuse 追踪（用于非 LangChain 场景）"""

    def __init__(self) -> None:
        self._client: Langfuse | None = None

    def _get_client(self) -> Langfuse | None:
        if self._client is None and settings.observability.langfuse_public_key:
            self._client = Langfuse(
                public_key=settings.observability.langfuse_public_key,
                secret_key=settings.observability.langfuse_secret_key.get_secret_value(),
                host=settings.observability.langfuse_host,
            )
        return self._client

    def start_trace(self, request_id: str, user_id: str = "", metadata: dict | None = None):
        """开始一个诊断会话的追踪"""
        client = self._get_client()
        if not client:
            return None

        trace = client.trace(
            id=request_id,
            name="aiops_diagnosis",
            user_id=user_id,
            metadata=metadata or {},
            tags=["aiops", settings.env.value],
        )
        return trace

    def log_llm_call(
        self,
        trace,
        model: str,
        messages: list[dict],
        response: str,
        usage: dict,
        latency_ms: int,
        metadata: dict | None = None,
    ):
        """记录一次 LLM 调用"""
        if trace is None:
            return

        generation = trace.generation(
            name=f"llm_{model}",
            model=model,
            input=messages,
            output=response,
            usage={
                "input": usage.get("prompt_tokens", 0),
                "output": usage.get("completion_tokens", 0),
                "total": usage.get("total_tokens", 0),
            },
            metadata={
                "latency_ms": latency_ms,
                "provider": metadata.get("provider", ""),
                "prompt_version": settings.prompt_version,
                **(metadata or {}),
            },
        )
        return generation

    def log_tool_call(
        self,
        trace,
        tool_name: str,
        params: dict,
        result: str,
        duration_ms: int,
        status: str = "success",
    ):
        """记录一次工具调用"""
        if trace is None:
            return

        trace.span(
            name=f"tool_{tool_name}",
            input=params,
            output=result[:2000],
            metadata={
                "duration_ms": duration_ms,
                "status": status,
            },
        )

    def log_score(
        self,
        trace,
        name: str,
        value: float,
        comment: str = "",
    ):
        """记录评估分数（如置信度、RAGAS 指标）"""
        if trace is None:
            return

        trace.score(
            name=name,
            value=value,
            comment=comment,
        )

    def flush(self):
        """刷新缓冲区（确保数据发送到 LangFuse）"""
        client = self._get_client()
        if client:
            client.flush()
```

### 3.3 LangFuse 评估闭环

LangFuse 不仅是追踪系统，更是 **LLM 质量持续改进** 的核心平台。通过"追踪 → 标注 → 评估 → 优化"的闭环，实现 Prompt 的数据驱动优化。

```
LangFuse 评估闭环流程：

                    ┌─────────────────────────────┐
                    │   生产环境 Agent 运行        │
                    │   (LangFuse Traces 自动记录) │
                    └──────────┬──────────────────┘
```

#### 3.3.1 从 LangFuse 导出评估数据集

```python
# python/src/aiops/evaluation/dataset_builder.py
"""
从 LangFuse 导出生产 traces → 生成评估数据集

WHY 从生产 traces 构建评估集：
1. 真实场景覆盖率高 — 测试用例难以覆盖生产中的长尾情况
2. 自动发现 bad cases — 低分 traces 直接进入回归测试
3. 数据驱动优化 — 不靠拍脑袋调 Prompt，靠数据说话

工作流：
  langfuse.get_traces(score < 3.0)
    → 筛选低质量诊断
    → 提取 input/expected_output
    → 存入 evaluation dataset
"""

from __future__ import annotations

from dataclasses import dataclass
from datetime import datetime, timezone, timedelta

from langfuse import Langfuse

from aiops.core.config import settings
from aiops.core.logging import get_logger

logger = get_logger(__name__)


@dataclass
class EvalCase:
    """单条评估用例"""
    trace_id: str
    input_alert: dict          # 原始告警输入
    expected_diagnosis: str    # 人工标注的正确诊断
    actual_diagnosis: str      # Agent 实际输出
    score: float               # 人工评分
    prompt_version: str        # 使用的 Prompt 版本
    metadata: dict             # 额外元数据


class DatasetBuilder:
    """从 LangFuse traces 构建评估数据集"""

    def __init__(self) -> None:
        self._client = Langfuse(
            public_key=settings.observability.langfuse_public_key,
            secret_key=settings.observability.langfuse_secret_key.get_secret_value(),
            host=settings.observability.langfuse_host,
        )

    def build_bad_cases(
        self,
        days_back: int = 7,
        score_threshold: float = 3.0,
        limit: int = 100,
    ) -> list[EvalCase]:
        """
        提取低分 traces 构建 bad cases 数据集

        WHY score_threshold=3.0：
        我们用 1-5 分制，3 分以下表示"诊断结果不准或不完整"，
        这些 case 最有优化价值。
        """
        # 获取近 N 天的低分 traces
        traces = self._client.fetch_traces(
            name="aiops_diagnosis",
            from_timestamp=datetime.now(timezone.utc) - timedelta(days=days_back),
            limit=limit,
            order_by="scores.diagnosis_quality",
        )

        eval_cases = []
        for t in traces.data:
            # 筛选有人工评分且低于阈值的
            quality_score = self._get_score(t, "diagnosis_quality")
            if quality_score is None or quality_score >= score_threshold:
                continue

            eval_cases.append(EvalCase(
                trace_id=t.id,
                input_alert=t.input or {},
                expected_diagnosis=self._get_annotation(t, "correct_diagnosis"),
                actual_diagnosis=t.output or "",
                score=quality_score,
                prompt_version=t.metadata.get("prompt_version", "unknown"),
                metadata={
                    "cluster_id": t.metadata.get("cluster", ""),
                    "alert_type": t.metadata.get("alert_type", ""),
                    "timestamp": t.timestamp.isoformat(),
                },
            ))

        logger.info(
            "bad_cases_extracted",
            total_traces=len(traces.data),
            bad_cases=len(eval_cases),
            days_back=days_back,
        )
        return eval_cases

    @staticmethod
    def _get_score(trace, score_name: str) -> float | None:
        """从 trace 中提取指定评分"""
        for s in (trace.scores or []):
            if s.name == score_name:
                return s.value
        return None

    @staticmethod
    def _get_annotation(trace, annotation_name: str) -> str:
        """从 trace 中提取人工标注"""
        for s in (trace.scores or []):
            if s.name == annotation_name:
                return s.comment or ""
        return ""
```

#### 3.3.2 LangFuse Prompt 版本管理

```python
# python/src/aiops/observability/prompt_management.py
"""
基于 LangFuse 的 Prompt 版本管理

WHY 不把 Prompt 硬编码在代码中：
1. 修改 Prompt 不需要重新部署 → 运维同学可以直接在 LangFuse UI 编辑
2. 自动追踪"哪个版本的 Prompt 产生了什么质量的结果"
3. 支持灰度发布（不同请求使用不同 Prompt 版本）

工作流：
  1. 在 LangFuse UI 中编辑 Prompt → 创建新版本
  2. Agent 运行时从 LangFuse 拉取最新版（有本地缓存）
  3. 每次 LLM 调用记录使用的 Prompt 版本
  4. 评估时按版本分组对比效果
"""

from langfuse import Langfuse

from aiops.core.config import settings
from aiops.core.logging import get_logger

logger = get_logger(__name__)


class PromptManager:
    """从 LangFuse 加载和管理 Prompt 版本"""

    def __init__(self) -> None:
        self._client = Langfuse(
            public_key=settings.observability.langfuse_public_key,
            secret_key=settings.observability.langfuse_secret_key.get_secret_value(),
            host=settings.observability.langfuse_host,
        )
        # 本地缓存，避免每次 LLM 调用都请求 LangFuse
        self._cache: dict[str, tuple[str, int]] = {}  # name → (content, version)

    def get_prompt(self, name: str, fallback: str = "") -> tuple[str, int]:
        """
        获取 Prompt（带本地缓存）

        Returns: (prompt_content, version_number)
        """
        if name in self._cache:
            return self._cache[name]

        try:
            prompt = self._client.get_prompt(name)
            content = prompt.prompt
            version = prompt.version
            self._cache[name] = (content, version)
            logger.info("prompt_loaded", name=name, version=version)
            return content, version
        except Exception as e:
            logger.warning("prompt_load_failed", name=name, error=str(e))
            return fallback, 0

    def clear_cache(self) -> None:
        """清除缓存（用于测试或强制刷新）"""
        self._cache.clear()
```

#### 3.3.3 LangFuse A/B 实验

```python
# python/src/aiops/observability/ab_experiment.py
"""
基于 LangFuse 的 Prompt A/B 实验

WHY 需要 A/B 实验：
修改 Prompt 后，不能直接全量上线。需要验证"新 Prompt 确实比旧的好"。
A/B 实验流程：
  1. 设定实验比例（如 20% 流量用新 Prompt）
  2. 在 LangFuse 中按 Prompt 版本分组
  3. 对比两组的诊断质量分数
  4. 统计显著后全量切换

WHY 不用 Feature Flag 系统（如 LaunchDarkly）：
  Prompt 版本管理天然在 LangFuse 中，用 LangFuse 的版本 + tag 做分组
  比引入额外 Feature Flag 系统更简单，且评估数据直接关联。
"""

import hashlib
from aiops.core.logging import get_logger

logger = get_logger(__name__)


class PromptABExperiment:
    """简单的 A/B 实验分流器"""

    def __init__(
        self,
        experiment_name: str,
        control_prompt: str,
        treatment_prompt: str,
        treatment_ratio: float = 0.2,
    ) -> None:
        self.experiment_name = experiment_name
        self.control_prompt = control_prompt
        self.treatment_prompt = treatment_prompt
        self.treatment_ratio = treatment_ratio

    def get_prompt(self, request_id: str) -> tuple[str, str]:
        """
        基于 request_id 的确定性分流

        WHY 用 request_id hash 而非 random：
        确保同一 request_id 在重试时分到同一组，结果可复现

        Returns: (prompt_content, variant_name)
        """
        hash_val = int(hashlib.md5(
            f"{self.experiment_name}:{request_id}".encode()
        ).hexdigest(), 16)
        bucket = (hash_val % 1000) / 1000.0

        if bucket < self.treatment_ratio:
            return self.treatment_prompt, "treatment"
        return self.control_prompt, "control"
```

> **🔧 工程难点：Prometheus 指标的分类设计与告警疲劳防治——AI Agent 系统的指标比传统应用多 3 倍**
>
> **挑战**：AI Agent 系统的可观测性维度比传统 Web 应用复杂得多——除了常规的 HTTP 延迟/错误率/QPS，还需要监控：LLM 调用（每个 model 的延迟/Token/成本/重试率）、Agent 流程（每个节点的耗时/路由分支命中率）、MCP 工具（42 个工具各自的成功率/延迟/熔断状态）、RAG 检索（Dense/Sparse 延迟、Reranker 延迟、缓存命中率）、安全（注入检测率/RBAC 拒绝率/脱敏触发率）。粗略估计有 200+ 个独立指标，如果不分类管理，Grafana Dashboard 会变成"数字墙"——信息太多等于没有信息。更实际的问题是**告警疲劳**——如果每个指标都配置告警规则，运维人员每天收到 50+ 条告警通知就会开始忽略所有告警（包括真正重要的）。
>
> **解决方案**：将 200+ 指标分为 4 层金字塔：**Layer 1（SLI/SLO 级别，~10 个）**——最核心的健康指标，如"端到端诊断成功率 > 95%"、"P99 延迟 < 30s"、"日均 Token 成本 < ¥500"，这些指标配置 PagerDuty 电话告警（只有这一层才打电话）；**Layer 2（模块级别，~30 个）**——每个主要模块的核心指标（Triage 分诊延迟、Diagnostic 收敛轮次、LLM 重试率、MCP 熔断状态），配置企微 Bot 告警；**Layer 3（组件级别，~80 个）**——细粒度指标（单个工具延迟、单个模型 Token 消耗、RAG 缓存命中率），只在 Grafana Dashboard 展示，不配置告警；**Layer 4（调试级别，~80 个）**——临时指标（Prompt 版本 A/B 对比、新中间件灰度监控），默认关闭，排查问题时临时开启。告警规则采用"升级策略"（§6.1）——P2 告警 5 分钟内无确认 → 升级为 P1 → 10 分钟无确认 → 升级为 P0 电话通知。SLO 定义（§6.2）使用"Error Budget"机制——每月允许 5% 的"不达标时间"（~36 小时），当月错误预算消耗 >80% 时自动冻结非紧急变更，保护系统稳定性。

### 4.4 自定义 Collector 与指标聚合

#### 4.4.1 WHY 使用自定义 Registry

在 §4.1 中我们使用了 `REGISTRY = CollectorRegistry()` 而非默认的 `REGISTRY`。这个决策值得展开。

```python
# 默认 registry 的问题：
from prometheus_client import Counter
# 这个 Counter 注册到 GLOBAL default registry
my_counter = Counter("my_counter", "...")  

# 问题 1：与第三方库冲突
# FastAPI、uvicorn、grpc 等库可能也往默认 registry 注册指标
# 导致 /metrics 端点输出一大堆无关指标，甚至名称冲突报错

# 问题 2：测试困难

# 一个测试修改了 counter，会影响另一个测试

# 问题 3：无法控制暴露范围
# 无法选择性暴露指标（比如只暴露 aiops_* 指标给外部）

# ═══════════════════════════════════════
# 正确做法：自定义 Registry

from prometheus_client import CollectorRegistry

# 1. 完全隔离 — 只包含我们定义的 aiops_* 指标
# 2. 测试友好 — 每个测试可以创建独立 registry
# 3. 多端点支持 — 可以有 /metrics（外部）和 /internal-metrics（内部）
#    分别暴露不同 registry 的指标
```

#### 4.4.2 自定义 Collector：Agent 运行时状态

除了标准的 Counter/Histogram，我们还需要收集一些**动态计算**的指标（如当前队列深度、缓存大小等）。这些用 `Collector` 接口实现。

```python
# python/src/aiops/observability/collectors.py
"""
自定义 Prometheus Collector

WHY 需要自定义 Collector：
标准 Counter/Gauge/Histogram 是"推送式"——代码执行时主动 inc()/observe()。
但有些指标是"拉取式"——只有 Prometheus scrape 时才需要计算，
例如"当前 Redis 队列深度"或"LRU 缓存当前大小"。
用自定义 Collector 可以做到"按需计算"，避免不必要的开销。
"""

from prometheus_client.core import (
    GaugeMetricFamily,
    CounterMetricFamily,
    REGISTRY,
)
from prometheus_client import CollectorRegistry

from aiops.core.logging import get_logger

logger = get_logger(__name__)


class AgentRuntimeCollector:
    """
    收集 Agent 运行时状态指标

    这些指标在每次 Prometheus scrape 时动态计算，
    而不是在业务代码中主动推送。
    """

    def __init__(self, registry: CollectorRegistry) -> None:
        self._registry = registry
        registry.register(self)

    def collect(self):
        """Prometheus scrape 时调用此方法"""
        # 1. HITL 审批队列深度
        # WHY 用 Collector 而非 Gauge：队列深度需要查 Redis，
        # 不应该每次请求都查，只在 scrape 时查一次
        queue_depth = self._get_hitl_queue_depth()
        g = GaugeMetricFamily(
            "aiops_hitl_queue_depth",
            "Current HITL approval queue depth",
        )
        g.add_metric([], queue_depth)
        yield g

        # 2. Semantic Cache 条目数和内存占用
        cache_stats = self._get_cache_stats()
        g = GaugeMetricFamily(
            "aiops_cache_entries",
            "Number of entries in semantic cache",
        )
        g.add_metric([], cache_stats.get("entries", 0))
        yield g

        g = GaugeMetricFamily(
            "aiops_cache_memory_bytes",
            "Memory used by semantic cache in bytes",
        )
        g.add_metric([], cache_stats.get("memory_bytes", 0))
        yield g

        # 3. RAG 知识库文档数
        g = GaugeMetricFamily(
            "aiops_rag_documents_total",
            "Total documents in RAG knowledge base",
            labels=["collection"],
        )
        for collection, count in self._get_rag_doc_counts().items():
            g.add_metric([collection], count)
        yield g

        # 4. 熔断器状态汇总
        g = GaugeMetricFamily(
            "aiops_circuit_breaker_open_count",
            "Number of currently open circuit breakers",
        )
        g.add_metric([], self._get_open_circuit_count())
        yield g

    def _get_hitl_queue_depth(self) -> int:
        """查询 Redis 获取 HITL 队列深度"""
        try:
            from aiops.hitl.redis_store import get_redis
            redis = get_redis()
            return redis.llen("aiops:hitl:pending") if redis else 0
        except Exception:
            return 0

    def _get_cache_stats(self) -> dict:
        """获取语义缓存统计"""
        try:
            from aiops.llm.cache import get_cache
            cache = get_cache()
            return cache.stats() if cache else {}
        except Exception:
            return {}

    def _get_rag_doc_counts(self) -> dict[str, int]:
        """获取各 RAG collection 文档数"""
        try:
            from aiops.rag.store import get_vectorstore
            store = get_vectorstore()
            return store.collection_stats() if store else {}
        except Exception:
            return {}

    def _get_open_circuit_count(self) -> int:
        """统计当前打开的熔断器数量"""
        try:
            from aiops.mcp.circuit_breaker import get_all_breakers
            return sum(1 for b in get_all_breakers() if b.state == "open")
        except Exception:
            return 0
```

#### 4.4.3 指标聚合策略

```yaml
# infra/monitoring/prometheus/recording_rules.yml
#
# WHY 预聚合（Recording Rules）：
# 1. 复杂 PromQL 查询在 Dashboard 刷新时实时计算，对 Prometheus 压力大
# 2. 预聚合后的指标查询延迟从 ~2s 降到 ~50ms
# 3. 仪表盘加载更快，用户体验更好
#
# 聚合策略：每 5 分钟计算一次，覆盖最常用的 Dashboard 查询
groups:
  - name: aiops_aggregations
    interval: 5m
    rules:
      # Agent 成功率（5 分钟窗口）
      - record: aiops:agent_success_rate:5m
        expr: |
          sum(rate(aiops_agent_requests_total{status="success"}[5m]))
          / sum(rate(aiops_agent_requests_total[5m]))

      # Agent P50/P99 延迟（5 分钟窗口）
      - record: aiops:agent_latency_p50:5m
        expr: |
          histogram_quantile(0.5,
            sum(rate(aiops_agent_duration_seconds_bucket[5m])) by (le))

      - record: aiops:agent_latency_p99:5m
        expr: |
          histogram_quantile(0.99,
            sum(rate(aiops_agent_duration_seconds_bucket[5m])) by (le))

      # LLM 每小时成本
      - record: aiops:llm_cost_per_hour:1h
        expr: |
          sum(increase(aiops_llm_cost_usd_total[1h])) by (model)

      # 语义缓存命中率
      - record: aiops:cache_hit_rate:5m
        expr: |
          sum(rate(aiops_llm_cache_hits_total[5m]))
          / clamp_min(sum(rate(aiops_llm_cache_requests_total[5m])), 1)

      # MCP 工具成功率（按工具分组）
      - record: aiops:tool_success_rate:5m
        expr: |
          sum(rate(aiops_mcp_tool_calls_total{status="success"}[5m])) by (tool)
          / sum(rate(aiops_mcp_tool_calls_total[5m])) by (tool)

      # 平均诊断轮次
      - record: aiops:avg_diagnostic_rounds:1h
        expr: |
          sum(rate(aiops_agent_collection_rounds_sum[1h]))
          / sum(rate(aiops_agent_collection_rounds_count[1h]))
```

```yaml
# infra/monitoring/alerting/aiops_rules.yml
groups:
  - name: aiops_agent
    interval: 30s
    rules:
      # Agent 错误率 > 10%
      - alert: AgentHighErrorRate
        expr: |
          sum(rate(aiops_agent_requests_total{status="error"}[5m]))
          / sum(rate(aiops_agent_requests_total[5m])) > 0.1
        for: 5m
        labels:
          severity: critical
          team: aiops
        annotations:
          summary: "Agent 错误率超过 10%"
          description: "当前错误率 {{ $value | humanizePercentage }}，持续 5 分钟"

      # Agent P99 延迟 > 30s
      - alert: AgentHighLatency
        expr: |
          histogram_quantile(0.99,
            sum(rate(aiops_agent_duration_seconds_bucket[5m])) by (le)
          ) > 30
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Agent P99 延迟超过 30 秒"
          description: "当前 P99 延迟 {{ $value | humanizeDuration }}"

  - name: aiops_llm
    interval: 30s
    rules:
      # 所有 LLM 供应商错误率 > 50%
      - alert: LLMAllProvidersDown
        expr: |
          sum(rate(aiops_llm_calls_total{status=~"error|timeout"}[5m]))
          / sum(rate(aiops_llm_calls_total[5m])) > 0.5
```

### 6.1 告警升级策略

告警不是发出去就完了。需要确保"该看到的人看到了"、"该处理的问题被处理了"。

#### 6.1.1 三级告警升级

```
告警升级链：

Level 1: 企微群通知（立即）
  │  所有 warning/critical 告警 → AIOps 告警群
  │  值班人 5 分钟内确认
  │
  │  未确认？
  ▼
Level 2: 电话通知（5 分钟后）
  │  critical 告警未确认 → 电话轮呼值班人
  │  按顺序：主值班 → 备值班 → 组长
  │
  │  仍未确认？
  ▼
Level 3: 升级主管（15 分钟后）
  │  critical 告警 15 分钟无人处理
  │  → 通知团队主管 + 发送邮件
  │  → 同时在企微群 @all
```

**WHY 需要告警升级：** AI Agent 在夜间自动运行，如果凌晨 3 点 Agent 错误率飙升但没人处理，可能导致大量诊断请求积压。三级升级确保 critical 告警在 15 分钟内有人响应。

### 6.2 SLO/SLI 定义

#### 6.2.1 WHY 需要 SLO

传统运维只看"系统是不是活着"（UP/DOWN）。SLO（Service Level Objective）提供了更精细的质量度量："系统够不够好？"

```
没有 SLO 的问题：

运维："系统正常啊，没有告警"
用户："诊断结果经常不准，我已经不信任这个系统了"
运维："...但是系统确实没挂"

有了 SLO 的对话：

运维："本周 Agent 可用性 99.2%，低于 SLO 目标 99.5%"
运维："P99 延迟 35s，超过 SLO 目标 30s"
运维："Error Budget 已消耗 85%，需要暂停新功能开发，优先修复稳定性"
产品："了解，这周 sprint 调整为稳定性专项"
```

**WHY 从 SRE 角度管理 AI 系统质量：** AI Agent 不同于传统服务，其"质量"不仅包括可用性和延迟，还包括诊断准确率。SLO 框架让这些质量指标有了量化目标和管理手段。

#### 6.2.2 SLI/SLO 定义表

```
┌──────────────────┬─────────────────────────────────────┬──────────┬──────────┐
│ SLI (指标)        │ 定义                                 │ SLO 目标  │ 测量窗口  │
├──────────────────┼─────────────────────────────────────┼──────────┼──────────┤
│ 可用性            │ 成功请求数 / 总请求数                │ ≥ 99.5%  │ 28 天    │
│ (Availability)   │ success / (success + error)          │          │ 滚动     │
├──────────────────┼─────────────────────────────────────┼──────────┼──────────┤
│ 延迟              │ Agent 端到端 P99 延迟                │ ≤ 30s    │ 28 天    │
│ (Latency)        │ histogram_quantile(0.99, ...)        │          │ 滚动     │
├──────────────────┼─────────────────────────────────────┼──────────┼──────────┤
│ 诊断质量          │ 诊断置信度中位数                     │ ≥ 0.7    │ 7 天     │
│ (Quality)        │ histogram_quantile(0.5, confidence)  │          │ 滚动     │
├──────────────────┼─────────────────────────────────────┼──────────┼──────────┤
│ 工具可用性        │ MCP 工具成功率                       │ ≥ 95%    │ 28 天    │
│ (Tool Health)    │ success / total (按工具分组)          │          │ 滚动     │
├──────────────────┼─────────────────────────────────────┼──────────┼──────────┤
│ 诊断完成率        │ 完成诊断 / 接收请求                  │ ≥ 98%    │ 28 天    │
│ (Completion)     │ 排除用户主动取消                      │          │ 滚动     │
└──────────────────┴─────────────────────────────────────┴──────────┴──────────┘
```

**WHY 可用性目标是 99.5% 而非 99.9%：**
- AI Agent 依赖 LLM API（外部依赖），LLM 本身的可用性约 99.5-99.9%
- 我们还依赖 MCP 工具（查 HDFS/YARN/ES），这些大数据组件本身就可能不稳定
- 99.5% 的 SLO 意味着 28 天允许 ~3.4 小时的不可用时间，对内部工具已经足够

**WHY 延迟目标是 P99 ≤ 30s：**
- 一次完整诊断包含 2-3 轮 LLM 调用 + 3-5 次工具调用
- 单次 LLM 调用 P99 约 5-8s，工具调用 P99 约 2-5s
- 30s 是"用户能接受等待"的上限（超过 30s 用户会觉得系统卡住了）

#### 6.2.3 Error Budget 计算

```python
# python/src/aiops/observability/slo.py
"""
Error Budget 计算与追踪

Error Budget = 1 - SLO 目标
例：SLO = 99.5% → Error Budget = 0.5%（28 天内允许约 3.4 小时不可用）

WHY 用 Error Budget 而非简单的"告警"：
1. 告警是"现在出问题了"，Error Budget 是"还剩多少犯错空间"
2. Error Budget 消耗速度决定决策：
   - 消耗 < 50%：正常迭代新功能
   - 消耗 50-80%：谨慎发布，加强测试
   - 消耗 > 80%：冻结新功能，专注稳定性
   - 消耗 100%：全员投入修复
"""

from datetime import datetime, timezone, timedelta
from dataclasses import dataclass


@dataclass
class ErrorBudgetStatus:
    """Error Budget 当前状态"""
    slo_target: float        # 如 0.995
    actual_availability: float  # 如 0.997
    budget_total: float      # 总预算（如 0.005）
    budget_remaining: float  # 剩余预算
    budget_consumed_pct: float  # 已消耗百分比
    window_start: datetime
    window_end: datetime
    burn_rate: float         # 消耗速率（>1 表示消耗加速）


def calculate_error_budget(
    slo_target: float,
    success_count: int,
    total_count: int,
    window_days: int = 28,
) -> ErrorBudgetStatus:
    """
    计算 Error Budget 状态

    Args:
        slo_target: SLO 目标（如 0.995）
        success_count: 窗口期内成功请求数
        total_count: 窗口期内总请求数
        window_days: SLO 窗口天数
    """
    now = datetime.now(timezone.utc)
    window_start = now - timedelta(days=window_days)

    budget_total = 1.0 - slo_target  # 0.005
    actual_availability = success_count / max(total_count, 1)
    actual_error_rate = 1.0 - actual_availability

    budget_remaining = budget_total - actual_error_rate
    budget_consumed_pct = (actual_error_rate / budget_total) * 100 if budget_total > 0 else 0

    # 消耗速率：如果按当前速度消耗，是否会在窗口期内耗尽
    # burn_rate > 1 表示消耗速度超过预期
    elapsed_fraction = 1.0  # 简化：假设整个窗口已过
    burn_rate = budget_consumed_pct / (elapsed_fraction * 100) if elapsed_fraction > 0 else 0

    return ErrorBudgetStatus(
        slo_target=slo_target,
        actual_availability=actual_availability,
        budget_total=budget_total,
        budget_remaining=max(budget_remaining, 0),
        budget_consumed_pct=min(budget_consumed_pct, 100),
        window_start=window_start,
        window_end=now,
        burn_rate=burn_rate,
    )
```

#### 6.2.4 SLO 违反告警规则

```yaml
# infra/monitoring/alerting/slo_rules.yml
#
# WHY 用 burn rate alert 而非简单阈值：
# 简单阈值（如"可用性 < 99.5%"）只在 SLO 已经违反时告警，太迟了。
# Burn rate alert 在"按当前速度将会违反 SLO"时就告警，给你修复时间。

groups:
  - name: aiops_slo
    interval: 60s
    rules:
      # ═══════════════════════════════════════
      # 可用性 SLO：99.5% over 28d
      # ═══════════════════════════════════════

      # 快速消耗告警（1 小时 burn rate > 14.4x）
      # WHY 14.4x：如果持续 1 小时消耗 14.4 倍 error budget，
      # 那 2 小时就会消耗掉 28 天总预算的 ~4.3%，需要立即响应
      - alert: SLOAvailabilityFastBurn
        expr: |
          (
            sum(rate(aiops_agent_requests_total{status="error"}[1h]))
            / sum(rate(aiops_agent_requests_total[1h]))
          ) > (14.4 * 0.005)
        for: 2m
        labels:
          severity: critical
          slo: availability
        annotations:
          summary: "可用性 Error Budget 快速消耗中"
          description: "1h 错误率 {{ $value | humanizePercentage }}，是 error budget 的 {{ $value | humanize }} / 0.005 倍"

      # 慢速消耗告警（6 小时 burn rate > 6x）
      - alert: SLOAvailabilitySlowBurn
        expr: |
          (
            sum(rate(aiops_agent_requests_total{status="error"}[6h]))
            / sum(rate(aiops_agent_requests_total[6h]))
          ) > (6 * 0.005)
        for: 15m
        labels:
          severity: warning
          slo: availability
        annotations:
          summary: "可用性 Error Budget 持续消耗"
          description: "6h 错误率 {{ $value | humanizePercentage }}，需要关注"

      # ═══════════════════════════════════════
      # 延迟 SLO：P99 < 30s over 28d
      # ═══════════════════════════════════════
      - alert: SLOLatencyBreach
        expr: |
          histogram_quantile(0.99,
            sum(rate(aiops_agent_duration_seconds_bucket[1h])) by (le)
          ) > 30
        for: 10m
        labels:
          severity: warning
          slo: latency
        annotations:
          summary: "Agent P99 延迟超过 SLO 目标 30s"
          description: "当前 P99 {{ $value | humanizeDuration }}，影响用户体验"

      # ═══════════════════════════════════════
      # 诊断质量 SLO：中位数 ≥ 0.7
      # ═══════════════════════════════════════
      - alert: SLOQualityDegraded
        expr: |
          histogram_quantile(0.5,
            sum(rate(aiops_diagnosis_confidence_bucket[6h])) by (le)
          ) < 0.7
        for: 30m
        labels:
          severity: warning
          slo: quality
        annotations:
          summary: "诊断质量低于 SLO 目标"
          description: "6h 置信度中位数 {{ $value }}，可能需要优化 Prompt 或更新知识库"
```

## 7. 指标收集 Middleware（Python FastAPI）

```python
# python/src/aiops/web/middleware/metrics.py
"""
FastAPI 请求指标中间件

自动收集：
- 请求总数（按路径、方法、状态码）
- 请求延迟（Histogram）
- 活跃请求数（Gauge）
"""

import time
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response

from aiops.observability.metrics import (
    AGENT_REQUESTS_TOTAL, AGENT_DURATION_SECONDS, AGENT_ACTIVE_SESSIONS,
)


class PrometheusMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next) -> Response:
        if request.url.path in ("/metrics", "/healthz"):
            return await call_next(request)

        AGENT_ACTIVE_SESSIONS.inc()
        start = time.monotonic()

        try:
            response = await call_next(request)
            status = "success" if response.status_code < 400 else "error"
        except Exception:
            status = "error"
            raise
        finally:
            duration = time.monotonic() - start
            AGENT_ACTIVE_SESSIONS.dec()

            path = self._normalize_path(request.url.path)
            AGENT_REQUESTS_TOTAL.labels(agent=path, status=status).inc()
            AGENT_DURATION_SECONDS.labels(agent=path).observe(duration)

        return response

    @staticmethod
    def _normalize_path(path: str) -> str:
        """路径归一化（避免高基数标签）"""
        parts = path.strip("/").split("/")
        if len(parts) >= 3:
            return f"/{parts[0]}/{parts[1]}/{parts[2]}"
        return path
```

```python
"""
告警规则 PromQL 表达式验证

WHY 测试告警规则：
- PromQL 语法错误不会在 Prometheus 加载时报错（只是告警不生效）
- 阈值设置是否合理需要验证
- 确保关键告警不会被遗漏

测试方法：
- 使用 promql-test 工具（prometheus/prometheus 自带）
- 构造测试数据 → 验证告警是否在预期条件下触发
"""

import yaml
import pytest


class TestAlertRules:
    """告警规则验证"""

    @pytest.fixture
    def alert_rules(self):
        """加载告警规则文件"""
        with open("infra/monitoring/alerting/aiops_rules.yml") as f:
            return yaml.safe_load(f)

    def test_all_rules_have_severity_label(self, alert_rules):
        """所有告警规则必须有 severity 标签"""
        # WHY：severity 是告警路由的关键标签，缺少会导致告警发不出去
        for group in alert_rules["groups"]:
            for rule in group["rules"]:
                if "alert" in rule:  # 是告警规则而非 recording rule
                    assert "severity" in rule.get("labels", {}), \
                        f"告警 {rule['alert']} 缺少 severity 标签"

    def test_critical_alerts_have_short_for(self, alert_rules):
        """Critical 告警的 for 时间不应超过 5 分钟"""
        # WHY：critical 告警如果 for 设 30m，那 30 分钟后才会通知，太迟了
        for group in alert_rules["groups"]:
            for rule in group["rules"]:
                if rule.get("labels", {}).get("severity") == "critical":
                    for_duration = rule.get("for", "0s")
                    minutes = self._parse_duration_minutes(for_duration)
                    assert minutes <= 5, \
                        f"Critical 告警 {rule['alert']} 的 for={for_duration} 超过 5 分钟"

    def test_all_alerts_have_annotations(self, alert_rules):
        """所有告警必须有 summary 注解"""
        for group in alert_rules["groups"]:
            for rule in group["rules"]:
                if "alert" in rule:
                    assert "summary" in rule.get("annotations", {}), \
                        f"告警 {rule['alert']} 缺少 summary 注解"

    def test_no_duplicate_alert_names(self, alert_rules):
        """告警名不应重复"""
        alert_names = []
        for group in alert_rules["groups"]:
            for rule in group["rules"]:
                if "alert" in rule:
                    alert_names.append(rule["alert"])

        duplicates = [name for name in alert_names if alert_names.count(name) > 1]
        assert not duplicates, f"重复的告警名: {set(duplicates)}"

    def test_slo_alerts_exist(self):
        """SLO 相关告警规则必须存在"""
        with open("infra/monitoring/alerting/slo_rules.yml") as f:
            slo_rules = yaml.safe_load(f)

        alert_names = []
        for group in slo_rules["groups"]:
            for rule in group["rules"]:
                if "alert" in rule:
                    alert_names.append(rule["alert"])

        # 关键 SLO 告警必须存在
        assert "SLOAvailabilityFastBurn" in alert_names
        assert "SLOAvailabilitySlowBurn" in alert_names
        assert "SLOLatencyBreach" in alert_names
        assert "SLOQualityDegraded" in alert_names

    @staticmethod
    def _parse_duration_minutes(duration_str: str) -> float:
        """解析 Prometheus 时间格式（如 '5m', '1h', '30s'）"""
        if duration_str.endswith("m"):
            return float(duration_str[:-1])
        elif duration_str.endswith("h"):
            return float(duration_str[:-1]) * 60
        elif duration_str.endswith("s"):
            return float(duration_str[:-1]) / 60
        return 0
```

```python
    @patch("aiops.observability.sampling.random")
    def test_normal_span_sampled_by_rate(self, mock_random):
        """正常请求应按 10% 比例采样"""
        mock_random.random.return_value = 0.05  # < 0.1 → 采样
        span = self._make_span(duration_ns=2_000_000_000)  # 2s
        self.processor.on_end(span)
        self.delegate.on_end.assert_called_once()

    @patch("aiops.observability.sampling.random")
    def test_normal_span_dropped_by_rate(self, mock_random):
        """正常请求按比例丢弃"""
        mock_random.random.return_value = 0.5  # > 0.1 → 丢弃
        span = self._make_span(duration_ns=2_000_000_000)  # 2s
        self.processor.on_end(span)
        self.delegate.on_end.assert_not_called()

    def test_same_trace_consistent_decision(self):
        """同一 trace 下的 Span 应该有一致的采样决策"""
        # 根 Span（错误 → 采样）
        root_span = self._make_span(
            trace_id=99999,
            status_code=StatusCode.ERROR,
        )
        self.processor.on_end(root_span)

        # 同一 trace 下的子 Span
        child_span = self._make_span(
            trace_id=99999,
            status_code=StatusCode.OK,
            parent=MagicMock(),
        )
        self.processor.on_end(child_span)

        # 两个 Span 都应该被发送到 delegate
        assert self.delegate.on_end.call_count == 2
```
