# 资源调度与成本优化 Agent

> AI Agent 赋能大数据平台 — 应用方向分析

## 1. 方向概述

资源调度与成本优化 Agent 旨在通过 AI 分析大数据平台的资源使用模式，实现智能调度、精准成本核算和资源浪费治理。在企业降本增效的大背景下，这个方向直接与业务 KPI 挂钩。

## 2. 核心场景分析

### 2.1 YARN/K8s 资源优化

**场景描述**：分析历史作业的资源使用模式（CPU、内存、磁盘 IO），发现资源配置不合理的作业和队列，给出优化建议。

**实际用性评估**：⭐⭐⭐⭐⭐（极高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 极高。大数据平台资源浪费是普遍问题——开发者习惯性超配资源（"多给点内存保险"），导致集群整体利用率通常只有 40-60% |
| **技术可行性** | 高。YARN ResourceManager / K8s Metrics Server 提供丰富的资源使用数据；分析方法成熟（时序统计 + 异常检测） |
| **落地难点** | ① 调参建议的安全性——降低资源配置可能导致 OOM/Spill；② 需要区分批处理和实时作业的不同资源模式；③ 组织阻力——资源调整涉及部门利益 |
| **ROI** | 极高。资源优化 10% 在中大规模集群（1000+ 节点）意味着每年节省百万级成本 |

**大数据平台特有挑战**：
- **YARN 队列调度复杂性**：Capacity Scheduler / Fair Scheduler 的配额分配直接影响多租户公平性
- **Spark 动态资源分配**：`spark.dynamicAllocation` 参数需要根据作业特征精细调优
- **混合负载**：同一集群跑 Spark 批处理、Flink 实时流、HBase 在线查询，资源争抢模式不同

**Agent 能做什么**：
```
历史作业资源使用数据（YARN/K8s metrics）
        ↓
Agent 分析：
  • 哪些作业申请了 8 核但平均只用 2 核？
  • 哪些队列的资源利用率低于 30%？
  • 哪些时段有资源空闲？
        ↓
生成优化建议：
  • 作业级：spark.executor.cores 从 8 调整到 4
  • 队列级：marketing 队列配额从 30% 降到 20%
  • 集群级：凌晨 2-6 点可以调度更多离线作业
```

### 2.2 作业级成本归因

**场景描述**：精确计算每个 Spark/Hive/Flink 作业消耗的计算资源和存储资源，按部门/项目/业务线分摊成本。

**实际用性评估**：⭐⭐⭐⭐（高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 高。大数据平台通常是公司最大的 IT 基础设施成本之一，但"谁用了多少"长期是笔糊涂账 |
| **技术可行性** | 中高。计算资源归因（CPU·时 × 单价）相对清晰，但存储归因（共享 HDFS、多副本、压缩率差异）复杂 |
| **落地难点** | ① 成本模型定义需要和财务/管理层对齐；② 共享资源（NameNode、Zookeeper）的分摊方式；③ 数据口径一致性 |
| **ROI** | 高。精准成本归因是推动各业务线优化资源使用的前提 |

**Agent 的智能化价值**：
- 不只是计算数字，还能**识别异常**："市场部本周计算成本环比增长 200%，主要由 job_xxx 导致，该作业扫描了未分区的全量表"
- 给出**优化建议**："如果该作业增加分区过滤条件，预计成本可降低 80%"
- 生成**趋势报告**：按周/月自动生成各部门的资源消耗趋势和 Top N 消耗作业

### 2.3 闲置资源发现

**场景描述**：自动扫描并识别长期未使用的资源——闲置队列、僵尸作业、废弃表/数据。

**实际用性评估**：⭐⭐⭐⭐⭐（极高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 极高。大数据平台的"熵增"是持续性的——表越建越多、作业越加越多、队列只增不减 |
| **技术可行性** | 高。扫描 YARN 调度日志 + Hive 审计日志 + HDFS 访问时间即可 |
| **落地难点** | ① "闲置"的定义需要和业务确认（季度报表可能一季度才跑一次）；② 删除操作的风险控制 |
| **ROI** | 极高。实际案例中，闲置资源治理通常能回收 15-30% 的存储空间 |

**大数据平台典型闲置场景**：
- 开发测试阶段创建的临时表，项目上线后从未清理
- 下线业务的调度作业仍在运行但输出无人消费
- 某部门撤销但 YARN 队列配额仍然保留
- HDFS 上的 Spark Checkpoint/Staging 目录堆积

**Agent 的自动化治理流程**：
```
定期扫描 → 生成闲置资源清单
  • 表：最后访问时间 > 90天，无下游血缘
  • 作业：输出表无消费者
  • 队列：过去 30 天利用率 < 5%
        ↓
通知资源所有者确认
        ↓
确认后执行清理 / 归档
```

### 2.4 弹性扩缩容建议

**场景描述**：根据历史负载模式预测未来资源需求，自动生成扩缩容计划。

**实际用性评估**：⭐⭐⭐⭐（高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 高。大数据平台负载有明显的周期性（月初/月末报表、双十一大促、年度结算），但扩缩容决策通常靠经验 |
| **技术可行性** | 中高。时序预测（Prophet、ARIMA、LSTM）技术成熟，但大数据集群的扩缩容不像 K8s 那样秒级弹性 |
| **落地难点** | ① 物理集群扩容有硬件采购周期；② 缩容需要考虑 HDFS 数据均衡和副本安全；③ 预测模型需要足够的历史数据 |
| **ROI** | 中高。价值在于"提前规划"——让采购和扩容不再是临时抱佛脚 |

**与云上弹性的结合**：
- 如果是 EMR/CDH on Cloud，Agent 可直接触发自动扩缩容
- 混合部署场景：固定节点承载基准负载，弹性节点承载峰值

## 3. 与已有系统的协同关系

```
资源与成本 Agent          智能运维 Agent（已有）
     ↕                         ↕
  共享数据源：
  • YARN RM Metrics        • 集群健康指标
  • K8s Metrics Server     • 组件状态监控
  • Spark History Server   • 告警数据
  • HDFS Audit Log         
     ↕                         ↕
  互补关系：
  • 运维 Agent → "系统出了什么问题？怎么修？"
  • 成本 Agent → "系统在浪费什么资源？怎么省？"
```

## 5. 面试亮点提炼

- **业务价值**：直接与成本 KPI 挂钩，"每年节省 X 百万"是非常有说服力的数据
- **数据驱动**：体现从数据中挖掘价值的能力，不只是技术实现
- **系统思维**：资源优化需要全链路视角（作业→队列→集群→成本中心），展示对大数据平台整体架构的理解
- **与运维互补**：运维解决"系统健不健康"，成本优化解决"系统经不经济"，两者结合才是完整的平台治理
