# 自助式数据服务 Agent（数据民主化）

> AI Agent 赋能大数据平台 — 应用方向分析

## 1. 方向概述

自助式数据服务 Agent 旨在打破"数据只有技术人员能用"的壁垒，让业务人员（产品经理、运营、市场）通过自然语言直接获取数据洞察。核心目标是**数据民主化**——让数据从"IT 部门的资产"变成"全公司的资产"。

## 2. 核心场景分析

### 2.1 对话式数据分析

**场景描述**：业务人员用自然语言提问，Agent 自动理解意图 → 生成 SQL → 查询数据 → 返回结果 + 可视化图表 + 简要分析说明。

**实际用性评估**：⭐⭐⭐⭐⭐（极高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 极高。数据分析师是企业稀缺资源，业务团队的数据需求经常排队等待数天。"等数据"是业务决策的最大瓶颈之一 |
| **技术可行性** | 中高。核心技术是 Text-to-SQL（与数据开发效能 Agent 复用），额外需要意图理解、结果解读和可视化能力 |
| **落地难点** | ① 业务人员的表述模糊性高（"最近业绩怎么样"→ 什么指标？什么时间范围？什么粒度？）；② 数据安全——业务人员不应看到超出权限的数据；③ 结果的准确性需要可验证 |
| **ROI** | 极高。每个自助查询替代的是"提需求→排期→开发→交付"的完整链路（通常 3-5 天），自助化后分钟级完成 |

**与 Text-to-SQL 的关系**：

对话式数据分析 = Text-to-SQL + 以下增强：
```
                     Text-to-SQL（技术侧）
                           ↓ 增强
┌──────────────────────────────────────────┐
│  对话式数据分析（业务侧包装）              │
│                                          │
│  + 多轮对话（追问、细化、对比）           │
│  + 意图澄清（模糊表述→精确查询）         │
│  + 结果可视化（自动选择图表类型）         │
│  + 自然语言解读（"环比增长15%，主要由…"） │
│  + 权限过滤（行级/列级安全）              │
│  + 查询历史和收藏                        │
└──────────────────────────────────────────┘
```

**多轮对话示例**：
```
👤 "上周华南区的 GMV 多少？"
🤖 "上周（3/17-3/23）华南区 GMV 为 1,234 万元，环比下降 5.2%"

👤 "跟华北对比呢？"
🤖 "华北区同期 GMV 为 1,567 万元，华南落后 26.9%。主要差距在电子品类"

👤 "电子品类具体差多少？"
🤖 "华南电子品类 GMV 312 万 vs 华北 789 万，差距 477 万。
    华南电子品类 GMV 本周环比下降 18%，建议关注是否有库存或促销变动"
```

### 2.2 自动化报表生成

**场景描述**：用户指定报表模板或描述需求，Agent 自动拉取数据、生成图表、撰写分析摘要，输出完整报表。

**实际用性评估**：⭐⭐⭐⭐（高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 高。大量周报/月报/经营分析报告的数据拉取和基础分析是重复性工作，分析师每周花 30-50% 时间在"出报表" |
| **技术可行性** | 中高。数据拉取+图表生成成熟；难点在"分析摘要"的质量——需要理解业务背景才能写出有价值的洞察 |
| **落地难点** | ① 报表模板标准化；② 分析摘要的准确性和深度（LLM 可能产生看似合理但实际错误的分析）；③ 数据口径一致性 |
| **ROI** | 高。每份报表节省 2-4 小时，如果每周产出 10+ 份报表，月节省时间显著 |

**Agent 的差异化**：
- 不只是 BI 工具的替代，而是**主动分析**——发现数据中的异常和趋势，而不是只展示数字
- 支持**自然语言定制**："帮我加一个和去年同期的对比""把粒度从周改成天"

### 2.3 数据目录导航

**场景描述**：用户不知道要查什么表时，向 Agent 描述需求，Agent 帮助在数据目录（Atlas/DataHub）中找到相关的表和字段。

**实际用性评估**：⭐⭐⭐⭐（高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 高。大数据平台表数量庞大（万级），命名规范不一，业务人员甚至新入职的数据开发都很难"找到正确的表" |
| **技术可行性** | 高。基于表注释、字段注释、业务词典的 RAG 检索即可实现 |
| **落地难点** | ① 表/字段注释的完整性（如果注释缺失或过时，Agent 也找不到）；② 同义表/同义字段的消歧（可能有多张表都包含"用户"数据） |
| **ROI** | 中高。降低找数据的时间成本，但前提是元数据治理到位 |

**Agent 对比传统搜索的优势**：
```
传统搜索：搜索 "用户" → 返回 200 张包含"用户"的表 → 用户懵了

Agent 方式：
  👤 "我想分析用户的购买行为"
  🤖 "推荐以下数据源：
      1. dwd.dwd_user_order_detail（推荐⭐）— 用户订单明细表，包含用户ID/商品/金额/时间
      2. dws.dws_user_behavior_1d — 用户日行为汇总，包含浏览/加购/下单次数
      3. dim.dim_user_profile — 用户画像维表，包含年龄/地域/会员等级
      
      如果你要做 RFM 分析，推荐用 #1；如果要看行为漏斗，推荐用 #2"
```

## 3. 目标用户与场景映射

| 用户角色 | 典型需求 | Agent 能力 |
|---------|---------|-----------|
| 产品经理 | "A/B 实验的结果怎么样？" | 对话式分析 + 可视化 |
| 运营人员 | "这个活动的转化率达标了吗？" | 对话式分析 + 自动对比 |
| 市场部门 | "给我一份本月渠道效果报告" | 自动报表生成 |
| 新入职开发 | "有没有用户画像相关的表？" | 数据目录导航 |
| 高管 | "本月经营状况概览" | 自动报表 + 关键指标摘要 |

## 4. 关键技术风险

### 4.1 准确性问题（最大风险）

业务人员使用 Agent 获取数据做决策，**如果 Agent 返回的数据不准确，后果可能比不用 Agent 更严重**。

**缓解措施**：
- 每次查询展示生成的 SQL，支持"查看 SQL"按钮
- 关键数据标注置信度和数据来源
- 与已有 BI 报表数据交叉验证
- 异常值自动标记和解释

### 4.2 安全性问题

业务人员可能无意中查询到超出权限的数据。

**缓解措施**：
- SQL 执行前经过 Ranger/Sentry 权限过滤
- 敏感字段自动脱敏展示
- 查询审计日志完整记录

## 5. 实际用性综合评估

| 子场景 | 实际用性 | 落地难度 | 推荐优先级 |
|--------|---------|---------|-----------|
| 对话式数据分析 | ⭐⭐⭐⭐⭐ | 中高 | P0 — 最核心场景，但准确性风险大 |
| 数据目录导航 | ⭐⭐⭐⭐ | 低 | P0 — 实现简单，对 Text-to-SQL 也是基础 |
| 自动报表生成 | ⭐⭐⭐⭐ | 中 | P1 — 需要报表模板体系 |

## 6. 面试亮点提炼

- **产品思维**：不只是技术实现，而是从用户体验角度思考如何让非技术人员使用数据
- **数据民主化**：这是近几年数据中台/数据平台的热门话题，展示对行业趋势的理解
- **风险意识**：能主动识别准确性风险并提出缓解方案，比单纯讲技术实现更成熟
- **与数据开发 Agent 的复用**：Text-to-SQL 引擎复用，但面向不同用户群做不同包装，展示架构复用能力
