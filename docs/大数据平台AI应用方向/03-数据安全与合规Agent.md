# 数据安全与合规 Agent

> AI Agent 赋能大数据平台 — 应用方向分析

## 1. 方向概述

数据安全与合规 Agent 聚焦于大数据平台的数据保护和合规管理，利用 AI 实现敏感数据自动发现、权限智能审计、异常访问检测和数据生命周期管理。在《个人信息保护法》《数据安全法》等法规日益严格的背景下，这个方向的刚需程度不断上升。

## 2. 核心场景分析

### 2.1 敏感数据自动发现

**场景描述**：Agent 自动扫描 Hive 表、ES 索引、HBase 表中的数据，利用 NLP + 正则规则识别 PII（个人可识别信息），如手机号、身份证号、银行卡号、邮箱、地址等。

**实际用性评估**：⭐⭐⭐⭐⭐（极高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 极高。企业大数据平台中散落着大量未标记的敏感数据，尤其是 ODS 层直接从业务库同步的原始数据 |
| **技术可行性** | 高。规则引擎（正则匹配手机号/身份证格式）+ NLP（命名实体识别地址/姓名）+ 字段名语义分析 三层组合，准确率可达 90%+ |
| **落地难点** | ① 扫描大规模数据集的性能（抽样 vs 全量）；② 误报率控制（"订单号"格式类似手机号）；③ 扫描结果后续的脱敏/加密动作的自动化 |
| **ROI** | 极高。一次合规事件的罚款/声誉损失远超技术投入；主动发现比被动暴露好 100 倍 |

**大数据平台特有挑战**：
- **数据量**：Hive 表动辄 TB/PB 级，全量扫描不现实 → 需要智能抽样策略
- **数据格式多样**：结构化（Hive）、半结构化（ES/JSON）、非结构化（HDFS 文件）都可能含敏感信息
- **动态增长**：新表每天都在创建，敏感数据发现需要持续运行而非一次性

**Agent 工作流**：
```
元数据预扫描（字段名语义分析）
  → 高疑似字段：phone, id_card, email, address, name
        ↓
数据抽样扫描（每表抽样 1000-10000 行）
  → 正则规则匹配 + NLP 实体识别
        ↓
敏感数据标记 + 分级（L1 高敏感 / L2 中敏感 / L3 低敏感）
        ↓
生成报告 + 推荐脱敏策略
  → 手机号 → 掩码（138****1234）
  → 身份证 → 哈希
  → 地址 → 模糊化（省级）
```

### 2.2 权限智能审计

**场景描述**：用户问"谁能访问用户表？""张三有哪些数据权限？""有没有过度授权的情况？"，Agent 自动聚合多个权限系统的信息，生成审计报告。

**实际用性评估**：⭐⭐⭐⭐（高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 高。大数据平台的权限体系通常涉及多个组件（Ranger/Sentry/HDFS ACL/HBase ACL/ES Security），人工审计极其耗时 |
| **技术可行性** | 中高。各组件的权限 API 可查询，但跨系统权限聚合和"过度授权"判断需要业务理解 |
| **落地难点** | ① 权限系统异构，数据聚合复杂；② "过度授权"的判断标准不统一（什么算"过度"？）；③ 权限变更的审批流程集成 |
| **ROI** | 高。最小权限原则是安全审计的基本要求，自动化审计替代人工可节省大量合规人力 |

**Agent 的智能化价值**：
- **跨系统视图**：一个用户的权限可能分散在 Ranger（Hive表权限）、HDFS（目录权限）、HBase（列族权限）、ES（索引权限）中，Agent 提供统一视图
- **过度授权检测**：对比"用户实际访问的表"与"用户被授权的表"，找出"有权但从不访问"的权限 → 建议回收
- **权限变更追溯**："这个用户的 admin 权限是什么时候授予的？由谁授予？"

### 2.3 数据访问异常检测

**场景描述**：Agent 实时监控数据访问行为，检测异常模式（如非工作时间大量导出、敏感表突发高频查询、跨部门异常访问）。

**实际用性评估**：⭐⭐⭐⭐⭐（极高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 极高。数据泄露的早期信号往往隐藏在访问日志中，但海量日志人工分析不现实 |
| **技术可行性** | 中高。基于用户行为基线的异常检测（UEBA）技术成熟，但大数据平台的访问模式比传统 IT 系统更复杂 |
| **落地难点** | ① 建立准确的用户行为基线需要足够的历史数据；② 误报率控制（月末跑报表导致的合理突发访问）；③ 实时性要求（检测到异常要及时阻断） |
| **ROI** | 极高。数据泄露事件的损失巨大，早期检测价值不可估量 |

**大数据平台的典型异常模式**：
```
⚠️ 时间异常：凌晨 3 点，某开发者查询了用户隐私表（该用户历史从不在凌晨工作）
⚠️ 量级异常：某用户今天 SELECT * FROM user_profile 执行了 50 次（历史日均 2 次）
⚠️ 权限异常：某实习生账号访问了核心财务数据表
⚠️ 路径异常：某用户突然开始访问从未接触过的业务域数据
⚠️ 导出异常：某用户通过 `INSERT OVERWRITE LOCAL DIRECTORY` 导出了 500GB 数据
```

**与运维 Agent 的协同**：
- 安全异常事件可触发运维 Agent 的联动处置（如临时冻结账户、阻断查询）
- 共享日志采集链路（Kafka → 安全分析 / 运维分析 分流处理）

### 2.4 数据生命周期管理

**场景描述**：Agent 根据数据分级和保留策略，自动识别需要归档、冻结或删除的数据，生成清理计划。

**实际用性评估**：⭐⭐⭐⭐（高）

| 维度 | 分析 |
|------|------|
| **痛点强度** | 高。大数据平台存储成本持续增长，但"谁也不敢删数据"是普遍心态 |
| **技术可行性** | 高。基于数据分级（公开/内部/机密/绝密）× 保留策略（30天/1年/永久）的规则引擎即可实现 |
| **落地难点** | ① 数据分级标准的制定需要业务和法务参与；② 删除前的影响分析（是否有下游依赖）；③ 合规留存要求（某些数据法律要求保留 N 年） |
| **ROI** | 高。存储成本通常占大数据平台总成本的 30-40%，生命周期管理可节省显著成本 |

**Agent 的差异化价值**：
- 结合血缘分析判断数据是否还有消费者
- 结合访问日志判断数据的实际活跃度
- 自动区分"可以安全删除"和"需要人工确认"的数据

## 3. 与已有系统的协同关系

```
数据安全与合规 Agent         智能运维 Agent（已有）
      ↕                            ↕
  安全事件 ──触发──→ 运维联动处置
  异常检测 ──共享──→ 告警收敛通道
      ↕                            ↕
  共享基础设施：
  • Kafka 日志管道
  • RAG 知识库（安全策略 + 合规文档）
  • MCP 工具层（Ranger API、HDFS API）
```

## 4. 实际用性综合评估

| 子场景 | 实际用性 | 落地难度 | 推荐优先级 |
|--------|---------|---------|-----------|
| 敏感数据发现 | ⭐⭐⭐⭐⭐ | 中 | P0 — 合规刚需 |
| 异常访问检测 | ⭐⭐⭐⭐⭐ | 中高 | P0 — 安全防线 |
| 权限审计 | ⭐⭐⭐⭐ | 中 | P1 — 审计必备 |
| 数据生命周期 | ⭐⭐⭐⭐ | 中 | P1 — 成本+合规双重价值 |

## 5. 面试亮点提炼

- **合规刚需**：在《个人信息保护法》《数据安全法》背景下，数据安全是企业不得不做的事
- **差异化竞争力**：大多数候选人只会讲运维，能讲数据安全 Agent 的人很少
- **跨系统整合**：展示对 Ranger、HDFS ACL、审计日志等多组件权限体系的理解
- **风险意识**：展示安全意识和合规思维，这是高级工程师的必备素质
